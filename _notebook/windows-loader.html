The Windows Loader,
or: how a DLL actually shows up in your address space.

A walk through the code path from LoadLibrary to DllMain — and the four families of attack and detection that exist nowhere else but inside that path.

blacksunCUBE field notebook · vol. 04 · builds on vol. 02 (PE) & vol. 03 (PEB/Ldr)

Vol. 03 ended with the loader's data structures: the PEB.Ldr list, each entry pointing to a loaded module's base. We treated those as static facts — here is the list, here are the bases. Now we ask the question that matters for offense and defense: how does an entry actually get into that list?

The answer is a code path inside ntdll.dll that is almost entirely undocumented, has been reverse-engineered at length by the security industry, and is the single most monitored region of code in any process running an EDR. Three things live in this path that exist nowhere else:

  1. The first arbitrary code execution opportunity — DllMain runs before any other code in a freshly-loaded DLL.
  2. The DLL search order — the algorithm that decides which DLL on disk satisfies a name, which is the entire foundation of DLL hijacking.
  3. Loader lock — a process-wide critical section that enforces serialization on every load, and which is the source of more deadlocks and EDR detection signals than any other primitive in user mode.

Same axes you know:

R · Read

Loader inputs

The DLL name, the search path, the module list. Things the loader reads to decide what to do next.

W · Write

Loader work

Mapping the file, applying relocations, patching the IAT, growing the LDR list. State the loader builds.

X · Execute

Code that fires

TLS callbacks, then DllMain. The moment when the DLL starts running its own code.

§ 01 Why a loader exists at all

A naive picture of program startup, and what's missing from it.

Imagine you compile a small C program that calls MessageBoxW. The naive mental model is: the OS reads your .exe, copies it into memory, jumps to main, and you're running. That picture is wrong in one critical way: MessageBoxW doesn't live in your .exe. It lives in user32.dll. When your code says call MessageBoxW, the CPU needs an address to jump to — and that address depends on where user32.dll ended up in your process's virtual memory, which is decided at runtime by ASLR.

So before main can run, somebody has to: find user32.dll on disk, map it into memory, walk its export table to find MessageBoxW, write that address into your IAT (vol. 02), and only then jump to main. That somebody is the loader. It lives in ntdll.dll. It runs before any of your code. And it doesn't stop after main — every LoadLibrary call you make later goes through the same machinery.

The loader is the closest thing the OS has to a runtime that runs inside your process. It's not a service, not a daemon — it's a chunk of code in ntdll.dll that the kernel hands the first execution to. Everything you call, including main, runs after the loader has already done its job. If you've ever wondered why a debugger that breaks on the EXE entry point already shows several DLLs loaded — this is why.

§ 02 Process startup: LdrInitializeThunk

The kernel's handoff to user mode lands here, not at your entry point.

When the kernel finishes setting up a new process — allocating the address space, mapping the EXE, creating the initial thread — it does not jump to your main. It jumps to a function inside ntdll.dll called LdrInitializeThunk. That function is the entry point of every process on Windows.

What follows is the canonical process-startup call chain. This is the call stack you'll see if you set a debugger breakpoint on the very first instruction of any program:

Notice the structure: all the frames in ntdll with names starting Ldrp* are internal loader work. The single frame that leaves ntdll — LdrpCallInitRoutine calling someone else's DllMain — is the moment when the loader briefly hands control to code it doesn't trust. That single transition is the most important boundary in the entire loader.

Signal for defenders: LdrpCallInitRoutine is exactly where EDR kernel callbacks fire for image-load notifications. Specifically, PsSetLoadImageNotifyRoutine dispatches to registered drivers before DllMain runs — this is a callout from the kernel-side image-mapping machinery, not from ntdll. The driver receives the new image's base, size, full path, and process context, and can examine, log, or block. For purple team work, this callback chain is your visibility into every DLL load in every process on the system, including ones that bypass user-mode hooks.

Both paths converge on LdrpCallInitRoutine. Whether the DLL is loaded statically (in the IAT, resolved during process init) or dynamically (via an explicit LoadLibrary at runtime), the moment of foreign-code execution is the same single function call. This is why every EDR kernel-callback chain anchors at image load — one watchpoint covers both.

§ 03 What LdrLoadDll actually does

The undocumented native API that LoadLibrary wraps.

NTSTATUS LdrLoadDll(...)
readwriteexecute

"Give me a DLL name. I'll find it, map it, relocate it, fix its imports, run its TLS callbacks, run its DllMain, add it to the LDR list, and give you back its base address."

NTSTATUS LdrLoadDll(
    [in, optional] PWSTR           SearchPath,        // optional override of search path
    [in, optional] PULONG          DllCharacteristics,
    [in]           PUNICODE_STRING ModuleFileName,    // the DLL name
    [out]          PHANDLE         ModuleHandle       // receives HMODULE = DllBase
);

Three things to pull out of that signature, all relevant to detection:

  • The name comes in as a UNICODE_STRING. Vol. 01 introduced this type; here you see why it matters. The Win32 wrappers (LoadLibraryA, LoadLibraryW) take a plain LPCWSTR, then call RtlInitUnicodeString internally to wrap it before invoking LdrLoadDll. The kernel and the loader speak UNICODE_STRING exclusively.
  • It returns an HMODULE by out-pointer. Same as LoadLibrary's return value. Same as the entry's DllBase in the LDR list. They are all the same value — the DLL's load address in your process — just typed differently.
  • It takes an optional SearchPath. If NULL, the loader uses the standard search order (next section). If non-null, it overrides — which is useful for legitimate apps that want to control where their dependencies come from, and the same primitive is used by LoadLibraryEx's LOAD_LIBRARY_SEARCH_* flags.

Internally, after acquiring loader lock, LdrLoadDll does roughly this:

// pseudocode of what LdrpLoadDll & friends do, in order

1.  LdrpFastCheckIfLoaded(name)              // already in LDR list? return existing entry
2.  LdrpResolveDllName(name, search_path)    // run the search-order algorithm
3.  LdrpMapDllNtFileName(full_path)            // NtCreateFile + NtCreateSection + NtMapViewOfSection
4.  LdrpProcessRelocationBlock(...)            // apply .reloc fixups (vol. 02)
5.  LdrpSnapModule(...)                        // patch IAT — resolve all imports
6.  LdrpInsertDataTableEntry(...)              // add LDR_DATA_TABLE_ENTRY to all 3 lists
7.  LdrpProcessTLSData(...)                    // fire TLS callbacks (vol. 02)
8.  LdrpCallInitRoutine(DllMain, DLL_PROCESS_ATTACH)  // finally — DllMain
9.  return ModuleHandle = DllBase

Steps 3, 4, 5 are pure mechanical work — map the file, fix up pointers. Step 6 is when the new DLL becomes visible to the rest of the process via the LDR list. Step 7 is the first foreign-code execution (TLS callbacks). Step 8 is the second (DllMain). Steps 7 and 8 both run under loader lock, which is why the lock matters so much — we'll come back to that.

§ 04 The DLL search order: where step 2 looks

An algorithm with security implications hiding in every directory entry.

When you call LoadLibrary("foo.dll") — with just the bare name, no full path — the loader has to decide which foo.dll on disk you meant. Multiple copies of the same DLL routinely exist in different places. The algorithm that picks one is the DLL search order, and getting between the loader and the DLL it's looking for is the foundation of an entire attack class.

The search runs from top to bottom and stops at the first match. With SafeDllSearchMode enabled (the default since Windows XP SP2), the order is:

Two registry values rule everything above

Both of these decide what the loader does:

HKLM\System\CCS\Control\
Session Manager\
SafeDllSearchMode		 If 1 (default), CWD is at position 5 in the order — after the system directories. If 0, CWD moves to position 2, and an attacker who controls the working directory of any LoadLibrary-calling process can trivially substitute system DLLs. Defenders should ensure this stays 1 — modern Windows enforces it but legacy and CIS-compliant environments verify explicitly.
HKLM\System\CCS\Control\
Session Manager\
KnownDLLs		 A list of DLL names. Any DLL in this list is loaded from a preinitialized section object the kernel set up at boot — not from disk at all. Bypasses the entire search order. kernel32.dll, user32.dll, advapi32.dll, ntdll.dll, and dozens more are KnownDLLs by default. This is what makes most "DLL hijacking" of core system DLLs impossible.
KnownDLLs is the reason you can't just drop a malicious kernel32.dll next to notepad.exe and have it loaded. The kernel mapped the real kernel32.dll as a section at boot — every process gets that section mapped in, and the loader hits the "already loaded" check before it ever consults the search order. To attack a KnownDLL you need either kernel-level write to the registry list (admin + reboot) or to find a DLL not in the KnownDLLs list that a target process loads with the unsafe order — which brings us to the next section.

§ 05 Loader lock: the most expensive critical section in user mode

A single process-wide lock that serializes every load, every unload, every DllMain.

The loader lock is a critical section called LdrpLoaderLock, exposed via PEB.LoaderLock. It exists for a brutal reason: the LDR list is shared by all threads, and DllMain is allowed to call LoadLibrary recursively, and TLS callbacks fire concurrently with thread creation. Without one big lock around all of it, the LDR list would corrupt instantly.

Three rules of loader lock that every shellcode author and every EDR engineer learns the hard way:

  • Held throughout DllMain. When your DllMain runs with DLL_PROCESS_ATTACH or DLL_THREAD_ATTACH, the loader lock is held by your thread. Microsoft's official documentation lists what you can and cannot do in DllMain precisely because of this lock. The forbidden list is long: don't call into user32, shell32, ole32; don't WaitForSingleObject on a non-loader-related event; don't create threads and wait for them.
  • Re-entrant on the same thread. Calling LoadLibrary from inside DllMain is technically legal — the lock is recursive, so the same thread can re-acquire it. The new DLL gets loaded, its DllMain runs, and we recurse. NetSPI's "Adaptive DLL Hijacking" research builds on exactly this property.
  • Cross-thread waits inside DllMain deadlock. If thread A is in DllMain (holding the lock) and waits on a thread B that itself tries to LoadLibrary, thread B blocks on the lock, A blocks on B, deadlock. This is the classic "DllMain best practices" failure mode — and a remarkable fraction of "process hangs at startup" bugs in production are exactly this.
why offensive code does the bare minimum in DllMain

An attacker who hijacks a DLL load gets DllMain execution — but under loader lock. The defensive guidance Microsoft publishes ("avoid these calls in DllMain") becomes offensive guidance: do as little as possible, don't load anything, don't wait on anything, and definitely don't try to reach the network. Spawn a thread (CreateThread is one of the few safe calls), have that thread do the real work, and return TRUE immediately. Every mature offensive framework knows this rule cold.

§ 06 Detection surface: what fires and where

Four chokepoints that EDR vendors converge on.

From a detection-engineering standpoint, the loader code path is dense with telemetry opportunities. The four chokepoints that matter most:

1 · image load kernel callback

PsSetLoadImageNotifyRoutine registered by EDR drivers fires for every image mapping in every process system-wide. Receives image base, size, full file path, process ID, and a flag indicating whether it's the main EXE or a DLL. This is the foundational EDR primitive — bypassing it requires kernel exploitation or the EDR's driver to be unloaded. Cannot be evaded from user mode.

2 · Sysmon Event 7 / Event ID 4688

Sysmon's image-load event captures every DLL load with hashes, signing status, and full path. Event ID 7 fires for every LdrpCallInitRoutine-driven load. High-volume but invaluable: a process that loads an unsigned DLL from a user-writable path during its first second of life is rarely benign. Pair with command-line + parent process for high-fidelity rules.

3 · ETW Microsoft-Windows-
Kernel-Process

Kernel-side ETW image-load events. Same data as Sysmon Event 7 but consumed by EDRs without a Sysmon dependency. Faster, no buffering, less likely to drop events under load. ETW-TI (Microsoft-Windows-Threat-Intelligence) also surfaces loads with extra context, but only PPL consumers can subscribe to it.

4 · user-mode hooks on LdrLoadDll

Many EDRs install inline hooks on LdrLoadDll in ntdll.dll to inspect the request before the load happens, with the option to block it. User-mode hooks can be bypassed by direct syscalls (vol. 05 territory) but they remain the front-line defense against most commodity malware.

§ 07 A taxonomy of DLL hijacking

Five flavors of the same idea: get the loader to load your DLL instead of the real one.

Every variation of "DLL hijacking" reduces to abusing one of the steps in the loader path. Knowing the taxonomy lets you predict the telemetry signature of each.

1. Search-order hijacking

Drop a malicious DLL with the same name as one the target loads, in a directory that the search hits before the legitimate one. Classic case: target loads VERSION.DLL by name; you place your VERSION.DLL in the application directory (search step 1), and the loader finds yours before System32's (step 2). Detection: image-load events for unsigned DLLs in non-standard paths, especially when a signed legitimate DLL of the same name exists in System32.

2. Side-loading (DLL planting)

Distribute a benign-signed application alongside a malicious DLL it depends on. The signed app launches, loads the DLL from its own directory (fully expected behavior), and the attacker's code runs under the cover of the signed process. Heavily used in supply-chain and initial access. Detection: signed parent process + unsigned image load in the same directory, where the unsigned DLL is unique to this victim's host.

3. Phantom DLL hijacking

Target a DLL that an application tries to load but which doesn't exist on the system. Because the loader gets a "file not found" and falls through, attackers can plant their DLL in a writable location and satisfy the load. The Unit 42 article on msdtc.exe loading a nonexistent oci.dll is the canonical case study. Detection: unsigned DLL appearing where there was previously a load failure for the same name.

4. KnownDLLs override (admin-only)

Modify the HKLM\System\CCS\Control\Session Manager\KnownDLLs registry list to remove a critical DLL, then plant your version. Requires admin and a reboot for changes to take effect (KnownDLLs sections are built at boot). High-impact but high-noise — registry modifications to that key are textbook hunt targets.

5. DLL search order via SetDllDirectory abuse

Instead of dropping a DLL, change where the loader looks. SetDllDirectory alters the search path for the calling process and, critically, disables SafeDllSearchMode while a path is set. An attacker with code execution can use this primitive to redirect subsequent LoadLibrary calls. Detection: SetDllDirectory calls themselves are not always logged, but the resulting unsigned image loads from non-standard paths are.

Hunt template: for any signed parent process, find image-load events where the loaded DLL is (a) unsigned, (b) in a directory writable by non-admin users, and (c) has a same-named twin in System32. That single query catches the majority of search-order hijacking and side-loading in the wild. False-positive rate is governed by how messy your software inventory is — in mature environments, the signal is clean.
§ 08  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all, you own the loader.

When the kernel finishes setting up a new process, where does it transfer control? Why isn't it your main?
It transfers to ntdll!LdrInitializeThunk. main can't run yet because the IAT hasn't been resolved — calls to imported functions would jump to garbage. The loader has to map every dependent DLL, run TLS callbacks, run each DllMain, and only then call the EXE's entry point (which itself calls main after CRT init).
What's the canonical call stack from LoadLibrary to DllMain?
kernelbase!LoadLibraryExW → ntdll!LdrLoadDll → ntdll!LdrpLoadDll → ntdll!LdrpLoadDllInternal → ntdll!LdrpPrepareModuleForExecution → ntdll!LdrpInitializeGraphRecurse → ntdll!LdrpCallInitRoutine → foo!DllMain. The single transition out of ntdll at LdrpCallInitRoutine is where foreign code first runs.
Why can you not hijack kernel32.dll by dropping a malicious copy next to a target executable?
Because kernel32.dll is in the KnownDLLs registry list. The kernel mapped the real kernel32.dll as a named section object at boot. When any process needs kernel32.dll, the loader finds it in the KnownDLLs section before consulting the disk-based search order. Bypassing this requires modifying HKLM\System\CCS\Control\Session Manager\KnownDLLs and rebooting — admin-only, high-noise.
SafeDllSearchMode is set to 0. What changes about the search order, and why is it dangerous?
The current working directory moves from position 5 (after the system directories) to position 2 (right after the application directory). An attacker who controls the CWD of any process — for example by tricking a user into opening a file from a network share — can plant DLLs that the process will load before the real ones in System32. SafeDllSearchMode = 1 is default since XP SP2; modern hardening guides verify it explicitly.
An attacker has hijacked a DLL load. Their DllMain runs. Why do they typically spawn a thread and return immediately, instead of doing real work in DllMain?
Loader lock. DllMain runs with the process's loader lock held. Calling APIs that themselves load DLLs (most of user32, shell32, ole32, networking) deadlocks the process. CreateThread is one of the few safe primitives. The thread runs after DllMain returns, with the lock released, and can do whatever it wants. Microsoft's "DllMain best practices" doc is the authoritative restriction list.
What does PsSetLoadImageNotifyRoutine do, and why is it the foundational EDR primitive?
It registers a kernel callback that fires every time the kernel maps an executable image into any process — main EXEs and DLLs both. The callback receives the image base, size, full path, and target process. This happens before DllMain runs and cannot be evaded from user mode. EDRs use this for: image hashing, signing verification, IOC matching, optional blocking. To bypass, you need to either unload the EDR's driver (kernel exploit / BYOVD) or get your code to run before the callback registers (typically impossible).
A process loads oci.dll from a user-writable directory; oci.dll doesn't exist in System32. What technique is this and how do you hunt it?
Phantom DLL hijacking. The application code calls LoadLibrary("oci.dll") expecting a system DLL that doesn't actually exist on the box. The loader falls through the search order, finds the attacker's planted file, loads it. Hunt: image-load events for DLLs (a) loaded from non-System32 paths, (b) unsigned, and (c) where the same name was historically not present on the host. Unit 42's analysis of msdtc.exe and Cortex XDR's "Phantom DLL Loading" alert template.
Walk through what an EDR sees when an attacker uses search-order hijacking against a signed application.
The signed parent process spawns normally (clean Sysmon Event 1). The kernel's image-load callback fires for the main EXE (signed, expected). Then a second image-load callback fires for the planted DLL: same parent process, but the DLL is unsigned, in the application directory, and has a same-named legitimate twin in System32. That triple — signed parent + unsigned child DLL + System32 twin exists — is the high-fidelity hunt query. Sysmon Event 7 has the path and signing status; ETW Microsoft-Windows-Kernel-Process has the same data without Sysmon dependency.

§ 09 Closing thought

Four volumes in. Here is what we have built:

  • Vol. 01 — the alphabet (data types).
  • Vol. 02 — the static skeleton (PE format).
  • Vol. 03 — the living state (PEB / TEB / LDR list).
  • Vol. 04 — the machinery that builds and maintains that state (the loader).

If you understand these four volumes well, you can read any tradecraft — reflective loading, manual mapping, hollowing, side-loading, doppelganging, herpaderping — and identify which structures are touched, which loader steps are bypassed or abused, and which detection chokepoints fire. None of those techniques are tricks pulled from a hat. They are all compositions of operations on the four layers above.

The same is true for detection engineering. Every credible EDR signal eventually grounds in: did this image-load callback fire from a path we trust? Does this loaded module's DllBase match its signed image? Is loader lock being held by a thread doing something it shouldn't? Did LdrLoadDll just return a path nobody expected?
