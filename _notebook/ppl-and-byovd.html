Protected Process Light & Bring-Your-Own-Vulnerable-Driver,
or: the one byte that gates everything, and the only realistic way past it.

The kernel byte at EPROCESS+0x87a that decides who can touch whom. The cross-signed driver loophole that's powered every BYOVD attack since 2018. The April 2026 Microsoft patch that finally closes it. ARC III, closed.

blacksunCUBE field notebook · vol. 12 · closes ARC III — builds on vols. 09, 10, 11

Eleven volumes have built up to this. Vol. 09 explained where bytes live. Vol. 10 explained how the kernel notifies drivers when something happens. Vol. 11 explained how the kernel publishes a stream of events for anyone authorized to read them. All three of those rely on a single concept that was deferred until now: authorization. Who is allowed to register kernel callbacks? Who is allowed to consume ETW-TI? Who is allowed to prevent another process from touching them?

The answer in every case is: processes running at sufficiently high PPL. Specifically PsProtectedSignerAntimalware-Light for ETW-TI consumption (vol. 11), and the broader PPL hierarchy for general process protection. PPL is the trust mechanism that makes EDR architecturally possible on Windows.

And BYOVD — Bring Your Own Vulnerable Driver — is the only realistic, repeatable, in-the-wild attack against PPL that doesn't require a 0-day. You don't bypass PPL with cleverness; you bypass it by getting kernel write. Kernel write requires a driver. Drivers must be signed. Therefore: load somebody else's signed-but-vulnerable driver, exploit it, get kernel write, patch the EPROCESS.Protection byte to zero, kill the EDR.

This volume covers four things, in this order:

  1. The PPL data model — EPROCESS.Protection, PS_PROTECTION structure, the bit layout, the 8 signer levels, the access check that gates everything.
  2. How a process becomes PPL — ELAM-signed drivers, anti-malware service registration, the boot path that registers Defender / CrowdStrike / SentinelOne as Antimalware-Light.
  3. BYOVD attack mechanics in 2026 — LOLDrivers ecosystem, the standard 4-step kill chain, EDR-killer commoditization (AuKill, Terminator, EDRKillShifter), specific 2025–2026 vulnerable drivers being weaponized.
  4. Defenses — Microsoft Vulnerable Driver Blocklist (WDB), HVCI, the April 2026 cross-signed trust removal (Microsoft's most consequential kernel security policy change in two decades), and the operational implications.

Same axes:

R · Read

The Protection byte

One byte at EPROCESS+0x87a on Win 11 24H2 (offset varies by build). Reading it tells you a process's PPL level and signer. Vergilius project tracks the offset across builds.

Achieved via BYOVD — a vulnerable signed driver exposing arbitrary write through an IOCTL. Once you have it, every PPL guarantee dissolves. Patch the byte, target's PPL is gone.

X · Execute

In PsTestProtectedProcessIncompatibility and related kernel routines. Compares signer levels: requestor's level < target's level → STATUS_ACCESS_DENIED. The check is purely data-driven, which is why patching the byte defeats it.

§ 01 The Protection byte

Eight bits in EPROCESS. Three sub-fields. The entire PPL hierarchy fits inside this.

Every EPROCESS structure in the kernel contains a single byte called Protection, of type PS_PROTECTION. From notscimmy's pplib, BorderGate's reverse-engineering, and corroborated by Microsoft's own (sparse) documentation:

typedef union _PS_PROTECTION {
    UCHAR Level;                  // the whole byte
    struct {
        UCHAR Type   : 3;          // bits 0-2: protection type
        UCHAR Audit  : 1;          // bit 3: audit-only mode (rarely set)
        UCHAR Signer : 4;          // bits 4-7: signer category
    } Flags;
} PS_PROTECTION, *PPS_PROTECTION;

// On Windows 11 24H2 / 25H2, the Protection byte sits at:
//   EPROCESS + 0x87a   (verified by community via Vergilius + WinDbg)
// On older builds it's at a different offset (e.g., 0x878 on Win 11 22H2,
// 0x5fa on Win 10 1903). Vergilius tracks per-build offsets.

Two enums populate the bit fields. From pplib source:

typedef enum _PS_PROTECTED_TYPE {
    PsProtectedTypeNone           = 0,    // not protected
    PsProtectedTypeProtectedLight = 1,    // PPL — the common case
    PsProtectedTypeProtected      = 2,    // PP (full)  — DRM, audiodg.exe, anti-cheat
    PsProtectedTypeMax            = 3
} PS_PROTECTED_TYPE;

typedef enum _PS_PROTECTED_SIGNER {
    PsProtectedSignerNone         = 0,
    PsProtectedSignerAuthenticode = 1,    // generic Authenticode-signed
    PsProtectedSignerCodeGen      = 2,    // .NET/CodeGen scenarios
    PsProtectedSignerAntimalware  = 3,    // EDRs, Defender, ELAM-registered AVs
    PsProtectedSignerLsa          = 4,    // LSASS specifically
    PsProtectedSignerWindows      = 5,    // general system processes
    PsProtectedSignerWinTcb       = 6,    // Trusted Computing Base — highest "normal"
    PsProtectedSignerWinSystem    = 7,    // the kernel itself / System process
    PsProtectedSignerApp          = 8     // store apps
} PS_PROTECTED_SIGNER;

Memorizing the common levels (0x31, 0x41, 0x61, 0x62) means you can read any process listing and immediately know which processes are off-limits to a non-PPL caller.

§ 02 The access check

How the kernel decides "you can't touch this".

When a process calls OpenProcess(target_pid, PROCESS_VM_READ | PROCESS_VM_WRITE | PROCESS_VM_OPERATION), the kernel runs through several checks. The PPL check is one of them, performed by PsTestProtectedProcessIncompatibility and similar internal routines. The simplified logic:

// Pseudocode of the kernel's access-check for protected processes:

NTSTATUS CheckPplAccess(EPROCESS *requestor, EPROCESS *target, ACCESS_MASK desired) {
    UCHAR reqProt = requestor->Protection.Level;
    UCHAR tgtProt = target->Protection.Level;

    // Unprotected target: standard ACL check applies, no PPL gate
    if (tgtProt == 0) return StandardAclCheck(...);

    // Protected target, unprotected requestor:
    // strip dangerous access rights, allow only PROCESS_QUERY_LIMITED_INFORMATION
    // and PROCESS_TERMINATE in some cases
    if (reqProt == 0) {
        if (desired & ~PROCESS_QUERY_LIMITED_INFORMATION)
            return STATUS_ACCESS_DENIED;
        return StandardAclCheck(...);
    }

    // Both protected: compare signer levels
    if (reqProt.Signer < tgtProt.Signer)
        return STATUS_ACCESS_DENIED;

    return StandardAclCheck(...);
}

Two things worth pointing out:

  • Even SYSTEM cannot bypass this from user mode. The check is done by the kernel against the requestor's EPROCESS.Protection, not against its token. NT AUTHORITY\SYSTEM running at protection level 0 cannot open Defender (level 0x31) for read or write — the kernel returns STATUS_ACCESS_DENIED.
  • The check is data-driven. It reads two bytes (requestor's and target's Protection), compares numbers, returns. If you can flip the requestor's byte to 0x72 and the target's to 0x00, every check passes. That's the entire BYOVD attack model in one sentence.

Handle stripping

Per S12's writeup and BorderGate: even when the access check returns success, the kernel may strip rights from the resulting handle. Asking for PROCESS_ALL_ACCESS against a higher-PPL process can return a valid handle with only PROCESS_QUERY_LIMITED_INFORMATION. The handle is real, but you can't write memory through it. This subtlety matters: an attacker checking only OpenProcess's return value misses that the handle is essentially useless. Code that walks WriteProcessMemory failures often reveals PPL where naive code doesn't.

§ 03 How a process becomes PPL

The ELAM driver bootstrap. Why "Antimalware-Light" requires Microsoft's blessing.

You can't just call SetProcessProtection(0x31) — no such API exists. PPL is established at process creation time inside the kernel, based on:

  1. Image signature certificate. The kernel parses the binary's signature chain. The certificate must chain to a Microsoft root and contain specific OIDs (e.g., 1.3.6.1.4.1.311.61.4.7 for Antimalware) embedded by Microsoft when they sign the image.
  2. For Antimalware-Light specifically: an associated ELAM driver. Early-Launch-Anti-Malware drivers are loaded by the kernel before any non-boot driver. They register with the kernel via EarlyLaunchAntiMalware infrastructure. When the AV's user-mode service starts, the kernel checks for the registered ELAM driver. If present and Microsoft-approved (specific signing requirements), the user-mode service is allowed to launch as PPL Antimalware-Light.
  3. Process creation token signaling. The service creation request signals the desired protection. The kernel verifies all preconditions match, sets EPROCESS.Protection at creation, and returns. Cannot be elevated post-creation through any documented API.

What this means for an EDR vendor

Becoming PPL Antimalware-Light is a multi-month engineering and certification process. The vendor must:

  1. Write an ELAM driver and submit it to Microsoft for review.
  2. Receive a special anti-malware code-signing certificate from Microsoft after passing the review.
  3. Sign both their ELAM driver and their user-mode service binary with the AM cert.
  4. Configure the user-mode service to register as a PPL service via the SCM.

This is why every commercial EDR you'll encounter — Defender, CrowdStrike Falcon, SentinelOne, Cortex XDR, Elastic Defend — runs as PPL Antimalware-Light, but a script kiddie cannot trivially run as Antimalware-Light. It's gated by Microsoft's review process.

The certificate gating is the entire reason ETW-TI's read-side monopoly was stable from 2014 to January 2026 (vol. 11). The SecurityTrace bypass that broke that monopoly didn't break PPL itself — it found a structural way to consume TI events without being PPL. PPL itself is still an effective gate; the gate just isn't the only thing protecting the events anymore.

§ 04 The BYOVD attack surface

Why kernel-mode code-signing creates a paradox the attacker exploits.

Kernel driver loading on Windows requires a valid Authenticode signature. This is enforced by Driver Signature Enforcement (DSE) on Windows since x64 Vista. The intent: prevent arbitrary code from running in kernel mode.

The unintended consequence: any legitimately signed driver, regardless of bugs, is accepted. A driver signed by a small hardware vendor in 2018 that contains an arbitrary kernel read/write vulnerability through a misconfigured IOCTL handler will load on a fully-patched 2026 system, unless explicitly blocklisted.

From the CrowdStrike taxonomy (corroborated by LOLDrivers project): driver abuse falls into three categories.

CategoryWhat it isExample
vulnerableHas unintended bug — improper IOCTL access control, exposed write primitive, etc. Vendor never intended this.Dell dbutil_2_3.sys (CVE-2021-21551), RTCore64.sys (MSI Afterburner)
weaponizableBy-design dangerous functionality exposed without sufficient access checks. Driver does what it advertises but the advertised feature is too privileged.ProcessHacker's kprocesshacker.sys, several anti-cheat drivers
maliciousPurpose-built to disable security tools, riding on a stolen or fraudulently-obtained signing certificate.RobbinHood ransomware's gdrv.sys, multiple Lazarus-attributed drivers

The first two categories — vulnerable and weaponizable — are the foundation of BYOVD. They're abundant: LOLDrivers project catalogs hundreds of unique vulnerable signed drivers as of 2026, and per MagicSword/LOLDrivers research published in 2026, more than 81% of those samples are signed via the cross-signed root program — the policy Microsoft is finally killing in April 2026 (§ 07).

Why CRL doesn't save you

From Huntress research referenced in 2026 BYOVD reporting: Windows does not check Certificate Revocation Lists when loading drivers. A driver signed by EnCase with a certificate revoked over 15 years ago still loads successfully on a fully-updated Windows 11. The trust model is "was this ever validly signed?" not "is the certificate currently trusted?". This is why the cross-signed-driver-trust removal (§ 07) is structural rather than incremental — it changes the trust model itself, not just the revocation list.

§ 06 The Microsoft Vulnerable Driver Blocklist

The defensive list. Necessary, not sufficient.

From Microsoft Learn (current as of August 2025) and Microsoft Support KB5020779: the Vulnerable Driver Blocklist (WDB) is a Microsoft-curated SiPolicy.p7b (WDAC code-integrity policy file) that contains hashes and certificate rules for known-bad drivers. When a driver matches an entry, the kernel refuses to load it.

Where it lives and how it's enforced

  • File: %windir%\System32\CodeIntegrity\DriverSiPolicy.p7b
  • Enabled by default since Win 11 22H2 — toggleable in Windows Security → Device security → Core isolation, but the toggle is grayed-out when HVCI is on (because HVCI requires the blocklist).
  • Updated 1–2 times per year via cumulative updates — not the AV definitions channel. This is the slow-update problem: new BYOVD drivers added between updates are not blocked until the next servicing release.
  • Enforced only when one of: HVCI on, Smart App Control on, S Mode on, or Windows 11 22H2+ with the blocklist toggle on (default).

The coverage gap

From Picus Security 2026 reporting and the LOLDrivers project: Microsoft's WDB covers a small fraction of LOLDrivers' catalog. Picus tracks the gap directly; LOLDrivers maintainers (Michael Haag, MagicSword) publish drivers Microsoft hasn't blocked. The reasons:

  • Compatibility caution. Microsoft is hesitant to block drivers used by legitimate software. Industrial control, medical devices, specialty hardware all depend on old signed drivers. Aggressive blocking causes operational pain.
  • Distribution latency. The blocklist updates ride on cumulative updates. New BYOVD drivers can be in active exploitation for months before Microsoft updates the blocklist.
  • Bypass possibilities. Per Windows forum coverage of community CVE advisories: certain WDAC rule qualifiers behave differently on systems without HVCI. An attacker who can disable HVCI (rare but possible on some configs) gets weaker enforcement of the blocklist.

Operational guidance

For detection engineers and EDR engineers:

  • Ingest the LOLDrivers feed directly, don't rely on WDB alone. github.com/magicsword-io/LOLDrivers publishes hashes and YARA rules continuously. The ASR (Attack Surface Reduction) rule "Block abuse of exploited vulnerable signed drivers" pairs well with LOLDrivers feeds.
  • Watch for the kill chain pattern in § 05, not for specific driver hashes alone. File drop → service creation → driver load → security process termination, within a 5–10 minute window from the same host: that's BYOVD regardless of which specific driver was used.
  • Out-of-band telemetry. If your EDR is what gets killed, it can't report its own death. A secondary monitor — network telemetry, separate log shipper, or backend "agent last-seen" tracking — is essential. Treat unexpected EDR silence as the alert.

§ 07 The April 2026 cross-signed trust kill

Microsoft's most consequential kernel-security policy change in two decades.

Cross-signed kernel driver trust removal
Microsoft Security & Trust, announced March 26, 2026
enforcement begins April 2026 Patch Tuesday

Effective with the April 2026 servicing release, Windows 11 24H2, 25H2, 26H1, and Windows Server 2025 stop trusting kernel drivers signed via the deprecated cross-signed root program. Per LOLDrivers research, this affects more than 81% of all currently-tracked vulnerable signed drivers. Rolling out in evaluation mode first to surface compatibility issues before full enforcement.

Background — what cross-signing was and why it existed

Microsoft introduced the cross-signed root program in the early 2000s to enable third-party CAs to issue Windows-trusted kernel-mode code-signing certificates. Driver vendors got their certificates from third-party CAs (DigiCert, Verisign, Comodo, etc.); Microsoft cross-certified those CAs as trusted for kernel-mode signing. The system worked by having the driver's signature chain end in a Microsoft-recognized cross-cert.

The program was officially deprecated in 2021, but cross-signed drivers continued loading by default on all Windows versions because of compatibility concerns. The result: a 20-year tail of legitimately-signed kernel drivers from defunct vendors, expired certificates, abandoned products. Per LOLDrivers' analysis published in early 2026, more than 81% of currently-cataloged vulnerable signed drivers carry cross-signed certificates. The cross-signed trust path was, effectively, the foundation of the BYOVD attack ecosystem.

What changes in April 2026

From Microsoft's official Tech Community announcement (March 26, 2026) and Microsoft Learn documentation:

  1. By default, only WHCP-signed drivers load. Windows Hardware Compatibility Program signing — the modern path, where drivers are tested by Microsoft and signed via the Hardware Dev Center — becomes the only default-trusted source of kernel drivers.
  2. Microsoft maintains an explicit allow list for "reputable cross-signed drivers" that are widely deployed and where breaking them would cause widespread outages. The allow list is small and curated.
  3. Initial rollout in evaluation mode. The kernel monitors driver loads for a defined period (100 hours across 3 boot cycles for Windows 11, 2 boot cycles for Server 2025). If all observed drivers are trust-policy-compliant, enforcement activates. If any non-compliant driver loads, evaluation resets.
  4. Application Control for Business (the rebranded WDAC) supports a new policy type for environments needing custom signers. The policy must be signed by an authority in the device's Secure Boot Platform Key (PK) or Key Exchange Key (KEK) variables. This gives enterprises a controlled escape hatch for internal-only signed drivers, without globally re-enabling cross-signed trust.

Why this is the most significant single change

strategic implication for the BYOVD landscape

Per MagicSword/LOLDrivers research, more than 81% of all currently-tracked vulnerable signed drivers depend on the cross-signed trust path. Once enforcement activates on a given system, those drivers stop loading by default. The remaining BYOVD attack surface contracts to: WHCP-signed-but-vulnerable drivers (rarer, since WHCP testing catches some classes of bug); compromised current certificates (requires active credential theft); and stolen certificates from current vendors (requires supply-chain compromise). The barrier to entry rises substantially.

For purple team operators in mid-to-late 2026: the LOLDrivers catalog you've been using will lose effectiveness rapidly on systems that have completed evaluation mode. Plan accordingly. The remaining viable drivers will be a small subset — and the WDB will be expanding to cover them too.

For detection engineers: the cross-signed-trust kill is partial protection, not a fix. Evaluation mode can take indefinitely long if there's a single non-compliant driver in the environment (per MagicSword analysis). Enterprises with old hardware, niche peripherals, or legacy software may stay in evaluation mode "indefinitely". Don't assume the policy is active just because the April 2026 update is installed. Monitor for the activation event in CodeIntegrity logs.

For attackers (purple side, in scope): the technique remains viable through 2026 against unpatched systems, and indefinitely viable against systems stuck in evaluation mode. But the long-term trajectory is a contracting attack surface — start planning for a world where BYOVD requires either a 0-day in a WHCP driver or a fresh credential theft.

Operational checklist

For detection / EDR engineers in 2026:

  1. Audit your fleet for systems where evaluation mode has activated vs. is still ongoing (CodeIntegrity event logs, specifically Event IDs 3076 and adjacent in Microsoft-Windows-CodeIntegrity/Operational).
  2. Identify legacy cross-signed drivers in your environment now — before enforcement activates and breaks production.
  3. Plan App Control for Business policies for any internal/confidential drivers that must continue to load.
  4. Don't relax your existing BYOVD detections — the policy is a long roadway, not an immediate fix, and many systems will be in evaluation mode for extended periods.

§ 08 HVCI, VBS, and the VTL1 direction

The hypervisor-tier defense and where Microsoft is taking it.

Hypervisor-Protected Code Integrity (HVCI), also called Memory Integrity, is the second layer of defense against BYOVD. From Microsoft Learn and the Veil Framework / Securityscientist 2026 BYOVD writeups:

What HVCI does

HVCI uses the Hyper-V hypervisor to run kernel-mode code integrity checks at VTL1 (Virtual Trust Level 1) — a more privileged context than VTL0 (where the normal kernel runs). The Code Integrity (CI) policy is loaded into a Secure Kernel running at VTL1; when the normal kernel tries to make a memory page executable, the request is sent to VTL1, which verifies the page contains only signed code matching the active CI policy. Pages with unsigned code, or with code matching a blocklist entry, are denied execute permission.

For BYOVD specifically: even if an attacker loads a vulnerable driver, achieves kernel write, and tries to write shellcode or modify protected kernel structures, HVCI's W^X enforcement at VTL1 prevents the write from creating an executable page. Many BYOVD chains either require an executable page (for shellcode-style payloads) or write to specific kernel structures that VTL1 also protects (PatchGuard structures, SSDT, IDT). HVCI doesn't block all BYOVD outcomes, but it raises the cost significantly.

Why HVCI isn't always on

  • Performance impact. Per Microsoft's own documentation and community benchmarks: HVCI imposes 5–15% overhead on syscall-heavy workloads. For latency-sensitive applications (gaming, real-time systems, virtualization), this is non-trivial.
  • Driver compatibility. HVCI requires drivers be HVCI-compatible (no self-modifying code, no executable pool allocations, etc.). Many older drivers fail. Microsoft maintains a compatibility list, but many production environments have at least one driver that's incompatible.
  • Default state varies. HVCI is on by default for Win 11 24H2+ Copilot+ PCs and most new Win 11 devices, but not on by default for upgrades from Win 10 or for devices with incompatible drivers detected during setup. Don't assume HVCI is active in any given environment — query the Device Guard Powershell APIs to verify.

Where this is going

Microsoft has been progressively pushing security-critical state into VTL1. Credential Guard moved LSA secrets into VTL1 in 2015. HVCI expanded VTL1 enforcement to driver code in 2017+. As of Win 11 25H2 / 26H1, additional kernel structures (token state, certain ETW provider registrations, parts of the object manager) are gradually moving toward VTL1 protection. The trajectory: over the next 2–3 years, the meaningful kernel attack surface for BYOVD will progressively shrink as more state migrates to VTL1, where even ring-0 code can't trivially modify it.

§ 09 Detection chokepoints in 2026

Six signals that catch BYOVD attacks, ordered by reliability.

1 · Driver hash lookup against LOLDrivers

Sysmon Event ID 6 (driver loaded) — every driver load gives you the hash. Automated lookup against LOLDrivers feed alerts on known-bad. Highest signal/noise ratio. Defeated only by drivers not yet in the catalog.

2 · The 4-step kill chain correlation

File drop (.sys) → service creation (SCM) → driver load → security process termination, all within a 5–10 minute window from the same host. Behavioral, technique-agnostic. Catches BYOVD regardless of which specific driver was used. Build this in your SIEM.

3 · EDR agent silence detection

Backend monitoring of agent last-seen timestamps. Alert on unexpected gaps. If your EDR goes silent and the host is still online, treat as compromised until proven otherwise. The detection happens off-box because the box may be compromised.

4 · ETW-TI DRIVER_LOAD with anomalous origin

Per vol. 11, ETW-TI emits driver load events with full context. Hunt for driver loads from unusual paths (%TEMP%, user-writable directories), or with timing patterns (driver loaded within seconds of high-privilege user logon). Lower signal, requires baselining.

5 · Kernel callback integrity check

EDR's own kernel driver periodically verifies its callback registrations are still active (vol. 10). Loss of callback indicates either driver unload (legitimate) or kernel-side tampering (BYOVD). EDR vendor tamper-protection logic. Defeated by attackers who patch quietly without unregistering.

6 · HVCI / WDB activation status monitoring

Track per-host whether HVCI is enabled and whether April 2026 cross-signed-trust enforcement is active vs in evaluation mode. CodeIntegrity event logs. Hosts not in enforcement mode get higher BYOVD risk score; pivot detections accordingly.

The pattern across all six: no single signal catches everything, and the high-fidelity ones (1, 2, 3) are independent of which specific driver is used. The theme of vols. 09–12 has been composite signals, and BYOVD is no different.

§ 10  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all without notes, you understand the kernel frontier.

Where does the PPL state of a process live, and what's its bit layout?
In EPROCESS.Protection, a PS_PROTECTION union: a single byte interpreted as bits 0–2 = Type (None/Light/Full), bit 3 = Audit (rarely set), bits 4–7 = Signer (Authenticode, CodeGen, Antimalware, Lsa, Windows, WinTcb, WinSystem, App). On Windows 11 24H2 the offset is 0x87a, on older builds it's at different offsets (0x878 on 22H2, 0x5fa on Win 10 1903). The Vergilius project tracks per-build offsets.
A SYSTEM-privileged process (token level: NT AUTHORITY\SYSTEM) tries to OpenProcess(MsMpEng.exe, PROCESS_VM_WRITE). What happens?
Returns STATUS_ACCESS_DENIED. Defender (MsMpEng.exe) runs at protection level 0x31 (Antimalware-Light), the requestor at protection 0x00. The kernel's PPL access check in PsTestProtectedProcessIncompatibility compares signer levels and refuses the open. SYSTEM token doesn't matter — PPL is a parallel access-control system that operates on EPROCESS.Protection, not on tokens.
What does a vendor have to do to make their EDR run at PPL Antimalware-Light?
Four things, gated by Microsoft: (1) write an ELAM driver and submit it to Microsoft for review; (2) receive a special anti-malware code-signing certificate from Microsoft after passing review; (3) sign both the ELAM driver and the user-mode service binary with the AM certificate; (4) configure the user-mode service to register as PPL via the SCM. Cannot be elevated to PPL post-creation through any documented API. The certificate gating is what makes "every commercial EDR is PPL Antimalware-Light" both possible and exclusive.
What is BYOVD, and why does kernel driver code-signing make it possible?
Bring Your Own Vulnerable Driver: load a legitimately-signed but vulnerable kernel driver onto the target, exploit a vulnerability in it (typically an arbitrary R/W primitive exposed through an IOCTL handler), gain ring-0 capability, then patch EPROCESS.Protection to disable PPL or unregister kernel callbacks. Possible because Windows kernel signing enforces "was this validly signed at any point?", not "is the certificate currently trusted, and is the driver bug-free?". A driver signed in 2018 by a now-defunct vendor with a now-expired certificate still loads on a 2026 system, unless explicitly blocklisted. Windows doesn't check CRL when loading drivers.
Name the four steps of the standard BYOVD kill chain.
(1) Drop the driver: WriteFile to a temp path. (2) Create kernel service: CreateServiceW(... SERVICE_KERNEL_DRIVER ...), registering an entry under HKLM\SYSTEM\CurrentControlSet\Services. (3) Load driver: StartServiceW, which calls NtLoadDriver in the kernel, leading to driver entry-point execution. (4) Communicate via IOCTL: CreateFile against the driver's device, DeviceIoControl with the malicious IOCTL code, achieving arbitrary kernel R/W. Steps 5–6 are post-exploitation: walk EPROCESS list, patch Protection on the EDR's process to 0x00, terminate or unregister the EDR.
What is the Microsoft Vulnerable Driver Blocklist (WDB) and why isn't it sufficient on its own?
A Microsoft-curated WDAC code-integrity policy file (DriverSiPolicy.p7b) at %windir%\System32\CodeIntegrity\ containing hashes and certificate rules for known-bad drivers. Enabled by default on Win 11 22H2+; enforced strongly when HVCI / Smart App Control / S Mode is on. Insufficient because: (1) updated only 1–2 times per year via cumulative updates, not continuously like AV signatures, leaving latency gaps; (2) covers a small fraction of LOLDrivers' catalog due to compatibility caution; (3) on systems without HVCI, certain WDAC rule qualifiers behave differently. Defenders should ingest LOLDrivers feeds directly to fill the gap, plus enable the ASR rule "Block abuse of exploited vulnerable signed drivers".
What is Microsoft's April 2026 cross-signed driver trust removal, and what's its impact on the BYOVD ecosystem?
Effective with the April 2026 servicing release, Windows 11 24H2/25H2/26H1 and Windows Server 2025 stop trusting kernel drivers signed via the deprecated cross-signed root program (a 2000s-era trust path that was deprecated in 2021 but kept active for compatibility). Per LOLDrivers research, 81%+ of currently-cataloged vulnerable signed drivers depend on this trust path. Once enforcement activates on a system (after evaluation mode completes — 100 hours across 3 boot cycles), those drivers stop loading by default. The remaining BYOVD surface contracts to WHCP-signed-but-vulnerable drivers, compromised current certificates, and stolen current vendor certificates. Microsoft maintains a small allow-list for widely-deployed cross-signed drivers, and provides PK/KEK-signed App Control policies as an enterprise escape hatch.
An attacker has loaded a vulnerable driver, achieved kernel write, and patched MsMpEng.exe's Protection byte to 0x00. Defender is now unprotected. What do they typically do next, and what's the best detection signal?
Typically: TerminateProcess on the now-unprotected MsMpEng.exe, OR call into the driver to unregister Defender's kernel callbacks (vol. 10) so it stops receiving telemetry without obviously dying. Best detection signal: EDR agent silence detection at the management plane. Backend monitoring of agent last-seen timestamps catches the attack regardless of which specific technique was used to disable the EDR — and crucially, it operates off-box, so it works even when on-box telemetry is gone. Pair with HVCI status monitoring to identify hosts at higher risk before an attack happens.

Twelve volumes. The kernel frontier, mapped.

The full ARC III synthesis

  1. Vol. 09 — Memory Manipulation Primitives. Where every byte that runs has been allocated. VirtualAlloc semantics, W^X / ACG / CFG / CET, sleep masks (Ekko, Cronos, FOLIAGE), pe-sieve and hollows_hunter as the scanning tools.
  2. Vol. 10 — Kernel Callbacks. The synchronous notification path. PsSetCreate*NotifyRoutine, ObRegisterCallbacks, CmRegisterCallback, minifilters, WFP. The 1500+ Mandatory-Access-Control surface that EDRs have. EDRSandblast as the canonical BYOVD-against-callbacks tool. PatchGuard and the WDB raising the cost.
  3. Vol. 11 — ETW Deep Dive. The asynchronous notification path. Provider/Session/Consumer/Controller. WMI_LOGGER_CONTEXT, EtwSiloState. ETW-TI as the kernel-side telemetry monopoly that powered EDR for a decade. The January 2026 SecurityTrace bypass that broke that monopoly.
  4. Vol. 12 — PPL & BYOVD. The trust mechanism that gates everything in vols. 10 and 11. The EPROCESS.Protection byte. The cross-signed driver loophole. The April 2026 Microsoft policy change that cuts 81% of the historical BYOVD attack surface.

The composite picture, in 2026:

An EDR exists as a PPL Antimalware-Light user-mode process backed by a kernel driver. The kernel driver registers callbacks (vol. 10) for synchronous visibility into process / thread / image / object events. The user-mode process consumes ETW-TI (vol. 11) for asynchronous high-fidelity events about kernel-sensitive operations. PPL (vol. 12) prevents non-PPL processes from tampering with either the user-mode side or its handles. Memory-integrity scanning (vol. 09) finds anomalies in process memory regardless of how they got there.

Each layer has its own bypass family. Vol. 09 is bypassed by sleep masks and call-stack spoofing. Vol. 10 is bypassed by BYOVD-driven callback-removal (EDRSandblast). Vol. 11 is bypassed at user-mode by patching EtwEventWrite/NtTraceEvent (commodity tradecraft) and at kernel-mode by SecurityTrace AutoLogger registration (Connor McGarr, January 2026). Vol. 12's PPL is bypassed by BYOVD with kernel-write achieving direct EPROCESS patching.

What remains true across all four is the composite signal model: any single layer can be defeated, but bypassing all four simultaneously requires kernel-mode capability AND careful evasion of the still-functional layers. The defender's job in 2026 is not to make any single layer impenetrable, but to ensure that bypassing any one of them leaves traces in the others. ETW-TI silencing is detected by callback events. Callback removal is detected by ETW-TI events. PPL stripping is detected by the EDR backend's agent-silence monitoring. Memory anomalies are detected by periodic scans regardless of how the memory got that way.

That's the kernel frontier as it actually is. Twelve volumes to map it because there's no shorter path that's honest.

§ 12 Closing thought, and what's next

ARC = thematic series of volumes

ARC I (vols. 01–04) gave you the foundations: data types, PE format, processes/threads, the loader. The alphabet, the skeleton, the running state, and the machinery that builds it.

ARC II (vols. 05–08) gave you the user-mode battlefield: syscalls, hooking, two-part process injection taxonomy. Where 80% of malware-vs-EDR fights happen, and where Hell's Gate, Halo's Gate, indirect syscalls, herpaderping, ghosting, AtomBombing, and module stomping all live.

ARC III (vols. 09–12) gave you the kernel frontier: memory primitives, callbacks, ETW, PPL/BYOVD. The defender's home turf, and the only remaining ground where a real engineering edge exists.

For purple team operators in mid-2026: the techniques in this volume are useful, but the timeline matters. BYOVD's golden age is ending. The April 2026 cross-signed trust kill removes 81% of the historical attack surface. By late 2026 / early 2027, plan for a world where BYOVD requires either a 0-day in a WHCP driver, a freshly-stolen current-vendor certificate, or operating against systems stuck in evaluation mode. The technique remains viable; the cost of viability is rising fast.

For detection engineers: the priorities are clear. Ingest LOLDrivers continuously (don't wait for WDB updates). Build the 4-step kill chain correlation in your SIEM. Set up backend agent-silence monitoring. Track HVCI status and April-2026-policy activation status per host. Layer detection across vols. 09–12, because no single layer is sufficient.

For EDR/XDR engineers: the four-pillar architecture (memory scanning + kernel callbacks + ETW-TI + PPL self-protection) is the production model. None of the four pillars is novel; the engineering edge is in how cleanly they're integrated, how quickly the backend correlates signals across them, and how robustly the on-box agent survives BYOVD. The McGarr SecurityTrace bypass and the cross-signed trust kill are both reasons to rethink your assumptions about who can see what — design as if the attacker can read your ETW-TI, and as if BYOVD will keep working for at least 18 more months on systems with old hardware.
