Memory Manipulation,
or: every byte that gets executed had to live somewhere first.

A walk through VirtualAlloc's real semantics, the W^X enforcement layers wrapped around it, the sleep-mask ecosystem that learned to hide RX memory, and the canonical defensive scanners that catch all of it.

blacksunCUBE field notebook · vol. 09 · opens ARC III; builds on vols. 02, 05, 06, 07, 08

ARC II was about composing primitives into techniques. ARC III is about understanding the substrates those primitives operate on. We start where everything starts: memory. Every shellcode that runs occupies pages. Every detection that catches it reads those pages' state. Every offensive technique we've covered through vol. 08 boils down to writing bytes to a region with executable protection and either getting a thread to execute them or making someone else's thread execute them.

Three things in this volume:

  1. The semantics — VirtualAlloc is a function people call without reading the documentation. We'll read it. RESERVE vs. COMMIT, what a "page state" actually is, what the protection flags really mean, and why PAGE_EXECUTE_READWRITE is the most-watched value in user-mode telemetry.
  2. The defenses — DEP, ACG, CFG, the W^X family. None of these prevent memory injection on their own; together they raise the cost. ACG (Arbitrary Code Guard) is the most aggressive of them, and worth understanding because Defender uses it for some of its own protected processes.
  3. The cat-and-mouse — sleep masks (Ekko, Cronos, FOLIAGE), how they encrypt their own pages between callbacks, and the defensive scanners (pe-sieve, hollows_hunter) that find the window where the pages are decrypted.

Same axes:

R · Read

Region inspection

Walking the VAD tree, calling VirtualQueryEx, reading page protections. The defender's primary tool. Also what attackers do to find their own loader, host modules, or call-stack-friendly addresses.

W · Write

Allocation & reprotection

VirtualAlloc, VirtualProtect, sleep-mask encrypt/decrypt cycles. The chokepoint where ETW-TI lives.

X · Execute

The page state at execution

What protection a page has when control is transferred to it. RX is normal; RWX is suspicious; RW with code in it is impossible (DEP throws); X with no R is unusual but legitimate for some hardening profiles.

§ 01 Memory primitives in two layers

Win32 wrappers that everyone calls; Nt-layer functions that actually do the work.

Vol. 05 introduced the user/kernel split. The memory APIs sit on top of that split exactly the way file APIs do: a thin Win32 layer in kernel32/kernelbase, calling Nt-layer functions in ntdll, which syscall into the kernel.

Win32Native (Nt)What it does
VirtualAlloc(Ex)NtAllocateVirtualMemoryReserve, commit, or both. Set initial protection.
VirtualFree(Ex)NtFreeVirtualMemoryDecommit a region or release a reservation.
VirtualProtect(Ex)NtProtectVirtualMemoryChange protection flags on already-allocated pages.
VirtualQuery(Ex)NtQueryVirtualMemoryInspect the state, type, and protection of a page.
WriteProcessMemoryNtWriteVirtualMemoryWrite to a region (your own or remote). Bypasses page protection if RWX or if a kernel-side write path is taken.
ReadProcessMemoryNtReadVirtualMemoryRead from a region. The defender's primary scanning primitive.

Two facts to anchor in this volume:

  • The Nt functions are the fundamental ones. Every Win32 wrapper resolves to one of these. ETW-TI hooks the Nt-layer kernel implementation, not the wrapper. Direct/indirect syscalls (vol. 05) bypass user-mode hooks but still hit the kernel-side Nt function and its telemetry.
  • Cross-process variants are different syscalls than local-process variants in some kernels. Specifically, NtAllocateVirtualMemory with a target process handle other than (HANDLE)-1 (NtCurrentProcess) fires ETW_TI_ALLOCVM_REMOTE; with the current process handle, fires ALLOCVM_LOCAL. The kernel knows local vs. remote and emits different events accordingly. This is why "writes to my own process's heap" is unwatched but "writes to my own process's loaded modules" is watched (different region type, even though both are "local").

§ 02 VirtualAlloc — the contract you didn't read

Three concepts: address space reservation, physical commitment, page protection.

LPVOID VirtualAlloc(...)
readwrite

"Reserve some address space, optionally commit some of it, and apply this protection."

LPVOID VirtualAlloc(
    [in, optional] LPVOID  lpAddress,        // NULL = system picks; non-NULL = you pick
    [in]            SIZE_T  dwSize,
    [in]            DWORD   flAllocationType, // MEM_RESERVE | MEM_COMMIT | MEM_RESET | ...
    [in]            DWORD   flProtect          // PAGE_NOACCESS, PAGE_READWRITE, PAGE_EXECUTE_READ, etc.
);

The three page states

memory page states · per Microsoft Memory API spec
Free
Address range not associated with anything. No physical storage. Touching it raises an access violation. The default state of every page in the address space at process startup.
Reserved
Address range claimed but no physical storage backs it. Touching a reserved page raises an access violation just like Free, but no other allocator can use the range. Used to claim contiguous virtual space in advance. MEM_RESERVE only.
Committed
Reserved and backed by RAM or the page file. Touching a committed page works (subject to its flProtect). MEM_COMMIT implies MEM_RESERVE if the range was previously free; can be applied to already-reserved range to commit it.

The Microsoft spec is explicit on the legal transitions:

  • MEM_RESERVE on a free range → reserved.
  • MEM_COMMIT on a reserved range → committed.
  • MEM_COMMIT | MEM_RESERVE on a free range → committed in one step. The most common combination.
  • MEM_COMMIT with a non-NULL lpAddress on an unreserved range → ERROR_INVALID_ADDRESS.
There's a gotcha here that catches new operators. MEM_COMMIT alone, with lpAddress = NULL, will succeed, because the system reserves and commits in one step when picking the address. With lpAddress non-NULL, it will fail if the range wasn't already reserved. Raymond Chen's "Old New Thing" called this out years ago: the documentation says MEM_COMMIT alone fails, and that's true for the explicit-address case but not the system-picks case. The clean idiom: always pass MEM_COMMIT | MEM_RESERVE together, then the function does the right thing regardless of whether you specified an address.

Why this matters offensively and defensively

From the offensive side: code injection allocates with MEM_COMMIT | MEM_RESERVE, flProtect = PAGE_EXECUTE_READWRITE, in a single call. That's the four-byte composite signature defensive scanners watch for. The "right" approach for stealthier loaders is staged: MEM_RESERVE first with PAGE_NOACCESS (essentially invisible), then MEM_COMMIT later with the real protection — ideally not RWX. Each step crosses fewer telemetry tripwires.

From the defensive side: NtAllocateVirtualMemory calls with PAGE_EXECUTE_READWRITE are the highest-fidelity single signal in the entire memory API surface. Microsoft's own Memory API documentation lists the protection constants — the writable-and-executable combinations are the ones every EDR rule keys on.

§ 03 Page protections and the W^X tax

RX is the default for code; RWX is the cost; RW for sleeping is the modern compromise.

Every committed page has a protection: a bit field describing which CPU operations are allowed against it. The relevant ones for offensive/defensive work:

page protection constants · what each one allows
ConstantReadWriteExecuteTelemetry weight
PAGE_NOACCESSnononolow — sleeping/reserved memory
PAGE_READONLYyesnonolow — string tables, .rdata
PAGE_READWRITEyesyesnolow — heap, stack, .data
PAGE_EXECUTE_READyesnoyeslow — normal code (.text)
PAGE_EXECUTE_READWRITEyesyesyesHIGH — RWX, watched by everyone
PAGE_EXECUTE_WRITECOPYyesyes (COW)yesHIGH — image .text after self-modification
PAGE_EXECUTEnonoyesunusual but legitimate (CFG hardening)

PAGE_EXECUTE_READWRITE is the constant that defines the entire detection landscape. Legitimate code is never both writable and executable simultaneously. Compilers produce RX .text; the loader maps it RX; the OS preserves that. The only programs that legitimately need RWX are JIT compilers (.NET, V8, Java HotSpot), and those announce themselves clearly via known process names.

The W^X family of mitigations

"W^X" (Write XOR Execute) is the architectural principle that no page should be both writable and executable at the same time. Windows enforces this through several layered mitigations:

  • DEP (Data Execution Prevention) — the CPU's NX bit. Pages without the EXECUTE protection raise STATUS_ACCESS_VIOLATION on instruction fetch. Hardware-enforced. On since Windows XP SP2.
  • ACG (Arbitrary Code Guard) — opt-in process mitigation. Disallows allocation of new executable memory and protection changes that would create writable+executable pages. Once enabled, you cannot allocate RWX, cannot promote RW to RX, cannot do anything that would let new executable code appear. Used by Edge, by Defender's protected services, and by hardened applications. The technique that breaks ACG is module stomping (vol. 08): no new executable memory is created, only existing executable memory is overwritten.
  • CFG (Control Flow Guard) — static-analysis-driven indirect-call validation. Compiler instruments indirect calls to check the target against a per-process whitelist of valid call targets. Bypasses are well-documented but increase loader complexity.
  • CET shadow stack (since 11th-gen Intel / Win 11 22H2 default) — CPU-level return address validation. Defeats most ROP-based bypasses. Increasingly common in practice.
why this matters in 2026

ACG-protected processes are a hard target. You cannot inject code into Edge by allocating RWX and writing to it — NtAllocateVirtualMemory with PAGE_EXECUTE_READWRITE just fails with STATUS_DYNAMIC_CODE_BLOCKED. You're forced to either find a non-ACG-protected target, or use module-stomping-style techniques against existing executable memory. CET makes ROP-based mitigation bypasses far more expensive than they were in 2018–2020. The trend in 2026 is clear: the OS gives the EDR more defensive primitives every release; the offensive techniques get more surgical to compensate.

§ 04 Inspecting memory: VirtualQuery and the VAD tree

How a defensive scanner walks a process's address space.

The address space of a Windows process is described by the Virtual Address Descriptor (VAD) tree — a kernel data structure listing every memory region: its base, size, protection, type, and backing object (if any). Each NtAllocateVirtualMemory, NtMapViewOfSection, or implicit allocation (heap growth, thread stack creation) adds or modifies a VAD node.

From user mode, you read the VAD indirectly via VirtualQueryEx:

SIZE_T VirtualQueryEx(...) → MEMORY_BASIC_INFORMATION
read

"Tell me about the memory region containing this address."

typedef struct _MEMORY_BASIC_INFORMATION {
    PVOID BaseAddress;       // start of region
    PVOID AllocationBase;    // the original allocation this region came from
    DWORD AllocationProtect; // initial protection at allocation time
    SIZE_T RegionSize;        // bytes in the region
    DWORD State;             // MEM_FREE / MEM_RESERVE / MEM_COMMIT
    DWORD Protect;           // CURRENT protection (may differ from AllocationProtect)
    DWORD Type;              // MEM_IMAGE / MEM_MAPPED / MEM_PRIVATE
} MEMORY_BASIC_INFORMATION;

The Type field is the most important one for detection:

  • MEM_IMAGE — backed by a mapped PE file (a loaded DLL or EXE). Has a known origin on disk. Can be hashed and compared.
  • MEM_MAPPED — backed by a non-PE file mapping (e.g., a data file mapped via CreateFileMapping).
  • MEM_PRIVATE — not backed by anything. Heap, stack, VirtualAlloc'd regions. Code in MEM_PRIVATE with executable protection is the textbook injection signature.

The basic shellcode scanning loop, used by every memory scanner from malfind onward:

// pseudo-code: scan all regions in a process for executable private memory
PVOID addr = NULL;
MEMORY_BASIC_INFORMATION mbi;

while (VirtualQueryEx(hProc, addr, &mbi, sizeof(mbi))) {
    if (mbi.State == MEM_COMMIT
     && mbi.Type  == MEM_PRIVATE
     && (mbi.Protect & (PAGE_EXECUTE | PAGE_EXECUTE_READ
                       | PAGE_EXECUTE_READWRITE
                       | PAGE_EXECUTE_WRITECOPY))) {
        // suspicious: executable, private, committed memory
        flag(mbi.BaseAddress, mbi.RegionSize, mbi.Protect);
    }
    addr = (PBYTE)mbi.BaseAddress + mbi.RegionSize;
}

That's the four-line algorithm at the heart of pe-sieve, hollows_hunter, Volatility's malfind, Moneta, and every other memory scanner. Variations add: hashing the region's first bytes against shellcode signatures, parsing PE headers if the bytes start with MZ, walking thread stacks to find call frames returning into private memory.

The reason sleep masks exist (next section) is to make this scan miss. If your shellcode is encrypted in PAGE_READWRITE memory while sleeping, a scan that runs during the sleep window finds nothing executable at all — or finds executable memory containing only the tiny stub that decrypts and re-encrypts. The full payload is never simultaneously plaintext and executable.

§ 05 Sleep masks: hiding RX memory between callbacks

The technique that addresses memory scanning by ensuring the suspicious bytes are never in a suspicious state.

An implant sleeps. Long-running C2 beacons typically sleep 30 seconds to 5 minutes between callbacks. During that sleep, a scanner can run, find the implant's RX shellcode region in MEM_PRIVATE, and dump it. Sleep masks make sure that during the sleep window, the bytes are encrypted and the protection is RW, not RX.

The basic sleep-mask cycle:

  1. Implant runs a callback (network I/O, command execution, etc.).
  2. About to sleep: VirtualProtect own pages from RX → RW.
  3. RC4-encrypt own bytes in place.
  4. Sleep for the configured duration.
  5. Wake: RC4-decrypt own bytes.
  6. VirtualProtect from RW → RX and continue.

The challenge: steps 4 and 5 happen on the same thread that's being protected. You can't put the encrypt-sleep-decrypt logic in the encrypted memory region itself, because then the decrypt code is inaccessible. The solution is to put the cycle's machinery somewhere else — this is what differentiates the various sleep mask implementations.

Three canonical implementations

Ekko
C5pider, 2022
used in Havoc C2

Use the timer queue (CreateTimerQueueTimer) to schedule a chain of callbacks: encrypt → sleep → decrypt. The current thread enters a wait state; timer-queue callbacks fire on a worker thread, which has access to the encrypt/decrypt routines via ROP gadgets that call NtContinue with controlled register state.

Earlier-generation technique that pioneered the encrypt-sleep-decrypt pattern using ROP chains directly, without timer-queue or waitable-timer abstractions. Closer to "Gargoyle" (2017) which is the conceptual ancestor. FOLIAGE's open-source clones power Havoc and several other modern C2s.

What sleep masks do not hide

Sleep masks address one specific detection: "is there RX private memory containing recognizable code right now?" They miss other detections:

  • Heap allocations — the implant's heap (containing C2 config, command queues, BeaconGate metadata) is typically not encrypted. Beacon's Sleep Mask Kit explicitly addresses this in its 4.x revisions; older masks left heap memory plaintext. Tools like BeaconEye scan heap for known structures.
  • Timer-queue heap signatures — WithSecure Labs (2022) demonstrated that CreateTimerQueueTimer calls produce a recognizable heap pattern: two linked allocations with predictable pointer chains. This is the signature Hunt-Sleeping-Beacons exploits.
  • Thread call stack analysis — pe-sieve's recent /threads mode (2024) walks every thread's call stack and flags any frame whose return address is in private memory. Sleep masks that hide code bytes don't hide the fact that some thread's call stack returns to a non-image region.
  • Working set bitmap — an undocumented kernel-side structure that Binary Defense's research (August 2025) demonstrated persists a bit indicating which private memory regions were ever marked executable. Even after the mask flips RX → RW, the working set bitmap retains the trace. Theoretical detection; not yet productized.
Detection in 2026: the most reliable single signal is thread call stack analysis — walk every thread's stack, classify each return address by MEM_IMAGE vs MEM_PRIVATE. Sleep-masked beacons typically have at least one thread whose call stack has a frame returning into private memory (the implant's loader). Combined with timer-queue heap signature scans (WithSecure technique) and BeaconEye-class heap structure scans, a defender can catch most current sleep-masked beacons. The arms race: 2024+ implementations (Beacon 4.10+ Sleep Mask Kit) explicitly clean up call stacks during sleep, addressing this detection. Nominally caught by Defender + thread-aware tooling; production reliability varies by implementation.

§ 06 pe-sieve and hollows_hunter

The two open-source defensive scanners every detection engineer should know.

Hasherezade's pe-sieve and hollows_hunter are the canonical reference implementations of memory scanning for Windows. They're worth understanding because (a) they're the tooling defensive analysts actually use during incident response, (b) commercial EDRs implement very similar logic internally, and (c) their detection categories map directly to the techniques we covered in vols. 06–08.

The relationship: pe-sieve scans one process; hollows_hunter scans the entire system, internally calling pe-sieve on each process. Both share the same detection engine and report format.

What they detect

IndicatorWhat it catches
Replaced PEProcess hollowing (vol. 07): on-disk image differs from in-memory image.
Implanted PEA second PE has been mapped into the process's address space without a corresponding loaded module record.
ShellcodesPrivate RX memory containing code-like byte patterns. Multiple categories (callable function, API hashing pattern, syscall stub pattern).
HooksIn-image patches: e.g., ntdll!Nt* bytes that don't match the on-disk image (vol. 06).
In-memory patchesAny deviation between mapped image bytes and on-disk file bytes — module stomping (vol. 08), module overloading, IAT modifications.
IAT hooksSpecific subset: import table entries pointing to addresses outside the imported module.
Thread anomaliesThreads with call stack frames in private memory; threads sleeping with suspicious wait reasons (vol. 08 WTH; sleep mask detection).

The detection engine is fully passive — it doesn't hook the target, doesn't inject anything, just reads memory. From the target's perspective, pe-sieve is invisible at the API level. From a kernel telemetry perspective: pe-sieve calls NtReadVirtualMemory against the target a lot, which is itself a signal an attacker can monitor for. Some advanced offensive loaders watch for memory-scanning behavior and respond by killing themselves.

Critical context for detection engineers: pe-sieve hints at suspicious indicators — it does not give a verdict. Hasherezade's own FAQ is explicit: "PE-sieve is a tool to detect code implants, not to make a judgment about them." A hooked function might be the work of an EDR; an implanted PE might be a legitimate plugin loader. The tool surfaces material; the analyst classifies. Production EDRs add behavioral context (parent process lineage, network behavior, sequence of operations) to convert raw signals into verdicts.

Recent additions (2024–2025)

The 2024–2025 release cycle of pe-sieve and hollows_hunter added several capabilities directly responding to the 2024+ injection techniques we covered:

  • Thread call-stack scan (/threads — pe-sieve, hollows_hunter v0.4.x): walks each thread's call stack and flags shellcode addresses. The defense response to sleep masks that flip RX → RW during sleep but leave call stacks pointing into the (now-encrypted) region.
  • ETW listener mode (/etw): hollows_hunter v0.3.7+ runs as a continuous ETW consumer, scanning processes when relevant kernel events fire (image load, memory protection change). Reduces blind-window: instead of polling, react to telemetry.
  • Improved call-stack reporting (PE-sieve v0.4.x): full call stack with symbol resolution for each detected anomaly. Makes triage faster — analyst sees "thread X is parked in WaitForSingleObjectEx with return address in private memory at +0x14fc".

§ 07 Detection chokepoints in 2026

Five signals, ordered by reliability.

1 · ETW-TI ALLOCVM / PROTECTVM (decisive)

Kernel-side events on NtAllocateVirtualMemory and NtProtectVirtualMemory. Local and remote variants distinguished. Fire regardless of direct/indirect syscall. The composite signal (allocate RWX, write, change to RX, execute) is high-fidelity for any technique that doesn't actively hide it.

2 · MEM_PRIVATE + EXECUTE detection

The classic VirtualQueryEx scan. Catches sleeping/stale shellcode, loader stages, RWX leftover from sloppy implementations. False-positive rate low (JIT processes are the only legitimate hits and known by name).

3 · Call-stack returns into private memory

For every active thread, walk the user-mode stack. Flag any return address in MEM_PRIVATE. Catches sleep-masked beacons even when bytes are encrypted, because the stack still shows the intended return target. Currently the most powerful cross-cutting signal.

4 · PE in-memory hash diff

For every loaded MEM_IMAGE region, compare the in-memory bytes to the on-disk file. Mismatches indicate hollowing, herpaderping, ghosting, module stomping, hooks. Slow to compute; usually triggered by other signals first.

5 · Timer-queue heap signatures

WithSecure's pattern (2022): identify heap allocations matching CreateTimerQueueTimer's internal layout (two linked allocations, predictable pointer chain). Catches Ekko-family sleep masks. Less generic than the others; high precision when applicable.

Note the pattern across all five: none of them rely on hooks in user mode. ETW-TI is kernel-side; VirtualQueryEx is a self-issued query of the OS state; call-stack analysis reads thread context; hash diff reads bytes. This is why sleep masks and direct syscalls don't actually hide the implant from a thorough scanner: the techniques target user-mode hook avoidance, but the scanners ask the kernel directly.

§ 08  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all without notes, you understand the memory layer.

What's the difference between a Reserved page and a Committed page, and what happens if you touch each?
A Reserved page has its address-space range claimed but no physical storage backing it — touching it raises an access violation, exactly like a Free page. A Committed page is reserved and backed by RAM or page file — touching it works, subject to its flProtect. MEM_RESERVE on a free range moves it to Reserved. MEM_COMMIT on a Reserved range moves it to Committed. MEM_RESERVE | MEM_COMMIT on a free range jumps directly to Committed (the most common combination in practice).
Why does VirtualAlloc(NULL, size, MEM_COMMIT, PAGE_READWRITE) succeed even though the docs say MEM_COMMIT alone fails on unreserved memory?
Because lpAddress = NULL tells the system to pick the address. When the system picks, it reserves and commits in one step internally — the "you must reserve first" rule applies only when you specify a non-NULL address pointing to unreserved memory. With NULL, MEM_COMMIT alone behaves equivalently to MEM_COMMIT | MEM_RESERVE. Raymond Chen's "Old New Thing" documented this; the safe idiom is to always pass MEM_COMMIT | MEM_RESERVE explicitly to avoid the case-dependent behavior.
What's the difference between MEM_IMAGE, MEM_MAPPED, and MEM_PRIVATE, and which one is the textbook injection signature?
MEM_IMAGE: backed by a mapped PE file (a loaded DLL/EXE). Has a known disk file, can be hashed and validated. MEM_MAPPED: backed by a non-PE file (e.g., a data file mapped via CreateFileMapping). MEM_PRIVATE: not backed by anything — heap, stack, or arbitrary VirtualAlloc. Code in MEM_PRIVATE memory with executable protection (PAGE_EXECUTE_*) is the textbook injection signature. The pe-sieve scan logic checks exactly this combination. Legitimate exception: JIT compilers (.NET, V8, Java) by known process name.
What does ACG (Arbitrary Code Guard) do, and why is module stomping its main bypass?
ACG is an opt-in process mitigation that disallows new executable memory creation and any protection change that would create writable+executable pages. Once enabled, VirtualAlloc with PAGE_EXECUTE_READWRITE fails with STATUS_DYNAMIC_CODE_BLOCKED; VirtualProtect from RW to RX also fails. Standard injection becomes impossible. Module stomping (vol. 08) is the bypass because it does not create executable memory — it overwrites bytes inside memory that is already RX (a legitimate DLL's .text). ACG checks operations, not byte content of existing executable regions.
Walk through what an Ekko sleep cycle does to its own memory pages.
(1) Implant runs callback. (2) Schedules a chain of timer-queue callbacks via CreateTimerQueueTimer. Each callback is NtContinue with controlled register state, effectively driving a ROP chain on a worker thread. (3) Callback 1: VirtualProtect(RX → RW). (4) Callback 2: SystemFunction032 RC4-encrypts own memory. (5) Implant thread is parked in a wait. (6) After sleep duration: callback 3 RC4-decrypts. (7) Callback 4: VirtualProtect(RW → RX). (8) Implant thread wakes; runs next callback. During steps 4–6, a memory scanner sees only encrypted bytes in RW memory — not a valid shellcode signature.
Sleep masks hide RX memory between callbacks. Name three things they don't hide that current detection tools exploit.
(1) Thread call stacks: even when bytes are encrypted, the implant's thread has a return address pointing into private memory. pe-sieve's /threads mode walks every thread and flags this. (2) Timer-queue heap signatures: CreateTimerQueueTimer produces two linked heap allocations with predictable pointer patterns. WithSecure's research (2022) demonstrated this is signaturable. (3) Heap structures: implant config, command queues, BeaconGate metadata. Older sleep masks left these plaintext. Beacon Sleep Mask Kit 4.x explicitly handles heap encryption; many homemade implementations do not.
An EDR is hooking NtProtectVirtualMemory in user mode. An attacker uses a direct syscall to bypass the hook. Does this evade the protection-change detection?
No. The user-mode hook is bypassed, but the kernel-side ETW-TI PROTECTVM_LOCAL (or PROTECTVM_REMOTE) event fires inside the kernel implementation of NtProtectVirtualMemory, regardless of how the syscall was invoked. The event includes target base address, region size, old protection, new protection, source PID. An EDR consuming ETW-TI sees a protection change to RWX or RX with write — high-fidelity IOC. Same lesson as vol. 05: the events fire in the kernel; user-mode evasion doesn't reach them.
What's the simplest scan logic that catches "shellcode in private memory" across all the techniques we've covered?
Walk every region in the process via VirtualQueryEx. For each region, check: State == MEM_COMMIT AND Type == MEM_PRIVATE AND Protect includes any PAGE_EXECUTE_* bit. Regions matching all three are private executable memory — the textbook injection signature. This is a four-line algorithm. It catches: classic remote thread, hollowing pre-PE-write, RWX shellcode loaders, sleep masks during their RX window. It misses: module stomping (memory is MEM_IMAGE, not MEM_PRIVATE), some sleep-masked beacons during the encrypted RW phase. False positives: known JIT processes by name. Foundation of pe-sieve and Volatility's malfind.

The competitive advantage of 2026 offensive tooling is composability and telemetry awareness; the competitive advantage of 2026 defense is composite signal scoring across layered chokepoints.
