Kernel Callbacks,
or: the function pointers an EDR's existence depends on.

A walk through every notification the Windows kernel offers a security driver — process, thread, image, handle, registry, file — the data structures they live in, and the BYOVD-class attacks that target those data structures directly.

blacksunCUBE field notebook · vol. 10 · builds on every prior volume; this is where the EDR really lives

Every previous volume has been a piece of a puzzle from the offensive side: how does an attacker run code, hide from telemetry, evade hooks. This volume is the defender's primary tool. When an EDR product says it "monitors process creation" or "watches for credential dumping" or "detects DLL injection", what it almost always means in practice is "my driver registered a callback for this kernel event, and I'm reading the data the kernel hands me."

That's not a metaphor. The Windows kernel exposes a small, well-defined set of callback registration APIs. Every commercial EDR (Defender for Endpoint, CrowdStrike Falcon, SentinelOne, Elastic Defend, Cortex XDR) uses essentially the same set. The differences between products are in what they do with the events — cloud correlation, behavioral scoring, response automation — not in where they get the data. The data sources are documented and finite.

Three things in this volume:

  1. The callback registration APIs — PsSetCreate*NotifyRoutine*, ObRegisterCallbacks, CmRegisterCallbackEx, minifilter registration, WFP callouts. Each one notifies a driver about a specific class of event, and each one has a maximum number of registrants and a kernel-side data structure storing the registrations.
  2. The kernel data structures behind those registrations. PspCreateProcessNotifyRoutine is an array. OBJECT_TYPE.CallbackList is a doubly-linked list. Knowing the layout matters because BYOVD attacks directly modify these structures from kernel-mode arbitrary-write primitives.
  3. The BYOVD attack family — once the attacker has kernel-mode write (via a vulnerable signed driver), what do they actually do? They walk these structures, identify EDR-registered callbacks by associated driver name, and zero out the entries. EDRSandblast is the canonical implementation; we'll walk through what it does.

Same axes:

R · Read

Event observation

The kernel calls into the driver. Driver inspects parameters (PID, image name, target object, requested rights) and decides what to do. Read-only path; no modification of kernel state.

W · Write

Modification or denial

For some callback families (Ex/Ex2 process, Ob handle pre-op, Cm registry pre-op), the driver can modify the operation: deny it, strip rights from a handle, alter registry data on the way through.

X · Execute

Where the callback fires

Each callback runs at a specific IRQL in a specific context. Process callbacks fire at PASSIVE_LEVEL. Object callbacks fire in the requesting thread's context. Registry callbacks fire mid-operation. The IRQL governs what the callback may safely do.

§ 01 The kernel notification model

A loaded driver registers an interest; the kernel calls back when the event happens.

Every callback in this volume follows the same conceptual pattern:

  1. A signed driver loads (boot-start, system-start, demand-loaded by a service, or via ELAM).
  2. The driver's DriverEntry calls one or more Ps*NotifyRoutine* / ObRegisterCallbacks / CmRegisterCallbackEx / FltRegisterFilter APIs, passing pointers to its callback functions.
  3. The kernel adds those pointers to the appropriate internal data structure.
  4. When a watched event happens (process created, handle requested, registry key modified, file accessed), the kernel iterates the data structure and calls every registered callback with event details.
  5. Each callback inspects, optionally modifies (depending on the API), and returns. The kernel proceeds with whatever it would otherwise have done.

The full registration surface, in the order we'll cover them:

§ 02 Process callbacks

The single most-used kernel notification in security software.

The Microsoft-documented prototype:

PsSetCreateProcessNotifyRoutineEx
readwrite

"Notify me when a process is created or exits. Let me block creation if I want."

NTSTATUS PsSetCreateProcessNotifyRoutineEx(
    [in] PCREATE_PROCESS_NOTIFY_ROUTINE_EX NotifyRoutine,
    [in] BOOLEAN                          Remove
);

// Callback prototype:
VOID CreateProcessNotifyEx(
    [inout] PEPROCESS              Process,
    [in]    HANDLE                 ProcessId,
    [inout, optional] PPS_CREATE_NOTIFY_INFO CreateInfo  // NULL on exit
);

// PS_CREATE_NOTIFY_INFO contains:
//   ParentProcessId, CreatingThreadId, FileObject, ImageFileName,
//   CommandLine, IsSubsystemProcess, CreationStatus
// CreationStatus can be set to a failure NTSTATUS to BLOCK process creation.

Three generations of this API matter, with subtly different capabilities:

APIYearDistinguishing capability

PsSetCreateProcessNotifyRoutineExVistaReceives PS_CREATE_NOTIFY_INFO with full image path and command line. Can block by setting CreationStatus.
PsSetCreateProcessNotifyRoutineEx2Win 10Same as Ex but also fires for WSL (PicoProcess) processes. Used by Defender for Endpoint to cover Linux subsystem activity.

The kernel-side data structure

Each entry is not a raw pointer — the low 3 bits are flag bits (used to differentiate Ex vs non-Ex registrations and similar metadata). To get the actual function address you AND with 0xFFFFFFFFFFFFFFF8. This is documented in EDRSandblast's source and in multiple research blogs.

The dispatcher function: nt!PspCallProcessNotifyRoutines. It walks the array, ANDs with the mask, and calls each non-null function.

EDRSandblast's "audit" mode dumps this exact array with symbolic resolution. Running it on a clean Windows 10/11 install with Defender enabled produces an output like the one above — you can see which kernel modules are watching process creation. This is the audit any defender should run when evaluating an EDR product: are the callbacks actually registered? Some lighter products advertise process monitoring but never show up in this array because they rely on user-mode hooks alone.

§ 03 Thread and image callbacks

Same model as process; different events; same kernel-side structure layout.

Two more Ps* callbacks complete the basic notification family:

Thread creation: PsSetCreateThreadNotifyRoutine(Ex)

NTSTATUS PsSetCreateThreadNotifyRoutine(
    [in] PCREATE_THREAD_NOTIFY_ROUTINE NotifyRoutine
);

// Callback:
VOID CreateThreadNotify(
    [in] HANDLE  ProcessId,
    [in] HANDLE  ThreadId,
    [in] BOOLEAN Create
);

Fires every time a thread is created or destroyed across the system. The callback receives source PID + TID + create/destroy flag. Thread creation events are extremely high-volume on a busy system — legitimate processes create and destroy threads constantly, so this callback alone produces low-fidelity signal. Combined with thread start address (available on the Ex variant), it becomes useful: a thread starting at an address in MEM_PRIVATE RWX memory is a strong injection signal.

Stored in nt!PspCreateThreadNotifyRoutine array, dispatched via nt!PspCallThreadNotifyRoutines. Same 64-callback limit, same low-3-bits-as-flags layout.

Image load: PsSetLoadImageNotifyRoutine(Ex)

NTSTATUS PsSetLoadImageNotifyRoutine(
    [in] PLOAD_IMAGE_NOTIFY_ROUTINE NotifyRoutine
);

// Callback:
VOID LoadImageNotify(
    [in, optional] PUNICODE_STRING FullImageName,  // can be NULL!
    [in] HANDLE          ProcessId,                  // 0 = driver loaded into kernel
    [in] PIMAGE_INFO     ImageInfo
);

Fires every time an executable image (EXE, DLL, or kernel driver) is mapped into a process. Receives the full image path, target PID (or 0 for kernel-mode loads), and image base + size + flags. This is the most reliable injection-detection signal on Windows because it fires at kernel mapping time, before any user-mode code in the new image runs.

a documented bug worth knowing about

FortiGuard Labs (formerly enSilo) disclosed in 2017 that FullImageName passed to load-image callbacks is sometimes invalid — specifically, the FILE_OBJECT.FileName buffer can be overwritten by NTFS rename operations between when the image is mapped and when the callback fires. A malicious actor can deliberately exploit this by mapping their image, then renaming the underlying file, causing the callback to receive an outdated path. The bug exists in every Windows version from 2000 through (verified) Win 10 Anniversary; defenders have to use alternative APIs like FltGetFileNameInformationUnsafe within the callback to get an authoritative path. If your detection logic trusts FullImageName blindly, it can be tricked.

Image-load limit history

The base PsSetLoadImageNotifyRoutine documentation page on Microsoft Learn says the maximum is 8 callbacks. This is the original NT-era limit and is technically still listed on the base page. However, the page explicitly notes that an April 2014 cumulative update for Windows 8.1 raised the limit from 8 to 64. The PsSetLoadImageNotifyRoutineEx page (Windows 10+) lists the limit as 64 directly. Every supported Windows version in 2026 has the limit-64 update applied. For 2026, treat the effective limit as 64 across all Ps*NotifyRoutine APIs.

§ 04 Object callbacks — ObRegisterCallbacks

Filter handle creation and duplication. The mechanism that lets EDRs intercept LSASS access.

ObRegisterCallbacks
Vista SP1+, Server 2008
canonical EDR LSASS protection

Intercept handle creation and duplication for Process, Thread, and Desktop objects. Pre-operation callbacks can modify the requested access rights; post-operation callbacks observe the result. The mechanism every modern EDR uses to detect "process X is trying to read LSASS memory."

The registration API

NTSTATUS ObRegisterCallbacks(
    [in]  POB_CALLBACK_REGISTRATION CallbackRegistration,
    [out] PVOID                     *RegistrationHandle
);

typedef struct _OB_CALLBACK_REGISTRATION {
    USHORT                     Version;       // OB_FLT_REGISTRATION_VERSION
    USHORT                     OperationRegistrationCount;
    UNICODE_STRING             Altitude;      // must be unique system-wide
    PVOID                      RegistrationContext;
    OB_OPERATION_REGISTRATION *OperationRegistration;
} OB_CALLBACK_REGISTRATION;

typedef struct _OB_OPERATION_REGISTRATION {
    POBJECT_TYPE              *ObjectType;   // PsProcessType, PsThreadType, ExDesktopObjectType
    OB_OPERATION              Operations;    // OB_OPERATION_HANDLE_CREATE | _DUPLICATE
    POB_PRE_OPERATION_CALLBACK  PreOperation;
    POB_POST_OPERATION_CALLBACK PostOperation;
} OB_OPERATION_REGISTRATION;

What the pre-operation callback can do

Per Microsoft Learn (and confirmed by Jonathan Johnson's reverse-engineering of ObpPreInterceptHandleCreate): pre-operation callbacks cannot outright deny a handle. They can only modify the requested access rights. If a process requests PROCESS_ALL_ACCESS on LSASS and the EDR's pre-callback strips everything except PROCESS_QUERY_LIMITED_INFORMATION, the calling process gets a handle — but a useless one. The handle creation succeeds; the access rights granted are reduced.

OB_PREOP_CALLBACK_STATUS PreOperationCallback(
    PVOID                          RegistrationContext,
    POB_PRE_OPERATION_INFORMATION  OperationInformation
) {
    if (OperationInformation->ObjectType == *PsProcessType) {
        PEPROCESS target = (PEPROCESS) OperationInformation->Object;

        // Is this a request to open LSASS?
        if (IsLsass(target)) {
            // Strip dangerous rights. Cannot deny entirely.
            OperationInformation->Parameters->CreateHandleInformation.DesiredAccess
                &= ~(PROCESS_VM_READ | PROCESS_VM_WRITE | PROCESS_VM_OPERATION);
        }
    }
    return OB_PREOP_SUCCESS;
}

The kernel-side structure

Object callbacks live in a different structure than process/thread/image callbacks. Each kernel object type — PsProcessType, PsThreadType, etc. — has an OBJECT_TYPE structure in kernel memory. The last field of this structure is CallbackList, a LIST_ENTRY (doubly-linked list) of CALLBACK_ENTRY_ITEM structures.

When ObRegisterCallbacks is called, the kernel walks the target object type's CallbackList, finds the right altitude position (per the altitude string the driver supplied), and inserts a new entry. Two drivers cannot register at the same altitude; that's why each EDR vendor publishes its own assigned altitude range.

When a handle creation or duplication happens, nt!ObpPreInterceptHandleCreate walks the same CallbackList and calls every registered pre-operation callback. ObpPostInterceptHandleCreate does the same for post-operation. This walk happens after the Security Reference Monitor's access check has already approved the operation — pre/post callbacks see only operations the SRM already permitted.

For purple team operators: Sysmon Event ID 10 is built directly on this mechanism. When you see a Sysmon "Process Access" event for LSASS with GrantedAccess: 0x1010, that's an Object callback firing in SysmonDrv.sys reporting the handle creation. The mask field in the event log is the granted access after any pre-operation callbacks ran — so if an EDR stripped rights, you'll see the reduced mask, not the requested one. Useful for triage: a request that asked for PROCESS_ALL_ACCESS but ended up with QUERY_LIMITED_INFORMATION means an EDR's Ob callback intercepted the request.

§ 05 Registry, file, and network callbacks

Three more callback families that round out an EDR's visibility.

Registry — CmRegisterCallbackEx

Registers a callback that fires for nearly every registry operation: key creation, value reads, value writes, deletions, enumerations, and roughly 60 distinct sub-operation types. The full list is in the REG_NOTIFY_CLASS enum.

NTSTATUS CmRegisterCallbackEx(
    [in]  PEX_CALLBACK_FUNCTION Function,
    [in]  PCUNICODE_STRING      Altitude,
    [in]  PVOID                 Driver,
    [in, optional] PVOID           Context,
    [out] PLARGE_INTEGER        Cookie,
    _Reserved_      PVOID           Reserved
);

Like ObRegisterCallbacks, registry callbacks have pre/post variants — pre-callbacks fire before the operation, post after. Pre-callbacks can fail an operation: returning STATUS_ACCESS_DENIED from the pre-callback causes the registry op to fail with that status. This is more powerful than Object pre-callbacks (which can only modify rights, not deny outright).

Used by EDR products to detect: registry persistence (Run keys, services, scheduled tasks); credential theft (SAM hive access); known bad value patterns (Cobalt Strike beacon configs occasionally written to registry).

File system — minifilters via FltRegisterFilter

The Windows Filter Manager (fltmgr.sys) provides a layered architecture for file system filtering. A driver registers as a minifilter with FltRegisterFilter, supplies a list of FLT_OPERATION_REGISTRATION entries (one per IRP type to filter, e.g., IRP_MJ_CREATE, IRP_MJ_WRITE), and the filter manager dispatches file operations through it.

FLT_OPERATION_REGISTRATION Operations[] = {
    { IRP_MJ_CREATE,        0, PreCreateCallback,  PostCreateCallback },
    { IRP_MJ_WRITE,         0, PreWriteCallback,   NULL },
    { IRP_MJ_SET_INFORMATION, 0, PreSetInfoCallback, NULL },
    // ... etc.
    { IRP_MJ_OPERATION_END }
};

Minifilter pre-callbacks can outright deny operations (return FLT_PREOP_COMPLETE with a failure status). This is how Defender's WdFilter.sys blocks file writes that match active threat signatures — it's a minifilter intercepting every IRP_MJ_WRITE and scanning the bytes before letting the write through.

Minifilters live in the FLT_FILTER structure tracked by fltmgr.sys. Enumerable from kernel mode (and from user mode via FltEnumerateFilters), with documented altitude ranges that vendors register for at WHQL signing time.

Network — Windows Filtering Platform (WFP)

WFP is the architectural successor to the older NDIS-level filtering. EDR drivers register callouts via FwpsCalloutRegister at specific filter layers (e.g., FWPM_LAYER_ALE_AUTH_CONNECT_V4 for outbound TCP connect, FWPM_LAYER_ALE_AUTH_RECV_ACCEPT_V4 for inbound). When a packet matches a configured filter, the kernel calls the registered callout function.

For EDR purposes, WFP is the canonical detection point for: outbound C2 connections, beacon traffic patterns, exfiltration attempts. Defender for Endpoint, CrowdStrike, and SentinelOne all register WFP callouts to inspect outbound network behavior.

§ 06 The kernel data structures behind it all

Why these structures matter to BYOVD attackers.

Every callback registration ultimately lives in one of three kinds of kernel data structures:

callback registration storage by API family
API familyStorage structureWhere it lives
Ps*NotifyRoutineFixed-size array of pointers (max 64)nt!PspCreate*NotifyRoutine globals
ObRegisterCallbacksDoubly-linked list (LIST_ENTRY)OBJECT_TYPE.CallbackList per type
CmRegisterCallback(Ex)Doubly-linked listnt!CallbackListHead global
FltRegisterFilterFLT_FILTER structureFilter manager's internal list
FwpsCalloutRegisterWFP callout listWFP engine in tcpip.sys

The two array-based structures (PspCreate*NotifyRoutine) are the simplest for an attacker with kernel write to manipulate: zero out the entries you don't want firing. The list-based structures require walking the list and unlinking nodes, which is slightly more complex but well within the capability of any BYOVD-class exploit.

Why arrays first, lists later

The Ps* APIs date to NT 4.0, when callback counts were small enough that a fixed-size array was a reasonable design. ObRegisterCallbacks arrived in Vista SP1 with the linked-list design that better supports the altitude-based ordering. CmRegisterCallback followed the same pattern. Modern callback APIs — if you were designing them now — would all use altitude-ordered lists. The arrays persist for backward compatibility.

§ 07 BYOVD: removing callbacks from kernel mode

EDRSandblast and the canonical attack against the entire callback model.

Once an attacker has kernel-mode read/write — usually via a Bring-Your-Own-Vulnerable-Driver (BYOVD) attack using a legitimately signed but exploitable driver — the entire callback model is exposed. Every structure described above is just memory; if you can write to that memory, you can remove EDR registrations.

EDRSandblast
Wavestone, 2021–present
canonical BYOVD callback removal

Loads a vulnerable signed driver to obtain kernel R/W primitives. Locates nt!PspCreateProcessNotifyRoutine, nt!PspCreateThreadNotifyRoutine, nt!PspLoadImageNotifyRoutine, and the relevant OBJECT_TYPE.CallbackLists. Identifies callbacks by their containing driver name (matching against a list of known EDR drivers). Zeros out matching entries. Optionally also disables ETW-TI by patching its ProviderEnableInfo.

The high-level algorithm:

EDRSandblast · canonical BYOVD callback removal flow
1
Load vulnerable signed driver (e.g., RTCore64.sys, gdrv.sys)
obtain kernel R/W primitive
2
Resolve nt!PsSetCreateProcessNotifyRoutine (via GetProcAddress)
user-mode anchor for kernel symbols
3
Disassemble PsSet... to find PspSet... offset, then PspCreateProcessNotifyRoutine array address
walk the call chain to find the array
4
For each entry: AND with 0xFFFFFFFFFFFFFFF8, look up containing module via MmGetSystemRoutineAddress equivalents
map array entries to driver names
5
Match against EDR driver list (WdFilter.sys, mssecflt.sys, CSAgent.sys, etc.)
identify which entries are EDR-registered
6
Use BYOVD R/W primitive to write 0x00 over EDR-registered entries
zero out the callback pointers
7
Repeat for thread, image, object callbacks; optionally disable ETW-TI
EDR is now blind to the relevant events

What this actually accomplishes

After step 6 completes, when a process is created, nt!PspCallProcessNotifyRoutines walks the array and skips the zeroed entries. Defender's callback no longer fires. Sysmon's no longer fires. The EDR's user-mode component is still loaded, still hooking ntdll, still consuming whatever ETW it has access to — but its kernel-side ground truth has been silenced. Combined with ETW-TI disable (separate primitive: patching EtwThreatIntProvRegHandle->ProviderEnableInfo), the EDR loses every kernel-side telemetry source it has.

EDRSandblast operates against three categories of EDR primitives in one tool: kernel notify-routine callbacks, object callbacks, and ETW-TI. The original disclosure (Wavestone, 2021) tested against three commercial EDR products successfully. The technique remains effective in 2026 against any product that doesn't actively monitor for tampering with its own callback structures — which is most of them. Microsoft's response has been to harden the BYOVD attack surface itself, via the Windows Vulnerable Driver Blocklist (default-on since Win 11 22H2), rather than protect the callback structures directly.

§ 08 Defense: PatchGuard, callback integrity, and the WDB

What stops BYOVD attacks, and what fails to.

The Windows kernel has several layers of self-protection:

PatchGuard (Kernel Patch Protection)

An undocumented, periodically-running integrity check introduced in Windows XP x64 / Vista. PatchGuard verifies that critical kernel structures haven't been modified: SSDT entries, IDT entries, MSRs, certain kernel function prologues. If it detects modification, it BSODs the system with bug check 0x109 (CRITICAL_STRUCTURE_CORRUPTION).

What PatchGuard does NOT cover: callback arrays. nt!PspCreateProcessNotifyRoutine is not in the protected list because it's designed to be modified at runtime by legitimate drivers. PatchGuard cannot distinguish a legitimate driver removing its own callback (during unload) from a malicious driver removing someone else's. The structure that BYOVD attackers target is the structure PatchGuard explicitly cannot protect.

Driver Signature Enforcement

Since Windows Vista x64, the kernel refuses to load unsigned drivers in normal boot mode. Test signing requires explicit configuration. This raises the bar for BYOVD: attackers must use a signed vulnerable driver, not an unsigned malicious one. The pool of usable signed-but-vulnerable drivers is finite; Microsoft's Vulnerable Driver Blocklist (WDB) ships with Windows 11 22H2+ as default-on and lists hundreds of known-vulnerable drivers that are blocked from loading.

The WDB approach has weaknesses: it only blocks known vulnerable drivers (researchers find new ones regularly); it can be disabled by an administrator; and ELAM-signed drivers from approved vendors are exempt. But it materially raises the difficulty of BYOVD attacks compared to the 2021–2023 era when essentially any signed driver could be used.

HVCI / Memory Integrity

Hypervisor-protected Code Integrity (HVCI) uses Hyper-V's VTL1 to enforce kernel code signing at runtime via the IOMMU. Even if an attacker achieves kernel write, they cannot execute new code in kernel mode — pages remain enforce-W^X. This neutralizes shellcode-based BYOVD payloads but does NOT protect callback structures (which are data, not code). An attacker on an HVCI-enabled system can still zero out callback array entries via data writes; what they cannot do is install their own malicious kernel code.

Microsoft Defender's "Tamper Protection"

A Defender-specific layer that monitors for modifications to its own configuration (registry settings, service state) and reverts changes from non-trusted callers. Does not currently cover callback array integrity — if an attacker zeros out Defender's WdFilter.sys callback entry, Tamper Protection doesn't notice because it doesn't watch that structure.

honest 2026 assessment

The callback model is fundamentally a kernel-data integrity problem. PatchGuard explicitly excludes it. HVCI doesn't cover data writes. Tamper Protection doesn't watch it. The defenses that exist — WDB and HVCI — raise the cost of BYOVD but don't close the structural attack. As long as a single signed-and-not-yet-blocklisted driver with kernel R/W exists, callback removal is achievable. Microsoft's long-term direction (visible in Win 11 25H2 and Win 12 previews) is to move sensitive structures into VTL1-protected memory, but as of 2026 this hasn't shipped for callback arrays. BYOVD callback removal remains an effective technique against most products.

§ 09  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all, you understand how an EDR actually sees the system.

Name the four canonical Ps* notification APIs and what each one watches.
PsSetCreateProcessNotifyRoutine{,Ex,Ex2} — process creation/termination; Ex adds PS_CREATE_NOTIFY_INFO with full image path + command line + ability to block via CreationStatus; Ex2 adds WSL coverage. PsSetCreateThreadNotifyRoutine{,Ex} — thread creation/termination; Ex adds thread start address. PsSetLoadImageNotifyRoutine{,Ex} — DLL/EXE/driver image map. All store entries in fixed-size arrays (nt!PspCreate*NotifyRoutine) of max 64 entries with low-3-bits flags.
Microsoft Learn says PsSetLoadImageNotifyRoutine max is 8. Is that the effective limit in 2026?
No. The base function reference page still lists 8 (the original NT-era limit), but explicitly notes that an April 2014 cumulative update for Windows 8.1 raised it to 64. The newer PsSetLoadImageNotifyRoutineEx page lists 64 directly. Every supported Windows version in 2026 has the limit-64 update applied. Treat the effective limit as 64 for all Ps*NotifyRoutine APIs.
In nt!PspCreateProcessNotifyRoutine, the array entries are not raw function pointers. What's the layout, and how do you extract the actual address?
Each entry is a kernel pointer with the low 3 bits used as flag bits (encoding things like which API generation registered the callback — Ex vs non-Ex). To get the actual function address, AND with 0xFFFFFFFFFFFFFFF8. This bitwise trick is documented in EDRSandblast's source and reproduced in multiple research blogs. The kernel does the same AND in nt!PspCallProcessNotifyRoutines when iterating the array.
What can a pre-operation Object callback do, and what can it NOT do?
It can modify the requested access rights on a handle creation or duplication — e.g., strip PROCESS_VM_READ from a handle being opened to LSASS. It cannot outright deny the handle. The Security Reference Monitor's access check has already approved the operation before the pre-callback fires; the callback only modifies the result. To block an operation entirely you need a different mechanism (process create callback's CreationStatus, or registry pre-callback's STATUS_ACCESS_DENIED return).
Sysmon Event ID 10 reports a process accessing LSASS with GrantedAccess: 0x1010. What's the technical chain that produced that event?
An attacker called OpenProcess(PROCESS_VM_READ | PROCESS_QUERY_INFORMATION, FALSE, lsass_pid). The call went through NtOpenProcess → ObOpenObjectByPointer → ObpCreateHandle. Inside ObpCreateHandle, after the SRM access check, ObpPreInterceptHandleCreate walked the PsProcessType.OBJECT_TYPE.CallbackList. SysmonDrv.sys's pre-callback was on that list, fired, observed the request, and returned. Sysmon's user-mode service later read the event from its driver and produced Event ID 10 with the granted access mask (0x1010 = PROCESS_VM_READ | PROCESS_QUERY_LIMITED_INFORMATION).
Why does PatchGuard not protect the kernel callback arrays?
Because the arrays are designed to be modified at runtime — legitimate drivers register callbacks at load time and unregister at unload time. PatchGuard's job is to detect modifications to structures that should not change (SSDT, IDT, certain kernel code). The callback arrays are explicitly excluded from PatchGuard's coverage because legitimate writes happen constantly. From a BYOVD attacker's perspective, this is the structural property that makes the technique viable: the data structure is mutable by design, so PatchGuard cannot distinguish legitimate from malicious modification.
EDRSandblast removes EDR callbacks from the kernel arrays. After removal, what telemetry does the EDR still have?
User-mode hooks (vol. 06) are still installed in ntdll, so the EDR's user-mode DLL still sees calls that go through ntdll — but only if the attacker doesn't direct-syscall around them. Standard ETW providers (non-TI) are still consumable. ETW-TI is also at risk — EDRSandblast's standard mode also patches the TI provider's ProviderEnableInfo to disable it. After full EDRSandblast execution: the EDR is effectively blind to process/thread/image creation, handle access events, and ETW-TI events. It has only its remaining user-mode visibility, which is bypassable via direct syscalls and call-stack spoofing. Cloud correlation can still notice anomalies, but local detection is largely silenced.
What is the Windows Vulnerable Driver Blocklist (WDB) and what does it actually block?
A Microsoft-maintained list of known-vulnerable signed drivers, shipped with Windows. Default-on since Windows 11 22H2 (and via Defender on earlier versions). When a driver in the list attempts to load, the kernel refuses. The list contains hundreds of entries including the most popular BYOVD targets (RTCore64.sys, gdrv.sys, various older Intel/AMD/MSI utility drivers). What it does not block: zero-day vulnerable drivers (added only after public disclosure), drivers from vendors with active ELAM signing, drivers used by enterprise software with admin-installed exemptions. The blocklist materially raises BYOVD difficulty but doesn't eliminate the technique — researchers add new vulnerable drivers to the public corpus regularly.
