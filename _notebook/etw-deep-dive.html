Event Tracing for Windows,
or: where the EDR's other half of the truth comes from.

An architectural walk through providers, sessions, and consumers the kernel-side data structures, the Threat-Intelligence provider that mature EDRs depend on, and the January 2026 disclosure that broke a 12-year defensive assumption.

blacksunCUBE field notebook · vol. 11 · builds on vols. 05, 06, 09, 10

Vol. 10 covered kernel callbacks the synchronous notification path. When a process is created, the kernel calls registered functions inside loaded drivers. Those callbacks fire in the kernel thread that's doing the work, the driver inspects, optionally modifies, and returns. The key word is synchronous.

This volume covers ETW the asynchronous one. Events are written to a buffer, a separate consumer process reads them later. The kernel doesn't wait. The producer doesn't know who the consumers are. It's a publish-subscribe system glued to the side of the kernel, designed originally for performance tracing and overloaded over time into the primary security telemetry channel for Windows.

Both vol. 10's callbacks and this volume's ETW are observable in user space, but the architectures are different and the attacks against them are different. Callbacks run inline so removing them blinds the consumer immediately. ETW sessions are decoupled, so you can disable the consumer without the producer ever knowing. Both matter; both have their own bypass families.

Three things to nail down here:

  1. The four-component architecture provider, session, controller, consumer. Drilled into Microsoft's own docs and reproduced in every blog written about ETW. Once you know what each one does, you can read any ETW disclosure in 10 seconds.
  2. The kernel-side data structures WMI_LOGGER_CONTEXT, EtwSiloState, EtwpLoggerContext array. These are where ETW lives in kernel memory. BYOVD attackers patch fields in WMI_LOGGER_CONTEXT to disable specific providers. Defensive forensics tools (JPCERT's etw-scan) walk these structures to recover dropped events.
  3. The Threat-Intelligence provider and the SecurityTrace flag what they're for, why ETW-TI was the gold standard for kernel telemetry from 2014 to 2025, and the January 2026 Connor McGarr disclosure that changed everything.

Same axes:

R · Read

Event consumption

The consumer side. Open a trace handle, attach to a session, receive events asynchronously. PPL processes (Antimalware-Light) get expanded access to "secure" sessions; everyone else gets filtered.

W · Write

Event emission

Provider calls EventWrite; data goes through the user-mode ETW APIs into the kernel via NtTraceEvent; kernel writes into the session buffer. This is the layer most user-mode bypasses target.

X · Execute

Where filtering decisions happen

At session-enable time (in EtwEnableTrace / EtwpEnableAutoLoggerProvider) for who can subscribe. At consume time (in EtwpQueryTrace) for who can introspect. The SecurityTrace flag governs both.

§ 01 The four-component architecture

Provider, session, controller, consumer. Memorize these before reading any ETW disclosure.

From Microsoft's own "Instrumenting Your Code with ETW" documentation, the four components and their responsibilities:

ETW architecture · data flow from producer to subscriber
Provider
Generates events. User-mode app, kernel driver, or kernel itself. Calls EventWrite → NtTraceEvent. e.g., amsi.dll, ntoskrnl, Microsoft-Windows-PowerShell, Microsoft-Windows-Threat-Intelligence
↓ events
Session
Kernel object. Holds buffers. Receives events from one or more enabled providers. WMI_LOGGER_CONTEXT in kernel memory. e.g., EventLog-Security, DefenderApiLogger, NT Kernel Logger
↓ buffered
Consumer
Application that reads events from a session, in real-time or from an .etl file. OpenTrace + ProcessTrace. e.g., Event Viewer, Defender, EDR consumer service, custom WPP analyzers
Controller
Starts/stops sessions, enables/disables providers in a session. Configures everything via the registry (AutoLoggers) or programmatically (StartTrace, EnableTraceEx2). e.g., logman.exe, Windows Performance Recorder, Defender's service initialization

Each component is an isolated unit of responsibility. Providers don't know who consumes their events. Consumers don't know which session they're attached to until they query. Controllers manipulate sessions externally. The decoupling is intentional: it's what allows the same provider (e.g., the kernel's process-creation events) to be consumed by Event Viewer, Defender, Sysmon, and CrowdStrike all at the same time, with different filters.

How many providers and sessions are running on a typical machine

From Nasreddine Bencherchali's primer (and verified on a clean Windows 11 install): "Windows comes pre-installed with over 1000 registered providers and at least a dozen trace sessions." Concretely:

§ 02 Provider registration and event emission

How an event makes it from a function call into the kernel buffer.

A provider's job is to register itself, then call EventWrite (or the Tracelogging variants) whenever it wants to emit an event. The actual call chain on Windows 10/11 x64:

// User-mode provider's API call:
EventWrite(RegHandle, &EventDescriptor, UserDataCount, UserData);

// Resolves to advapi32!EventWrite, then:
ntdll!EtwEventWrite
    → ntdll!EtwpEventWriteFull
        → ntdll!NtTraceEvent          // syscall transition
            → nt!NtTraceEvent            // kernel side
                → nt!EtwpProcessEvent     // dispatch into appropriate session
                    → write into WMI_LOGGER_CONTEXT's buffer

The user-mode chokepoints EtwEventWrite at ntdll! and the syscall NtTraceEvent are exactly what user-mode bypasses target. We'll come back to that in § 07.

Where providers live

A provider is identified by a GUID. Microsoft's manifest-based providers ship as .man XML files compiled into the binary that hosts them; the manifest tells consumers what events the provider emits and what fields they have. Modern providers use TraceLogging, which embeds the metadata directly in the binary as opaque blobs (Matt Graeber documented the format).

The kernel itself is a provider. The NT Kernel Logger session, when started, captures kernel-emitted events process create, thread create, image load, file I/O, registry, network via the Windows Kernel Trace classic provider. This is the oldest ETW path; modern security work uses the manifest-based providers like Microsoft-Windows-Kernel-Process instead, which provide finer-grained control and richer event data.

§ 03 Sessions and the kernel-side data structures

Where ETW actually lives in kernel memory.

Every active session is represented in the kernel by a WMI_LOGGER_CONTEXT structure. The list of all active sessions is kept in an array. From Trail of Bits' November 2023 reverse-engineering work and corroborated by Connor McGarr's 2026 research, the path to the array on a non-isolated process is:

// Kernel debugger expression that walks to the array:
((nt!_ESERVERSILO_GLOBALS*)&nt!PspHostSiloGlobals)
    → EtwSiloState
        → EtwpLoggerContext     // array of WMI_LOGGER_CONTEXT pointers
        → MaxLoggers           // size of the array

// On Windows 11 the array size is 0x50 (80 entries).
// Unused entries contain the value 1, not NULL.

Each WMI_LOGGER_CONTEXT structure contains everything the kernel needs to dispatch events to the session: name, security descriptor, flags, buffer list, registered providers, current state. The fields that matter for security analysis are SecurityDescriptor and Flags. The latter holds the SecurityTrace bit, which we'll cover in § 05–06.

How forensic tools recover events

JPCERT's etw-scan (a Volatility plugin published November 2024) walks the EtwpLoggerContext array, finds each WMI_LOGGER_CONTEXT, follows its GlobalList buffer field, and extracts events from the kernel buffer in memory. Even after an attacker stops a session and deletes the .etl file, the buffered events may still be in kernel memory until the buffer is reused. This makes ETW recovery a powerful forensic primitive.

For purple team operators: knowing this means an attacker who only stops the session and removes the file has not erased their telemetry the buffer contents persist briefly. Live-response tooling that snapshots kernel memory captures these residual events. Properly-engineered ETW evasion has to either operate at the user-mode emission point (so events never reach the buffer) or patch the buffer itself.

§ 04 The Threat-Intelligence provider, revisited

Vol. 05 introduced this. Now we open it up.

Microsoft-Windows-Threat-Intelligence ETW-TI is a kernel-side provider that emits events on a curated list of security-sensitive operations. The events fire inside the kernel implementation of the corresponding Nt* function, which means they're invariant to user-mode invocation method (vol. 05). Direct syscalls don't bypass them. Hardware breakpoints don't bypass them. Hooks at ntdll don't bypass them. The events fire when the kernel does the work, and the kernel always does the work.

The set of events ETW-TI emits, current as of Windows 11 25H2:

Event classFires when
ALLOCVM_LOCAL / REMOTENtAllocateVirtualMemory in current vs. another process
PROTECTVM_LOCAL / REMOTENtProtectVirtualMemory protection changes
READVM_LOCAL / REMOTENtReadVirtualMemory the "LSASS read" canary
WRITEVM_LOCAL / REMOTENtWriteVirtualMemory cross-process writes
MAPVIEW_OF_SECTIONNtMapViewOfSection section mapping into a process
QUEUE_USER_APCNtQueueApcThread(Ex) APC queueing across processes
SET_THREAD_CONTEXTNtSetContextThread thread RIP redirection
CREATE_THREAD_EXNtCreateThreadEx with start address details
DRIVER_LOAD / IMAGE_LOADkernel driver / image mapping events with full context

That list is a near-perfect map onto the offensive techniques covered in vols. 06–08. Every primitive used by classic injection, hollowing, doppelgänging, herpaderping, ghosting, AtomBombing, module stomping, sleep masks each one fires at least one ETW-TI event. The attack model that "we'll just direct-syscall to bypass the user-mode hook" is correct for the user-mode hook and irrelevant for ETW-TI.

Who is allowed to consume ETW-TI

From Wavestone's EDRSandblast research and confirmed in Microsoft's documentation: only processes running at PsProtectedSignerAntimalware-Light protection level (one specific PPL tier, requiring an Early-Launch-Anti-Malware signed driver vetted by Microsoft) can subscribe to ETW-TI. Defender, CrowdStrike Falcon, SentinelOne, Elastic Defend, Cortex XDR, all run their telemetry consumer at this protection level. From a regular admin process, attempting to enable the Threat-Intelligence provider via EnableTraceEx2 returns STATUS_ACCESS_DENIED.

This was the property that made ETW-TI trustworthy: a normal admin attacker, even one with kernel-mode capabilities short of a malicious PPL driver, couldn't watch what the EDR was watching. The attacker is blind to the same telemetry that's lighting up the EDR cloud. Until January 2026.

§ 05 AutoLoggers and the SecurityTrace flag

The kernel-side mechanism that bootstraps EDR telemetry at boot.

An AutoLogger is a special class of trace session configured in the registry and started automatically by the kernel at boot, before any user-mode services are running. Configuration lives in:

HKLM\SYSTEM\CurrentControlSet\Control\WMI\Autologger\<LoggerName>

Each AutoLogger key contains: enable settings, buffer configuration, and a list of provider GUIDs as subkeys. Concrete real-world AutoLoggers visible on a Windows 11 machine with Defender:

  • EventLog-Security classic Windows Security Event Log feed.
  • DefenderApiLogger Defender's primary kernel telemetry intake.
  • DefenderAuditLogger Defender audit-mode telemetry.
  • DiagLog, Diagtrack-Listener OS diagnostics.
  • Circular Kernel Context Logger rolling kernel-event buffer.

What makes AutoLoggers special the security implication

Per Connor McGarr's January 2026 reverse-engineering of EtwpEnableAutoLoggerProvider: AutoLogger sessions don't go through the standard EnableTraceEx2 path. They're started in the context of the System process effectively the kernel itself via EtwpEnableAutoLoggerProvider. Because the requesting "process" is the kernel, there's no Antimalware-Light check at provider-enable time; the kernel can enable any provider, including Microsoft-Windows-Threat-Intelligence and Microsoft-Windows-Kernel-Audit-Api-Calls.

To compensate for the missing enable-time check, the kernel sets a flag on the resulting session: the SecurityTrace bit in WMI_LOGGER_CONTEXT.Flags. The EtwpQueryTrace and EtwpEnableTrace code paths check this flag and refuse to operate on a "secure" session unless the requesting process is Antimalware-Light. That's the entire access-control model: enable-time check skipped because there's no process; consume-time check enforced via the SecurityTrace flag.

The SecurityTrace flag is automatically set by EtwpEnableAutoLoggerProvider when an AutoLogger configures either Microsoft-Windows-Kernel-Audit-Api-Calls or Microsoft-Windows-Threat-Intelligence as one of its providers. This is the indirect-via-AutoLogger method of getting the flag set; there's also a direct programmatic method (separate EnableTraceEx2 call path) and an undocumented direct WMI_LOGGER_INFORMATION.Flags manipulation. McGarr identified all three in his disclosure.

§ 06 The January 2026 disclosure: the SecurityTrace bypass

A 12-year-old defensive assumption, broken by reading Microsoft's own kernel code carefully.

SecurityTrace AutoLogger Bypass
Connor McGarr (Origin), January 2026
prelude PoC published days later

Register your own AutoLogger that subscribes to Microsoft-Windows-Threat-Intelligence. The kernel sets the SecurityTrace flag automatically because it doesn't check who configured the AutoLogger registry key just whether the provider GUID matches. After reboot, the AutoLogger is running. Consume from it without Antimalware-Light by abusing a separate gap in the consume-side check.

The disclosure is structured around three independent observations, each of which is an OS bug. Together they yield a working bypass.

Observation 1 AutoLogger configuration is privileged but not protected

Writing to HKLM\SYSTEM\CurrentControlSet\Control\WMI\Autologger\<Name> requires admin. It does not require Antimalware-Light. Any admin process can create a new AutoLogger key, populate it with the Threat-Intelligence provider GUID ({F4E1897C-BB5D-5668-F1D8-040F4D8DD344}), and reboot. The next boot, the kernel reads the registry, calls EtwpEnableAutoLoggerProvider for each provider, and (because the provider is TI) sets the SecurityTrace flag on the session. An admin attacker has now created an ETW-TI-emitting session entirely from user mode, before any PPL driver was involved.

Observation 2 the consume-time check has gaps

To consume a session marked SecurityTrace, the standard path requires Antimalware-Light. McGarr identified that the actual check happens in EtwCheckSecurityLoggerAccess, called from EtwpQueryTrace when the consumer queries session properties. The check is for query operations; the actual real-time event flow uses a different code path through EtwpStartTraceFromHandle and the consumer notification list. By going directly to that path with knowledge of the logger ID, a non-Antimalware-Light consumer can subscribe to events.

Observation 3 logger IDs are deterministic

The logger ID is needed to attach to a session via the real-time path. McGarr observed that AutoLogger logger IDs are assigned in deterministic order during boot new AutoLoggers get the next free slot, and the slot is consistent across reboots given the same configuration. Combined with knowledge of which AutoLoggers exist, an attacker can compute the likely logger ID without admin queries to the kernel.

prelude's github.com/preludeorg/ThreatIntelligenceConsumer repository (published days after the McGarr blog) packages this as a reference implementation. It's tested and working on Windows 11 24H2 and 25H2 as of disclosure.

why this matters the strategic implication

From 2014 (when the SecurityTrace mechanism was introduced) until January 2026, every commercial EDR vendor's threat model for ETW-TI was: "We are the only consumer because PPL Antimalware-Light gates the read side. Our detection rules can be sensitive because the attacker can't observe them." That assumption is now false. An admin attacker can read identical events to the EDR. They can profile which actions trigger which alerts, build tradecraft that avoids the watched syscalls, and update faster than the EDR vendor publishes new rules. This isn't a vulnerability that gets patched in a Windows update it's a structural redesign that requires Microsoft to add per-provider PPL checks at AutoLogger configuration time, which is a multi-month engineering effort with backwards compatibility implications.

For purple team operators in 2026: this is the most significant single offensive primitive disclosed in the last 24 months. Use it (within scope) to audit which detections actually fire on which actions; the visibility you gain is the same visibility the EDR has.

For detection engineers: assume your ETW-TI consumption is no longer secret. Defender for Endpoint shipped a partial mitigation in early 2026 (signature for the registry-write pattern that creates a TI-subscribed AutoLogger), but the bypass technique itself remains effective for attackers who can stage the registry write through a less-obvious channel or use one of the alternate enable methods McGarr described.

§ 07 User-mode ETW patching: the cheap, ubiquitous bypass

A separate problem from kernel ETW-TI. Affects user-mode emission, not kernel-side events.

Long before ETW-TI mattered, malware authors discovered that ETW emission in user mode goes through specific functions in ntdll: EtwEventWrite and EtwEventWriteFull. Both eventually call the syscall NtTraceEvent. Patching any of these in your own process disables ETW for that process. The kernel still fires its own events for kernel-side operations, but anything the application reports about itself (PowerShell script execution, .NET method calls, AMSI scans) is silenced.

The canonical patch: overwrite the first few bytes of EtwEventWrite with an instant return.

// x64 patch (single byte): jump in, immediately return.
   0xC3            ; ret

// Common variants seen in samples:
   0xC3                            ; bare ret (Mr-Un1k0d3r style)
   0x48 0x33 0xC0 0xC3              ; xor rax,rax; ret  (return success)
   0xC2 0x14 0x00                  ; ret 0x14  (cleanup stack on return)

// x86 patch:
   0x33 0xC0 0xC2 0x14 0x00         ; xor eax,eax; ret 0x14
   0xB8 0xFF 0x55                   ; mov eax, 0x000055FF (return-style stub)

The Turla backdoor (analyzed February 2025), Remcos RAT (February 2025), and dozens of commodity loaders all patch EtwEventWrite in their own process this way. It's a one-line evasion that defeats the AMSI-driven and .NET-driven detection categories. White Knight Labs's "Bypassing ETW for Fun and Profit" (referenced in 2025) documents the technique.

The escalation: patching NtTraceEvent instead

EDRs noticed the EtwEventWrite patch and started monitoring those bytes for tampering. The next step in the arms race was patching NtTraceEvent the syscall stub itself, one level down. Mr-Un1k0d3r's AMSI-ETW-Patch repository explicitly notes: "Instead of patching EtwEventWrite, simply patch the syscall NtTraceEvent which is called by a lot of functions." The advantage: a single patch site silences every ETW emission path in the process at once. The disadvantage: NtTraceEvent is part of the syscall surface and its bytes are equally watchable.

Patchless ETW bypasses

The most evasive approach avoids byte patching entirely. SharpBlock (CCob, 2020+) implements a patchless ETW bypass by hooking NtProtectVirtualMemory (in the parent process, against a child) to block the EDR's own DLL from successfully completing its initialization specifically, blocking the DLL from making its VirtualProtect calls during DllMain, which prevents it from setting up its hooks. The EDR's DLL technically loads but cannot install its instrumentation. Patchless because no ntdll bytes are modified.

Where these bypasses do and don't help

Bypass techniqueDefeatsDoesn't defeat
Patch EtwEventWriteAMSI calls, .NET runtime telemetry, application-level eventsKernel ETW-TI events, Microsoft-Windows-Kernel-* events
Patch NtTraceEventAll ETW emission from this process user and kernel APIsOther processes' events, kernel-emitted events about this process
SharpBlock (DLL block)EDR's user-mode hooks & ETW consumption from this processKernel callbacks (vol. 10), ETW-TI in EDR's own consumer

The pattern: user-mode ETW patching defeats user-mode-emitted events from your own process. It does not defeat kernel-emitted events about your process. Every kernel callback (vol. 10) and every kernel-side ETW event (TI, Kernel-Process, Kernel-Audit-Api-Calls) fires regardless of any user-mode patching.

§ 08 Detection chokepoints in 2026

Five signals that catch ETW-tampering, ordered by reliability.

1 · EtwEventWrite integrity scan

Periodic check of the first ~10 bytes of ntdll!EtwEventWrite against the on-disk image. Catches every commodity ETW patcher in one query. Defender for Endpoint runs this; Elastic Defend runs this. False positive rate: zero. Defeated by: NtTraceEvent patching (one level down) or patchless bypasses.

2 · AutoLogger registry monitoring

Watch for HKLM\...\WMI\Autologger\ writes via the Cm registry callback (vol. 10). Specifically alert on writes that add the Threat-Intelligence GUID to a non-Microsoft-managed AutoLogger. Direct counter to the SecurityTrace bypass. Defender added this signature in early 2026 per the McGarr disclosure followup.

3 · ETW session enumeration anomaly

Periodically enumerate active sessions via logman query or the equivalent ETW API. Alert on new sessions consuming privileged providers (TI, Kernel-Audit-Api-Calls) that aren't in a known whitelist. Catches the bypassed SecurityTrace consumer once it's running. Lower-priority alert; prone to false positives during legitimate driver installs.

4 · WMI_LOGGER_CONTEXT.Flags kernel scan

Forensic and IR-tier: kernel-mode scan of all active sessions for SecurityTrace-flagged sessions whose configuration was added recently. JPCERT's etw-scan Volatility plugin can do this offline. Production EDRs typically don't run continuous kernel-memory scans for this; it's a hunt-time signal.

5 · ProviderEnableInfo patching detection

EDRSandblast and similar BYOVD tools disable ETW-TI by patching the provider's ProviderEnableInfo field in kernel memory. Detection requires periodic kernel-memory checksumming of the TI provider's registration structure, which EDRs typically do as part of their tamper-protection. Combined with the WDB (vol. 10) raises BYOVD cost.

Note across the five: the first is user-mode (cheap, broad), the next three target specific bypasses, and the last is the kernel BYOVD chokepoint. No single signal catches everything; composite scoring across all five is the production model.

§ 09  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all without notes, you understand the second half of EDR's ground truth.

Name the four ETW components and their responsibilities.
Provider: generates events (e.g., amsi.dll, ntoskrnl, the TI provider). Session: kernel object that holds buffers; receives events from one or more enabled providers (WMI_LOGGER_CONTEXT in kernel memory). Consumer: application that reads events from a session in real-time or from an .etl file (e.g., Event Viewer, EDR consumer service). Controller: starts/stops sessions, enables/disables providers in a session (e.g., logman, Windows Performance Recorder, the kernel itself for AutoLoggers).
Why does ETW-TI fire regardless of how a syscall was invoked?
Because the events are emitted inside the kernel implementation of the corresponding Nt* function. The user-mode invocation method through hooks, direct syscall, indirect syscall, hardware breakpoint only changes how control enters the kernel. Once inside, every path runs the same Nt* kernel function, which contains an EtwTiLog* call near its critical section. The events fire there. User-mode evasion doesn't reach them.
Until January 2026, who could consume Microsoft-Windows-Threat-Intelligence?
Only processes running at PsProtectedSignerAntimalware-Light protection level one specific PPL tier requiring an Early-Launch-Anti-Malware (ELAM) signed driver vetted by Microsoft. Defender, CrowdStrike Falcon, SentinelOne, Elastic Defend, and Cortex XDR all run their telemetry consumer at this level. From a regular admin process, attempting to enable TI via EnableTraceEx2 returned STATUS_ACCESS_DENIED. This was the single most important property making ETW-TI trustworthy as defensive ground truth.
What is the SecurityTrace flag and how does it normally get set?
A bit in WMI_LOGGER_CONTEXT.Flags that marks a session as containing privileged provider data. When set, the session can only be queried/consumed by Antimalware-Light processes the kernel's EtwCheckSecurityLoggerAccess enforces this. Normally set by EtwpEnableAutoLoggerProvider when an AutoLogger configures Microsoft-Windows-Threat-Intelligence or Microsoft-Windows-Kernel-Audit-Api-Calls as one of its providers. Because AutoLoggers are started by the kernel itself (System process context), there's no caller-process check at enable time; the flag substitutes for that missing check.
What does Connor McGarr's January 2026 SecurityTrace bypass achieve, and what's the high-level mechanism?
An admin-only attacker (no kernel driver, no ELAM, no PPL) can read Microsoft-Windows-Threat-Intelligence events the same events Defender and other EDRs read. Mechanism: create a custom AutoLogger via registry write at HKLM\...\WMI\Autologger\MyName, add the Threat-Intelligence provider GUID as a subkey, reboot. The kernel's AutoLogger initialization sets SecurityTrace on the session because it sees the TI provider, but doesn't check who configured the registry. After reboot, the attacker uses a separate gap in the consume-time check (real-time path bypasses EtwpQueryTrace's EtwCheckSecurityLoggerAccess) to attach as a consumer despite not being PPL.
A piece of malware patches its own ntdll!EtwEventWrite with 0xC3. What does this defeat, and what does it not defeat?
Defeats: every ETW emission that goes through EtwEventWrite in this process AMSI scan calls, .NET runtime telemetry events, PowerShell script-block logging, application-level events. Does not defeat: kernel-emitted events about this process Microsoft-Windows-Kernel-Process, Microsoft-Windows-Kernel-File, ETW-TI. The kernel emits its events via separate code paths inside the kernel, not through user-mode ntdll. A typical commodity loader using only EtwEventWrite patching avoids AMSI alarms but still trips kernel-side detections at the syscall level.
Why is patching NtTraceEvent "better" than patching EtwEventWrite from an evasion perspective?
EtwEventWrite is one of many user-mode wrappers that emit events; the .NET runtime, AMSI, EventLog APIs, and others have their own emission paths that don't necessarily go through it. NtTraceEvent is the syscall every ETW emission from user mode eventually goes through it. Patching NtTraceEvent silences the entire user-mode ETW surface from this process in one operation. The downside: NtTraceEvent bytes are equally watched by the same EDR integrity-scan logic that watches EtwEventWrite; you've moved the patch site without making it harder to find.
An attacker stops an ETW session and deletes the .etl file. Are the events recoverable?
Often yes. Buffered events remain in kernel memory until the buffer is reused. JPCERT's etw-scan Volatility plugin (Nov 2024) walks the EtwpLoggerContext array, finds WMI_LOGGER_CONTEXT entries (including stopped ones with not-yet-freed buffers), and extracts the events from the GlobalList buffer field. Forensic capture of a memory image during or shortly after the incident often recovers the deleted telemetry. For an attacker, properly silencing ETW means either patching the user-mode emission point so events never reach the buffer, or zeroing the buffer in kernel memory directly via BYOVD.

§ 10 Closing thought

Eleven volumes in. ARC III is at three-quarters:

  • Vol. 09 memory: where every byte that runs has been allocated.
  • Vol. 10 kernel callbacks: the synchronous notification path.
  • Vol. 11 ETW: the asynchronous notification path, including the most significant single offensive disclosure of the last 24 months.
  • Vol. 12 (next, closes ARC III) PPL and BYOVD: the protection layer that gates ETW-TI consumption, the BYOVD-class attacks against it, and the 2025/2026 mitigations.

The two-mechanism model from vols. 10 and 11 is now complete. Kernel callbacks are how the kernel actively notifies a driver. ETW is how the kernel passively publishes a stream that anyone with the right protection level can consume. Both feed into the EDR; both have their own bypass families; both are what an EDR vendor's marketing means when they say "real-time visibility into kernel-level activity."

For 2026, the practical synthesis: ETW-TI was the single most important defensive primitive of the previous decade and its read-side monopoly is now broken. The events still fire reliably, kernel-side, regardless of user-mode evasion. But the assumption that "only the EDR can read them" no longer holds, and detection rule design that depended on attacker invisibility needs to factor that in.

For purple team operators: the techniques in this volume are operationally useful. The user-mode ETW patches are baseline tradecraft. The SecurityTrace bypass is a reconnaissance primitive with no current public mitigation that fully prevents it use it within scope to see exactly what the EDR you're up against is watching for.

For detection engineers: design your rule pack as if the attacker can see your ETW-TI consumption. Sensitive alerts that depend on surprise are no longer reliably surprising. The composite-signal model from prior volumes combining ETW-TI events with callback events with cross-process write classification with call-stack analysis remains your competitive edge. No single signal is reliable in isolation; the combination still is.
