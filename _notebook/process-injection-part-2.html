Process Injection,
or: the techniques that arrived after the kernel learned to watch.

Six modern injection methods — thread hijacking, AtomBombing, module stomping, EDR-Preloading, and Outflank's Early Cascade — built for an era where the hooks themselves are the obvious thing not to trip on.

blacksunCUBE field notebook · vol. 08 (part 2 of 2) · builds on every prior volume; closes ARC II

Part 1 walked the seven foundational injection techniques — classic remote thread, hollowing, APC, doppelgänging, herpaderping, ghosting, early bird. By 2026, every one of them is reliably caught by mature commercial EDRs. The techniques in this volume are different: they're the responses to that detection, designed by researchers and offensive engineers from 2016 onwards but most active in the 2024–2025 window. Each one targets a specific weakness in modern detection pipelines.

The pattern across all six is the same: find a normal-looking primitive that a defender's call-stack heuristics will accept, and use it to do something the heuristics weren't built to look for. Thread hijacking re-purposes SetThreadContext. AtomBombing re-purposes the global atom table. Module stomping re-purposes a legitimate DLL's .text section. EDR-Preloading re-purposes an undocumented ntdll callback. Early Cascade re-purposes the loader's own dependency-loading loop.

Same axes, same rigor:

R · Read

Re-purposing

Each technique starts by reading existing process state — threads, modules, callbacks — and finds a primitive built for legitimate use that can be subverted.

W · Write

Indirect placement

Payload writes happen via unconventional paths: APC-driven memcpy, atom table copies, writes to .mrdata instead of new RWX. The cross-process write call gets avoided whenever possible.

X · Execute

Borrowed legitimacy

Execution always happens through a frame the EDR's call-stack analysis will recognize: a legitimate module's .text, a kernel-driven APC, a callback the loader was about to fire anyway.

§ 01 Where part 1 left off

A summary of the detection environment these techniques were designed to escape.

By the end of vol. 07, we established the five detection chokepoints that catch the foundational techniques in 2026:

  1. ETW-TI cross-process events — ALLOCVM_REMOTE, WRITEVM_REMOTE, PROTECTVM_REMOTE fire in the kernel regardless of how the call was invoked.
  2. Sysmon Event ID 25 (image tampering) catches in-memory PE drift from on-disk image.
  3. FileObject abnormality at process-creation callback catches doppelgänging, herpaderping, ghosting.
  4. Cross-process APC queueing catches Early Bird and basic APC injection.
  5. In-memory PE hash diff vs. on-disk catches anything that swaps PE content after creation.

The techniques in this volume each target a specific weakness in this lattice. Thread hijacking sidesteps cross-process write detection by abusing handles already granted. AtomBombing sidesteps cross-process write entirely. Module stomping sidesteps RWX private memory detection. EDR-Preloading sidesteps everything by running before the EDR loads. Early Cascade combines the previous two with a careful study of which EDRs initialize when.

Two new defensive primitives become decisive in this volume:

  • Kernel-side call-stack capture on every sensitive syscall (Elastic 8.8+, Defender for Endpoint, CrowdStrike). The EDR walks the full call stack at the point a syscall fires and asks: does each frame's return address resolve to a backed image, or to private memory? Module stomping was designed precisely to defeat this. Early Cascade was designed precisely to fire before this telemetry pipeline is initialized.
  • EDR initialization-timing visibility. Mature EDRs in 2025/2026 publish (internally) when in process creation their detection pipeline becomes active. Outflank's Early Cascade research mapped this for several "top-tier EDRs" and found exploitable gaps before kernel32.dll initialization completes.

§ 02 Thread Execution Hijacking

Don't create a new thread — commandeer an existing one.

Thread Execution Hijacking
MITRE T1055.003
~2010 (mature), still active 2026

Find an existing thread in the target. Suspend it, write payload, change its instruction pointer, resume. No new thread is created — no CreateRemoteThread telemetry fires.

canonical flow · classic thread execution hijacking
1
CreateToolhelp32Snapshot + Thread32First/Next
enumerate target's threads
2
OpenThread(THREAD_ALL_ACCESS, FALSE, tid)
obtain handle to victim thread
3
SuspendThread(hThread)
freeze the victim
4
VirtualAllocEx(hProc, NULL, size, PAGE_EXECUTE_READWRITE)
RWX in target
5
WriteProcessMemory(hProc, addr, shellcode, size, NULL)
copy shellcode
6
GetThreadContext + ctx.Rip = addr + SetThreadContext
redirect victim's instruction pointer
7
ResumeThread(hThread)
victim runs shellcode in its own context

The structural advantage: no thread creation event fires. PsSetCreateThreadNotifyRoutine never sees this. The victim thread continues running — just at a different address than where it was suspended. From a process-tree view, nothing new happens.

The structural disadvantage, well known since Elastic's 2017 survey: suspending a thread mid-syscall causes the kernel to roll back, but suspending it mid-NTDLL can corrupt thread state when resumed. Mature implementations check ctx.Rip against the address range of ntdll.dll before redirecting, and retry if the thread is currently in a syscall. Without that check, the technique sporadically crashes the host process, which itself is a detection signal.

What fires in 2026: NtSetContextThread on a remote thread fires Microsoft-Windows-Kernel-Audit-API-Calls ETW events (vol. 06). The combination of (a) OpenThread with THREAD_SUSPEND_RESUME | THREAD_GET_CONTEXT | THREAD_SET_CONTEXT rights cross-process, (b) NtSuspendThread on a victim, (c) NtSetContextThread with Rip pointing to a private memory region, all within milliseconds — that's a high-fidelity composite signal. Defender for Endpoint and Elastic Defend ship rules for it. Detection in 2026: reliable when call-stack capture is enabled.

§ 03 Waiting Thread Hijacking

Same goal as classic thread hijacking, but no SuspendThread, no SetThreadContext.

Waiting Thread Hijacking (WTH)
no formal MITRE ID yet
2025 (Check Point Research, hasherezade)

Find a thread that's already waiting (not running) — e.g., parked in WaitForSingleObjectEx. Modify its return address on the kernel stack via stack-frame writes, not via SetThreadContext. When the wait returns, the thread "returns" to your shellcode.

This is the most important new technique in vol. 08. Disclosed by hasherezade at Check Point Research in April 2025, WTH addresses the central weakness of classic thread hijacking: the SetThreadContext call, with its Microsoft-Windows-Kernel-Audit-API-Calls telemetry, is itself the highest-fidelity IOC.

The insight: most processes have threads that spend nearly all their time blocked in waits. explorer.exe alone has dozens. A thread blocked in WaitForSingleObjectEx has its return address on its kernel-side stack — visible via ReadProcessMemory if you can identify the stack region. Overwriting that return address through WriteProcessMemory redirects execution when the wait completes. No suspend, no context modification.

why this matters as a research milestone

WTH is the first widely-discussed injection technique that completely avoids the thread-state-modification primitives EDRs have built rules around since 2018. The Check Point disclosure noted: "WTH avoids using APIs that trigger most alerts, such as SuspendThread/ResumeThread and SetThreadContext." It still requires cross-process write, so ETW-TI WRITEVM_REMOTE fires — but the write target is a stack address, not RWX private memory or an image's .text, which is a less-watched region. Detection engineers in 2026 are still calibrating signal-to-noise on stack-region writes.

What fires in 2026: NtWriteVirtualMemory cross-process fires regardless — that's the unavoidable signal. The novelty is that the target address looks innocuous (a stack region in a legitimate thread, not RWX or .text). Detection engineers are responding by adding "writes to thread stack memory" as a new IOC category. Defender for Endpoint added detection signatures specifically for stack-region writes in late 2025; Elastic Defend's rule pack covers it as of 8.16. Detection in 2026 is currently asymmetric — mature EDRs have caught up; lighter products may still miss it.

§ 04 AtomBombing

Use the global atom table as a write-what-where primitive — without WriteProcessMemory at all.

AtomBombing
MITRE T1055.014
2016 (enSilo / Tal Liberman)

Store shellcode in the global atom table via GlobalAddAtom. Use APC to make the target process call GlobalGetAtomName, which copies the shellcode into the target's own memory space. Combine with ROP to flip permissions and execute.

This was the first widely-publicized injection technique that did not use WriteProcessMemory. The atom table is a system-wide string-storage facility, accessible across processes on the same window station. Adding an atom from process A and reading it from process B copies the data via the kernel; from the target's perspective, no remote write occurred.

canonical flow · AtomBombing
1
GlobalAddAtomA(shellcode_bytes)
store payload in global atom table
2
find a target thread + RWX/RX code cave in target process
where the atom name will be copied
3
NtQueueApcThread(hThread, GlobalGetAtomNameA, atom, dest, size)
queue APC: target calls GlobalGetAtomName itself
4
NtQueueApcThread(hThread, ROP_chain) — flip page to RX, return into shellcode
DEP bypass via target's own ROP gadgets
5
— wait for thread to enter alertable state —
APCs fire, payload executes
AtomBombing's reception was mixed. enSilo presented it as a major novel technique in 2016; Carbon Black's response paper called it "largely over-hyped" because the execution still relied on NtQueueApcThread, which by 2016 was already a heavily-watched primitive. Both views are partly right: the write primitive was novel and remains hard to detect; the execution primitive was conventional. In 2026 terms: AtomBombing avoids one IOC (cross-process write) while introducing another (cross-process APC with arguments shaped like atom-table operations).
What fires in 2026: the cross-process APC pattern is the durable signal — NtQueueApcThread with source PID ≠ target PID, regardless of what the APC routine is. Some EDRs additionally hook GlobalAddAtom as a cheap secondary signal but it's high-noise (legitimate apps use atoms heavily). The key in 2026 is that AtomBombing-style techniques no longer need NtQueueApcThread — modern variants use SetWindowLongPtr + window subclass abuse instead, which is harder to ring-fence. Original AtomBombing (2016 PoC): caught reliably. Modern variants: harder.

§ 05 Module Stomping (DLL Hollowing)

Overwrite a legitimate, loaded DLL's .text with your shellcode. Calls now look like they originate from a signed Windows module.

Module Stomping / DLL Hollowing / Module Overloading
MITRE T1055 (variant); related to T1574
~2019 (Forrest Orr, William Knowles), 2024+ active in HijackLoader/IDATLoader

Load a benign signed DLL into the target (or use one already loaded). Overwrite its .text section with shellcode. Execute. Call-stack analysis sees frames inside amsi.dll or winmm.dll — legitimate modules — instead of unbacked private memory.

This is the technique that defeats kernel call-stack analysis as a detection method. The basic problem with classic injection: when a syscall fires, the EDR walks the call stack, asks "does each return address map to a known-good module?" If the answer for any frame is "no, this is private RWX memory," it's malicious. Module stomping makes the answer "yes, the frame is in winmm.dll" — even though the bytes at that address are now your shellcode.

canonical flow · module stomping
1
LoadLibrary("winmm.dll") in target (or find one already loaded)
benign signed DLL is mapped at known address
2
parse PE headers, find .text section RVA + size
determine where the existing code lives
3
VirtualProtectEx(hProc, .text_addr, size, PAGE_EXECUTE_READWRITE)
make .text writable
4
WriteProcessMemory(hProc, .text_addr, shellcode, size, NULL)
overwrite legitimate code with payload
5
VirtualProtectEx(hProc, .text_addr, size, PAGE_EXECUTE_READ)
restore RX so it looks like normal code
6
CreateRemoteThread(hProc, ..., addr_in_text_section, ...)
execute — return address is in winmm.dll!

Refined variants address remaining IOCs:

  • Module Overloading (Forrest Orr, 2019): instead of overwriting an existing DLL, map a fresh DLL into private memory yourself, then patch its .text. The payload-bearing DLL is "legitimately loaded" from the EDR's perspective — but the file backing it is one you wrote. Mitigates the integrity-check detection angle.
  • Module Shifting (naksyn, 2024+): write the payload to a non-.text section of the hosting DLL (e.g., .rdata if it has executable hosts, or pad-aligned regions in .text). Reduces the byte-count of overwritten legitimate code, so behavioral diff detectors miss it.
  • Restoring on completion: after shellcode runs, the malware writes the original bytes back. The DLL appears unmodified to any post-incident scan.

HijackLoader (also known as IDATLoader, GHOSTPULSE, SHADOWLADDER) is the canonical real-world deployment. It's used by LummaC2, Rhadamanthys, SectopRAT, AsyncRAT, Amadey, QuasarRAT, Tofsee, Remcos. VMRay's 2025.2 release added explicit module-stomping detection driven directly by HijackLoader analysis; CrowdStrike's 2025 blog series on hooks and stomps covers the call-stack-aware detection logic.

What fires in 2026: the protection change on a non-image private region is rarely watched, but the protection change on an image's .text section is. NtProtectVirtualMemory targeting a memory region inside a mapped image, changing protection to include write, is the highest-fidelity signal — same chokepoint as the unhook detection in vol. 06. Sysmon Event ID 25 catches the in-memory image hash mismatch. Caught reliably by Defender, Elastic Defend, CrowdStrike. Module Overloading and Shifting variants reduce signal but don't eliminate it.

§ 06 EDR-Preloading

Run before the EDR loads, by hijacking a callback the loader fires while only ntdll is mapped.

EDR-Preloading
no formal MITRE ID
February 2024 (Marcus Hutchins / MalwareTech)

Hijack ntdll!AvrfpAPILookupCallbackRoutine in a freshly-spawned suspended process. When the new process resumes and LdrLoadDll begins loading dependencies, that callback fires — before kernel32.dll even loads. Your shellcode runs in a process where the EDR's DLL has not yet been injected.

Recall vol. 04's loader call chain. When a process starts, LdrInitializeThunk calls LdrpInitialize → LdrpInitializeProcess → (loads dependencies via LdrLoadDll) → (eventually) drains the APC queue and calls main. The EDR's DLL is loaded as one of those dependencies. Code that runs during the dependency-loading phase, before the EDR DLL is mapped, runs in a process where there are no hooks, no ETW user-mode subscriptions, no nothing.

Hutchins's insight (February 2024): inside LdrLoadDll, before kernel32.dll initialization completes, the loader calls a verifier callback through ntdll!AvrfpAPILookupCallbackRoutine — a function pointer in ntdll itself, gated by a boolean ntdll!AvrfpAPILookupCallbacksEnabled. The pointer is normally null and the boolean false. If you can write to the suspended process's ntdll data section to set the pointer to your shellcode and the boolean to 1, the loader will call your shellcode during its own dependency walk.

canonical flow · EDR-Preloading (Hutchins, Feb 2024)
1
CreateProcess("notepad.exe", CREATE_SUSPENDED, ...)
target process: only ntdll loaded so far
2
find ntdll!AvrfpAPILookupCallbackRoutine + ntdll!AvrfpAPILookupCallbacksEnabled in target's ntdll
resolve via own ntdll's exports/data
3
VirtualAllocEx(hProc) + WriteProcessMemory(payload)
place shellcode in target
4
WriteProcessMemory(callback_routine_addr, payload_addr)
point the callback at our shellcode
5
WriteProcessMemory(callbacks_enabled_addr, 1)
enable the callback
6
ResumeThread(hThread)
LdrLoadDll fires the callback before EDR.dll loads
why this is a structural breakthrough, not a tactical trick

Every previous injection technique we've covered targets a process that has already booted — meaning the EDR DLL is loaded, hooks are placed, and the EDR's user-mode telemetry pipeline is alive. EDR-Preloading attacks before any of that is true. It runs in a window of milliseconds when only ntdll is mapped. There are no user-mode hooks to bypass because no user-mode hooks have been installed yet. This is qualitatively different from every technique in part 1 and the earlier sections of part 2.

The key constraint: the parent process — the one creating the suspended child — is still being watched. The cross-process writes still fire WRITEVM_REMOTE in the parent's context. So EDR-Preloading evades detection in the child at the cost of generating telemetry in the parent. If the parent itself is unhooked or pre-existing-trusted, this is a win. If the parent is the very payload the user just downloaded, the writes still get caught.

§ 07 Early Cascade Injection

EDR-Preloading + Early Bird, fused. The state of the art in 2024–2025.

Early Cascade Injection
no formal MITRE ID
October 2024 (Guido Miggelenbrink, Outflank)

Combine EDR-Preloading's pre-EDR-load timing with Early Bird's APC-driven execution — but without cross-process APC queuing. Inject a two-part payload, hijack the shim engine callback, fire the second part in the child with no parent-side APC visible.

Outflank's research (October 2024) generalizes EDR-Preloading. Hutchins's original used the verifier callback specifically; Outflank's variant uses the shim engine callback path inside ntdll's LdrLoadDll — a path that fires reliably on every process startup, with the same "before EDR loads" timing.

The structural improvements over EDR-Preloading and Early Bird:

  • No cross-process APC. Early Bird's signature IOC (NtQueueApcThread from parent to suspended child) is gone. The execution trigger is a callback inside the child's own ntdll, fired by the loader itself, not by an external APC.
  • Writes go to .mrdata / .data. Both sections are writable by default in ntdll. The protection-change call (NtProtectVirtualMemory) that classic injection requires is avoided. WRITEVM_REMOTE still fires, but the target region is a normal data section, not RWX private memory or an image's .text.
  • Two-stage payload. Stage 1 is tiny (just enough to hijack the callback). Stage 2 is the real payload, queued via in-process APC after stage 1 fires — so even if cross-process telemetry catches stage 1's setup, the actual malicious work happens via local APC, which is heavily backgrounded in normal operation.

Outflank's report tested Early Cascade against several "top tier EDRs" (vendor names redacted in their public post) and found it defeated user-mode detection in all of them. Multiple PoCs are publicly available: 0xNinjaCyclone/EarlyCascade (shim engine variant), Cracked5pider/earlycascade-injection, vvinoth/Early-Cascade-Injection.

What makes Early Cascade significant for 2026 detection engineering: it forces defenders to instrument the loader itself, not just the syscall layer. The cross-process writes to ntdll.data regions in a suspended child process are not on most EDR rule packs as of late 2025; the read-side telemetry is on the kernel-side WRITEVM_REMOTE events, and the rule writers have to add "writes to ntdll data sections" as a new hunt category. Defender for Endpoint shipped detection signatures for Early Cascade in early 2026; Elastic Defend's coverage as of 8.18 (March 2026) includes the technique. Standard CrowdStrike Falcon and SentinelOne coverage timelines are non-public but multiple incident response teams have observed signature additions.
What fires in 2026: in the parent — WRITEVM_REMOTE events targeting ntdll.data/ntdll.mrdata in a suspended child. Combined with absence of standard imports being mapped (no kernel32 yet), this is a near-unique pattern. The rule template: "any cross-process write to a known-image data section in a suspended target whose import resolution has not completed" → high-confidence Early Cascade. False positive rate is low because legitimate cross-process write at this stage of process startup is essentially nonexistent.

§ 08 Detection in 2026 — the call-stack era

Five chokepoints, refined.

Vol. 07 listed five detection chokepoints. By the techniques in this volume, two of them are obsoleted by sufficiently careful malware (in-memory PE hash diff is bypassable by module shifting/restoring; FileObject inspection only catches the old image-tampering family). The remaining three become more important, and two new ones become decisive in 2026:

1 · Kernel call-stack capture (decisive)

The single most important defensive primitive of 2024–2026. On every sensitive syscall, the kernel walks the user-mode call stack and identifies any frame whose return address resolves to private memory rather than a backed image. Module stomping was designed to defeat this; its detection signal moves into the protection-change pattern instead.

2 · Cross-process write target classification

Modern rules don't just detect NtWriteVirtualMemory cross-process — they classify the target region: private RWX, image's .text, image's data section, thread stack, or normal heap. WTH writes to thread stacks; Early Cascade writes to ntdll.data. Both are now distinct signal categories with their own rules.

3 · ETW-TI THREATINT_PROTECTVM_LOCAL on image regions

Protection changes on memory regions inside a mapped image — particularly .text sections — are the most reliable single signal in 2026. Module stomping and module overloading both trigger this. Early Cascade specifically avoids it by writing to already-writable sections.

4 · Process-startup phase awareness

EDRs in 2025/2026 increasingly track which phase of process startup the target is in (only ntdll mapped, kernel32 mapped, dependencies fully resolved, main running). Cross-process operations from external sources during the early phases is now suspicious by default. This rule class catches Early Cascade and EDR-Preloading.

5 · Behavioral chains across techniques

The strongest detections combine signals across multiple syscalls. "Suspended-child created by parent" + "parent writes to child's ntdll" + "child resumed" + "child immediately performs network I/O" is a higher-fidelity composite than any single signal. Defender for Endpoint, CrowdStrike, Elastic Defend, SentinelOne all use composite scoring.

§ 09  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all of these without notes, you have the full modern injection taxonomy memorized.

What's the structural advantage of Thread Execution Hijacking over classic remote thread injection, and what's its remaining IOC?
No new thread is created — PsSetCreateThreadNotifyRoutine never fires. The hijacked thread continues running in the legitimate process, just at a different address. Remaining IOC: NtSetContextThread on a remote thread fires Microsoft-Windows-Kernel-Audit-API-Calls ETW events, and the OpenThread with THREAD_SUSPEND_RESUME | SET_CONTEXT rights is a high-fidelity composite when paired with subsequent NtSuspendThread. Stable detection in 2026.
Waiting Thread Hijacking (April 2025) avoids two specific APIs that older techniques rely on. Which two, and what does it use instead?
Avoids SuspendThread/ResumeThread and SetThreadContext/NtSetContextThread — the two highest-fidelity IOCs of classic thread hijacking. Instead, it finds threads already blocked in waits (e.g., WaitForSingleObjectEx) and overwrites the return address on their kernel stack via WriteProcessMemory. When the wait completes, the thread "returns" to the attacker's shellcode. Cross-process write still fires telemetry, but the target is a stack region rather than RWX or .text.
In AtomBombing, how does shellcode get into the target process without WriteProcessMemory?
The attacker stores shellcode bytes in the global atom table via GlobalAddAtom in their own process. Then they queue an APC to a target thread pointing at GlobalGetAtomNameA (with NtQueueApcThread's extended argument form). When the APC fires, the target process calls GlobalGetAtomName on its own — which copies the atom data (the shellcode) into a destination buffer the attacker specified. The kernel handles the copy; from the EDR's viewpoint, no remote write happened. Execution still requires conventional APC + ROP, which is the part that gets caught.
Why is Module Stomping the technique that specifically defeats kernel call-stack analysis?
Kernel call-stack analysis asks "does each frame's return address resolve to a backed image?" Classic injection writes shellcode to private RWX memory, so when the shellcode runs, the call stack contains a frame in private memory — immediate red flag. Module stomping overwrites the .text section of a legitimate loaded DLL (e.g., winmm.dll). The shellcode now lives at an address inside a signed, mapped, expected module. The call stack frame resolves to winmm.dll — legitimate to a heuristic that doesn't check the bytes match the on-disk file.
Module Stomping has a remaining detection signal that mature EDRs catch. What is it?
NtProtectVirtualMemory targeting a memory region inside a mapped image with new protection that includes write access. Image .text sections are loaded as RX (read+execute) and never legitimately need to be written to after load. Any protection change to RW or RWX on an image region is high-fidelity malicious. ETW-TI PROTECTVM_LOCAL fires; Sysmon Event 25 catches the resulting in-memory hash mismatch. Module Overloading and Module Shifting variants reduce signal (writing to less-overwritten regions, or restoring afterward) but don't eliminate it.
In EDR-Preloading (Hutchins, Feb 2024), what specifically gets hijacked, and why does it run before the EDR DLL is loaded?
Two values in ntdll's data section: ntdll!AvrfpAPILookupCallbackRoutine (a function pointer) and ntdll!AvrfpAPILookupCallbacksEnabled (a boolean). The attacker writes the shellcode address to the pointer and 1 to the boolean from the parent process (target is in CREATE_SUSPENDED). When the target resumes, LdrInitializeThunk → LdrLoadDll begins loading dependencies. Inside that initialization path, before kernel32.dll initialization completes, the verifier callback fires — calling the attacker's shellcode in a process where only ntdll is loaded. The EDR's DLL isn't mapped yet, so no user-mode hooks exist to evade.
Early Cascade Injection (Outflank, Oct 2024) combines two prior techniques. Which two, and what's the structural improvement over each?
Combines Early Bird APC Injection and EDR-Preloading. Improvements: (1) over Early Bird — eliminates cross-process APC queuing; the execution trigger is an in-process callback fired by the loader, not an external NtQueueApcThread. (2) over EDR-Preloading — uses the shim engine callback path, which is more reliable than the verifier callback Hutchins used. Both improvements share a third trick: writes go to ntdll's .mrdata/.data (already writable), avoiding a protection-change call. The technique was effective against multiple "top tier EDRs" at disclosure time; Defender for Endpoint and Elastic Defend shipped detection in early-to-mid 2026.
By 2026, kernel call-stack analysis is the dominant EDR primitive. Which detection signals replace it as decisive when techniques (like module stomping) defeat call-stack analysis?
Three: (1) protection changes targeting memory inside mapped images — the chokepoint module stomping creates while trying to write to .text. (2) cross-process write target classification — private RWX vs. image .text vs. image data vs. thread stack are now separate signal categories with separate rules. (3) process-startup phase awareness — cross-process operations during the early phases of a child process's startup (only ntdll mapped) are intrinsically suspicious. Composite scoring across all five chokepoints (vol. 07 + this volume) is the actual production detection model.

§ 10 Closing thought — ARC II is now complete

Eight volumes in. ARC II (the user-mode battlefield) is now closed:

  • Vol. 01 — the alphabet (data types).
  • Vol. 02 — the static skeleton (PE format).
  • Vol. 03 — the living state (PEB / TEB / LDR).
  • Vol. 04 — the machinery that builds that state (the loader).
  • Vol. 05 — the border crossing (syscalls).
  • Vol. 06 — how the EDR rewrites bytes near that border, and how attackers undo it.
  • Vol. 07 — foundational injection composed from those primitives.
  • Vol. 08 — modern injection composed against the kernel-side telemetry that catches the foundational set.

A working purple team operator in 2026 should be able to read any new injection paper, identify which detection chokepoint it targets, predict which EDRs catch it and which don't, and design the corresponding hunt query. A working detection engineer should be able to look at any cross-process telemetry event and know which technique generated it. That's what ARC II was for. The composability is the point.

The honest 2026 state of play: against legacy AVs, every technique in vols. 06–08 still works. Against mature commercial EDRs (Defender for Endpoint, CrowdStrike Falcon, SentinelOne, Elastic Defend) running with kernel callstack capture and ETW-TI consumption enabled, the techniques in part 1 are reliably caught, and the techniques in part 2 are caught with delays of 6–18 months from disclosure to signature. Early Cascade Injection (October 2024) has had detection added to Defender by early 2026. Waiting Thread Hijacking (April 2025) has had detection added by late 2025. The detection arms race is real, and the vendors are keeping up.

For offensive operations, this means the lifespan of any single technique is roughly its time-to-disclosure plus the EDR detection cycle. The modern operator's competitive advantage isn't a single bypass — it's the ability to compose, to chain unobtrusive primitives in unique combinations, and to be telemetry-aware. For defensive operations, this means hunt rules need to be composite, not single-signal, and they need updating with every research disclosure.
