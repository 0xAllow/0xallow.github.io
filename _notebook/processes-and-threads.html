Processes & Threads,
or: what lives inside a process after the loader walks away.

A reading of the Process Environment Block, the Thread Environment Block, and the doubly-linked module list that makes API resolution possible without a single import — written for people who hunt these structures or hide inside them.

blacksunCUBE field notebook · vol. 03 · builds on vol. 01 & vol. 02

Vol. 02 ended with the loader finishing its job: a PE mapped into virtual memory, sections protected with their R/W/X flags, imports resolved, entry point ready to fire. Now the question becomes: where does the process keep all the state that the kernel and user-mode runtime need to coordinate?

The answer is two structures — the PEB and the TEB — and one doubly-linked list. Together they are the process's autobiography. Every shellcode loader, every API hash resolver, every anti-debug check, every EDR detection of all of the above, lives or dies by what these structures contain and how to find them.

Two warnings before we start, because I have seen both bite people in production code:

The offsets in this notebook are for x64 Windows 10 / 11. The PEB and TEB are undocumented structures — Microsoft reserves the right to move fields, and they have done so. ReWolf's blog catalogues 11 unique PEB layouts across Windows versions. If you're targeting Windows 7, Server 2003, or x86, verify with WinDbg's dt _PEB and dt _TEB on your target system. Hardcoding offsets without verifying is how shellcode ends up dereferencing garbage.

Same color axes you know from vol. 01-02:

R · Read

Fields you consult

Identity, debug flags, module lists, command line. The runtime reads these to answer questions about itself.

W · Write

Fields the runtime mutates

LastError, TLS slots, the loader's lists growing with each LoadLibrary. Live state, written across the process's lifetime.

X · Execute

Fields that drive control flow

Module base addresses for export resolution, callback table pointers, entry points. Touch these and code runs.

§ 01 Where the OS hides the PEB and TEB

A segment register, a fixed offset, and you're inside.

Every thread on Windows has a TEB. Every process has exactly one PEB, shared by all its threads. The kernel sets up these structures before user-mode code starts running, and exposes their addresses through segment register tricks that have been stable since the NT family began.

On x64, the GS segment register's base address points at the current thread's TEB. On x86, the FS register does the same. This is not a memory load through the segment in the classical x86 sense — in long mode the GS base is a hidden register set up by WRGSBASE / SWAPGS by the kernel. From user mode, you read it as if it were a memory operand:

// x64 — get a pointer to the current TEB, then to the PEB
PTEB teb = (PTEB)__readgsqword(0x30);  // TEB.NtTib.Self — offset 0x30 inside TEB
PPEB peb = (PPEB)__readgsqword(0x60);  // TEB.ProcessEnvironmentBlock

// or in raw asm — the form you see in shellcode constantly:
//   mov rax, gs:[0x60]      ; rax = PEB pointer (x64)
//   mov eax, fs:[0x30]      ; eax = PEB pointer (x86)

Two things to internalize from this:

  1. You don't need any API call to find the PEB. No GetCurrentProcess, no NtQueryInformationProcess, no import. A two-byte instruction (65 48 8B 04 25 60 00 00 00 for mov rax, gs:[0x60]) and you have the entire process's autobiography.
  2. Every shellcode in existence starts here. Locating kernel32.dll by walking the PEB is the foundational primitive. If you can't do this, you can't call any API at all without an import.
Signal for defenders: The instruction mov rax, gs:[0x60] (bytes 65 48 8B 04 25 60 00 00 00) is a textbook YARA pattern for shellcode. Legitimate code rarely accesses the PEB directly — it uses the documented API NtCurrentTeb(), which the compiler still emits as a GS read, but in well-known function prologues from kernelbase or the CRT. The same instruction in a region without a corresponding loaded module — private executable memory, RWX page, JIT-style allocation — is one of the strongest indicators of in-memory shellcode. Pair this with ETW Microsoft-Windows-Threat-Intelligence for kernel-side allocation events.

§ 02 The PEB: process-wide autobiography

Documented in winternl.h as a wall of Reserved fields. Reverse-engineered in full by half the security industry.

Microsoft publishes a "public" version of the PEB in winternl.h with most fields blacked out as Reserved1, Reserved2, etc. The intent is to discourage you from depending on the layout. The reality is that the layout has been documented externally for over twenty years and is absolutely depended upon by half the malware industry, all the EDRs, and Microsoft's own legacy code.

Here is the layout for x64 Windows 10/11, with the offsets you actually use:

I'll walk through the four hot fields, because each one is a primitive that ties to a specific category of tradecraft and detection.

+0x002 BeingDebugged — the laziest anti-debug check in history

BeingDebugged (UCHAR)
read

"I'm 1 if a user-mode debugger is attached, 0 otherwise. The kernel updates me when DbgUiConnectToDbg is called."

// what IsDebuggerPresent() actually does, from kernel32.dll:
BOOL IsDebuggerPresent(void) {
    PPEB peb = (PPEB)__readgsqword(0x60);
    return peb->BeingDebugged;          // that's it. that's the whole function.
}

This is the canonical anti-debug check, and the canonical anti-anti-debug bypass. Malware that wants to detect a debugger reads BeingDebugged directly from the PEB to avoid calling IsDebuggerPresent (which is hooked or watched by every analysis sandbox). Defenders / debugger authors flip the byte to zero before letting the analyzed code run. ScyllaHide, x64dbg's anti-anti-debug plugins, and TitanHide all do this.

+0x010 ImageBaseAddress — the in-memory copy of the main EXE

This is the loaded base of the running executable — what the loader resolved ImageBase to (after ASLR moved it from its preferred address). It's the same value GetModuleHandle(NULL) returns. From here you can walk the PE headers exactly as in vol. 02: e_lfanew → NT Headers → export directory → functions.

→ operator's view · process hollowing

Process hollowing rewrites this field. The attack: spawn a benign EXE suspended, unmap its image, allocate a new one at a different base, write your payload, then patch PEB.ImageBaseAddress to point at your image and resume the thread. Every introspection API now reports the benign EXE's identity but executes your code. Detection: compare the on-disk image at the process's reported path with the actual mapped image at ImageBaseAddress.

→ defender's view · image-base mismatch

Tools like pe-sieve and hollows_hunter walk every running process and compare ImageBaseAddress's mapped contents byte-by-byte against the on-disk image at the process's FullDllName. Mismatches indicate hollowing, doppelganging, or in-memory patching. ETW's Microsoft-Windows-Threat-Intelligence surfaces image-mapping events in real time.

+0x018 Ldr — the doubly-linked module list (the most exploited field on Windows)

Ldr (PPEB_LDR_DATA)
readexecute

"I point to a structure containing three doubly-linked lists of every DLL loaded in this process. Walk me to find ntdll, kernel32, anything you need."

This is the field that makes shellcode possible. Vol. 02 explained how HMODULE equals the base address of a loaded module. Now we see how to find those bases without calling GetModuleHandle — without, in fact, importing anything at all.

PEB.Ldr points to a PEB_LDR_DATA structure:

typedef struct _PEB_LDR_DATA {
    ULONG      Length;
    BOOLEAN    Initialized;
    PVOID      SsHandle;
    LIST_ENTRY InLoadOrderModuleList;          // +0x10 (x64)
    LIST_ENTRY InMemoryOrderModuleList;        // +0x20
    LIST_ENTRY InInitializationOrderModuleList; // +0x30
} PEB_LDR_DATA, *PPEB_LDR_DATA;

Three lists, same modules, different sort orders. Each list entry is a LDR_DATA_TABLE_ENTRY containing — among other things — the module's base address and its full DLL name:

typedef struct _LDR_DATA_TABLE_ENTRY {
    LIST_ENTRY     InLoadOrderLinks;
    LIST_ENTRY     InMemoryOrderLinks;
    LIST_ENTRY     InInitializationOrderLinks;
    PVOID          DllBase;          // <-- the HMODULE you want
    PVOID          EntryPoint;
    ULONG          SizeOfImage;
    UNICODE_STRING FullDllName;      // e.g. L"C:\Windows\System32\ntdll.dll"
    UNICODE_STRING BaseDllName;      // e.g. L"ntdll.dll"
    // ...
} LDR_DATA_TABLE_ENTRY, *PLDR_DATA_TABLE_ENTRY;

A typical shellcode resolution loop:

// 1. Get PEB
PPEB peb = (PPEB)__readgsqword(0x60);

// 2. Get the head of InLoadOrderModuleList
PLIST_ENTRY head = &peb->Ldr->InLoadOrderModuleList;

// 3. Walk the doubly-linked list
for (PLIST_ENTRY cur = head->Flink; cur != head; cur = cur->Flink) {
    PLDR_DATA_TABLE_ENTRY entry =
        CONTAINING_RECORD(cur, LDR_DATA_TABLE_ENTRY, InLoadOrderLinks);

    // 4. Hash the BaseDllName, compare to a precomputed hash for "ntdll.dll"
    if (hash_unicode(&entry->BaseDllName) == NTDLL_HASH) {
        ntdll_base = entry->DllBase;          // got it. now parse PE headers.
        break;
    }
}
The InLoadOrderModuleList is, predictably, sorted by load order. The first entry is always the running EXE itself; the second is ntdll.dll; the third historically was kernel32.dll but on Windows 10+ it's typically kernelbase.dll first. Never trust position — always hash the name. MalwareTech wrote about EDRs that intentionally swap the second and third entries with honeypot modules to trip naive shellcode that assumes order.

+0x020 ProcessParameters — command line, environment, image path

typedef struct _RTL_USER_PROCESS_PARAMETERS {
    // ... padding ...
    UNICODE_STRING ImagePathName;     // "C:\Windows\notepad.exe"
    UNICODE_STRING CommandLine;       // "notepad.exe foo.txt"
    PVOID          Environment;       // pointer to env block (KEY=VALUE\0KEY=VALUE\0\0)
    // ... and dozens more fields ...
} RTL_USER_PROCESS_PARAMETERS;

This is what every EDR reads to populate the "Command Line" column in its process tree. The values come from what the parent process passed to CreateProcess — which the parent can lie about.

command-line spoofing — why the field cannot be trusted

The RTL_USER_PROCESS_PARAMETERS block is writable from within the process. An attacker can spawn a child suspended with a benign command line ("notepad.exe"), open the child's PEB through NtReadVirtualMemory / NtWriteVirtualMemory, overwrite CommandLine.Buffer with the real malicious arguments after the parent has already logged the spoofed version, then resume the thread. EDRs that snapshot CommandLine only at process creation see the lie. EDRs that re-read it post-init see the truth. This is why mature EDRs hook on every NtWriteVirtualMemory targeting a remote PEB.

Signal for defenders: Sysmon Event 1 (process creation) captures CommandLine from the parent's CreateProcess call. ETW Microsoft-Windows-Kernel-Process event 1 captures the same. Neither sees post-init spoofing. The robust signal is correlating Sysmon Event 1's CommandLine with a re-read of PEB.ProcessParameters.CommandLine a few seconds into process lifetime — if they differ, you've caught a spoofer. Tools like hollows_hunter have this check built in.

§ 03 The TEB: per-thread state

One per thread. Holds the stack bounds, exception chain, TLS, and a pointer back to the PEB.

The PEB is process-wide; the TEB is thread-local. Every thread gets its own. The kernel allocates a fresh TEB during thread creation and updates the GS base accordingly on context switches, so gs:[0] always points to your TEB regardless of which thread you're on.

The first ~0x38 bytes of the TEB are the NT_TIB — "Thread Information Block" — an even older substructure that predates the TEB. The TIB is the part you reach for most often:

Why TEB-relative addressing is genius

Notice NtTib.Self at +0x30: it points back to the TEB itself. This is not redundancy — it's a load-the-base-without-knowing-it trick. On x86, you can't directly read the FS-base register from user mode. But you can read through FS at offset 0x18 (the x86 location of Self):

// x86 — get the linear address of the current TEB
//   mov eax, fs:[0x18]   ; eax = TEB linear address
// On x64 the equivalent is gs:[0x30], but x64 also has __readgsqword(0).

Once you have the linearized TEB pointer, you can pass it to functions that expect a regular pointer, hand it to debuggers, or use it for cross-thread inspection — none of which work with the segment-based form.

StackBase and StackLimit — where your stack actually lives

These two fields define the current thread's stack region. StackBase is the top (highest address); StackLimit is the bottom of the currently committed portion. Note that the reserved stack is usually 1 MB (set by SizeOfStackReserve in the PE OptionalHeader, vol. 02), but only a few pages are actually committed at thread start. As the thread grows the stack downward and hits a guard page, the kernel commits more pages and updates StackLimit.

→ operator's view · stack pivot

Stack pivots in exploitation rely on knowing or controlling the stack region. Sleep-mask techniques (Ekko, FOLIAGE, Cronos) move the stack to a fake region during sleep, encrypt the original, then swap back — this requires writing valid StackBase / StackLimit values so unwind metadata still resolves. Get them wrong and the next exception walks into garbage and the process dies.

→ defender's view · thread stack scanning

EDRs that hunt sleeping beacons do so by walking each thread's stack between StackBase and StackLimit and looking for return addresses that point into private (non-image) executable memory. The classic indicator: a thread blocked in WaitForSingleObject with a return-address chain pointing into a non-backed RWX region. BeaconEye automated this; modern EDRs have it kernel-side via PsSetCreateThreadNotifyRoutine.

ClientId — the canonical (PID, TID) pair

typedef struct _CLIENT_ID {
    HANDLE UniqueProcess;   // PID, but typed as HANDLE
    HANDLE UniqueThread;    // TID, also HANDLE-typed
} CLIENT_ID;

These are the same values GetCurrentProcessId() and GetCurrentThreadId() return. The HANDLE typing is a quirk — they are not actual handles you can CloseHandle on; they're integers wrapped in HANDLE for legacy reasons. You'll see them throughout the native API: NtOpenProcess takes a CLIENT_ID as an out-of-band identifier.

§ 04 The shellcode primer: from gs:[0x60] to a function pointer

Twelve lines of pseudo-asm that summarize three vols of this notebook.

Vol. 02 ended on a cliffhanger: how do you call NtAllocateVirtualMemory without importing it? Now we have the full picture. Here is the complete walk, every offset traced:

; ========================================================================
; Resolve NtAllocateVirtualMemory in ntdll.dll, no imports, x64 Windows 10/11
; ========================================================================

; --- Step 1: get the PEB --------------------------------------------------
mov     rax, gs:[0x60]              ; rax = PEB

; --- Step 2: get PEB.Ldr --------------------------------------------------
mov     rax, [rax + 0x18]            ; rax = PEB_LDR_DATA*

; --- Step 3: head of InLoadOrderModuleList -------------------------------
mov     rax, [rax + 0x10]            ; rax = first LIST_ENTRY (Flink) — actually points
                                  ;       at the running EXE's LDR_DATA_TABLE_ENTRY

; --- Step 4: walk to ntdll.dll (typically the second entry) --------------
mov     rax, [rax]                ; Flink->Flink — now pointing at ntdll's entry

; --- Step 5: extract DllBase from the LDR_DATA_TABLE_ENTRY ---------------
mov     rbx, [rax + 0x30]            ; rbx = ntdll.dll base address (HMODULE)
                                  ;       offset 0x30 = DllBase in the in-memory-order entry

; --- Step 6: parse PE headers (vol. 02 territory) ------------------------
mov     eax, [rbx + 0x3C]            ; e_lfanew (DOS header)
add     rax, rbx                  ; rax = NT Headers
mov     eax, [rax + 0x88]            ; DataDirectory[0].VirtualAddress (export table)
add     rax, rbx                  ; rax = IMAGE_EXPORT_DIRECTORY

; --- Step 7: walk the export name array, hash each name, compare --------
;             (omitted for brevity — same pattern as in vol. 02)
;             on match, return ntdll_base + AddressOfFunctions[i]

; rax now holds the VA of NtAllocateVirtualMemory.
; You can call it. Or, better, read the syscall number from its first
; few bytes (mov eax, <ssn>) and issue the syscall directly.

That is what every reflective loader, every Cobalt Strike beacon, every Metasploit stager does in its first dozen instructions. Three vols of this notebook reduce to that block.

§ 05  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all of them without notes, you own this volume.

How does an x64 user-mode program find its PEB without calling any API?
Read the qword at gs:[0x60]. The kernel sets up the GS base of every thread to point at that thread's TEB, and the TEB at offset +0x60 contains the ProcessEnvironmentBlock pointer. Two-instruction primitive (__readgsqword(0x60) in C); single-instruction primitive in asm.
What does IsDebuggerPresent() actually do under the hood?
It reads one byte: PEB.BeingDebugged at offset +0x002. The kernel sets this byte to 1 when a user-mode debugger attaches. Malware reads it directly to skip the API call (which is hooked in sandboxes); analysts patch it back to 0 with tools like ScyllaHide or x64dbg's anti-anti-debug plugin.
There are three module lists in PEB_LDR_DATA. Why three, and which one do shellcoders typically walk?
All three contain the same modules in different sort orders: load order, memory order, initialization order. Shellcoders typically walk InLoadOrderModuleList at offset +0x10 because the running EXE is always first and ntdll.dll is always second — predictable, but this is also why EDRs sometimes inject honeypot modules at those positions to catch order-based shellcode. Hash the BaseDllName, never trust position.
Why are PEB and TEB offsets dangerous to hardcode?
Both structures are undocumented; Microsoft reserves the right to change layout. ReWolf catalogued 11 unique PEB layouts across Windows versions. Most fields are stable from XP through Windows 11 on x64, but some shift. Always verify with WinDbg's dt _PEB / dt _TEB on the target system, or use exported helpers like NtCurrentTeb().
A process is launched with command line "notepad.exe". A few seconds later a SOC analyst inspects PEB.ProcessParameters.CommandLine and sees something else. What likely happened?
Command-line spoofing. The parent passed a benign command line to CreateProcess (logged by Sysmon Event 1), then while the child was suspended, the parent overwrote CommandLine.Buffer in the child's PEB with the real arguments before resuming. Sysmon's snapshot at creation captures the lie; a runtime re-read captures the truth. Mature EDRs detect this by hooking NtWriteVirtualMemory targeting remote PEBs.
What do StackBase and StackLimit in the TIB tell you, and why do sleep-mask techniques care about them?
StackBase is the top (highest address) of the thread's stack; StackLimit is the bottom of the currently committed region. Sleep-mask techniques (Ekko, FOLIAGE) temporarily relocate the stack during a sleep cycle to evade EDRs that scan thread stacks for return addresses pointing into private RWX memory. To make the relocation valid, they must write coherent StackBase/StackLimit values — otherwise stack-walk-based detection breaks the process when an exception fires.
An attacker rewrites PEB.ImageBaseAddress after process creation. What's the technique called and how is it caught?
Process hollowing. The attacker spawns a benign EXE suspended, unmaps its image, allocates a new image, writes their payload, and updates PEB.ImageBaseAddress to point at the new image before resuming. Detection: tools like pe-sieve compare the bytes at ImageBaseAddress against the on-disk image at the process's reported path. Mismatch = hollowing. ETW Microsoft-Windows-Threat-Intelligence surfaces image-mapping events real-time.
A YARA rule fires on the byte sequence 65 48 8B 04 25 60 00 00 00 in private memory regions. What is the rule looking for and why is the location significant?
That's mov rax, gs:[0x60] — reading the PEB pointer. Legitimate code uses this in well-known function prologues from kernelbase / CRT, which are mapped from on-disk images. The same instruction in private executable memory (RWX page, JIT-style allocation, or unmapped region) is the textbook opening of in-memory shellcode resolving APIs without imports. Pair with stack-walk telemetry for high-confidence detection.

§ 06 Closing thought

Three volumes in, here is what we have actually built: a complete model of how a Windows process exists in memory.

Vol. 01 gave us the alphabet — the integers, pointers, handles, and strings that the kernel and runtime exchange. Vol. 02 gave us the static skeleton — the PE format that describes how an executable is laid out on disk and how the loader unpacks it into memory. This volume gave us the living state — the PEB describing the process, the TEB describing each thread, and the Ldr list connecting every loaded module by base address.

From here, the offensive techniques you will read about — reflective loading, manual mapping, process hollowing, doppelganging, sleep masks, indirect syscalls, API hashing, IAT-less calls — are all recombinations of operations on these three layers. None of them is a separate skill. They are all the same skill, applied with different intent.

The same is true for detection. Every credible EDR signal eventually grounds in: does the byte at this PEB offset look right? Does the on-disk image match the in-memory image at ImageBaseAddress? Is this thread's stack walking through expected modules? Is this NtWriteVirtualMemory targeting a remote PEB? The vocabulary is small. The composition is endless.
