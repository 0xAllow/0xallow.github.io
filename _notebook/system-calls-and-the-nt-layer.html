System Calls,
or: the eight bytes where the war is fought.

A walk from kernel32!CreateFileW down to syscall, through the CPU's privilege boundary, and back with a tour of every gate, every detection, and every detection bypass that lives in those four instructions.

blacksunCUBE field notebook · vol. 05 · builds on vol. 03 (PEB) & vol. 04 (loader)

If vol. 04 was about the road that brings DLLs into your address space, vol. 05 is about the border crossing at the end of that road. Every Windows program, sooner or later, needs to do something only the kernel can do allocate memory, open a file, start a thread, talk to a driver. The instruction that performs that crossing is syscall. It executes in roughly four CPU cycles. And around it, an entire industry of attack and detection has grown up over the last eight years.

This volume is denser than the previous four. There's a reason: this is where almost every modern EDR evasion technique either lives or dies. Hell's Gate, Halo's Gate, Tartarus' Gate, FreshyCalls, SysWhispers, indirect syscalls, call-stack spoofing you'll meet all of them, in the order they appeared, with the detection technique each one was built to defeat and the counter-detection it triggered.

§ 01 Why the NT layer exists

A user/kernel split that ARM and x86 enforce in hardware, and a DLL that bridges them.

Modern CPUs have rings privilege levels. On x86_64, the ones that matter are ring 3 (user mode) and ring 0 (kernel mode). Code in ring 3 can't touch hardware, can't write to physical memory, can't read another process's address space. Code in ring 0 can do all of those things. The CPU enforces this split at every instruction: the wrong opcode in ring 3 produces a fault, not a silent override.

That means your code, sitting comfortably in ring 3, has to ask the kernel to do anything privileged on its behalf. The asking happens via the syscall instruction. And the asking has to be structured: which operation? with what arguments? where does the result go? That structure is what the Nt layer in ntdll.dll provides.

Every Nt* function in ntdll is, fundamentally, a four-instruction template that:

  1. Saves RCX (the first argument register) into R10, because syscall clobbers RCX.
  2. Loads the System Service Number (SSN) into EAX the integer that tells the kernel which operation you want.
  3. Issues the syscall instruction. Privilege transition happens here.
  4. Returns to the caller.

That's it. That's the whole pattern. You'll see it now:

; ntdll!NtAllocateVirtualMemory  (Windows 10/11 x64, unhooked)
mov     r10, rcx              ; preserve RCX (1st arg) before syscall trashes it
mov     eax, 0x18             ; SSN — value varies by Windows build!
syscall                       ; cross into ring 0; kernel dispatches based on EAX
ret                           ; return to caller

Two facts to anchor everything else in this volume:

  • The SSN above (0x18) is not stable. It's representative for some Windows 10/11 builds; on other builds it's something else. Microsoft reorders its System Service Descriptor Table between releases. j00ru maintains the canonical online table tracking SSNs across every build which is why hardcoded SSNs are a recipe for an exploit that works on your test VM and fails on a customer's box.
  • The pattern itself is fixed. Every single Nt* function in ntdll begins with this exact 4-instruction sequence (in the unhooked case). That's what makes Hell's Gate work: you don't need symbols, you don't need imports if you find the byte sequence, you can extract the SSN.

§ 02 The full descent: from CreateFileW to syscall

Five frames, two DLLs, one transition.

When you call a Win32 API like CreateFileW, the call passes through several layers before it reaches the kernel. Each layer has a job:

← user mode · ring 3 ring 0 · kernel mode →
0
your code: CreateFileW(...)
your .text section
1
kernel32!CreateFileW
forwarder → kernelbase
2
kernelbase!CreateFileW
argument validation, A→W conversion
3
ntdll!NtCreateFile
syscall stub (4 instructions)
·
syscall instruction — privilege transition
opcode 0F 05 — ring 3 → ring 0
4
nt!KiSystemCall64

5
nt!NtCreateFile
actual implementation in ntoskrnl.exe

The frame that matters operationally is frame 3. ntdll!NtCreateFile is the last user-mode function before the kernel. Everything above it can be hooked, replaced, unloaded; everything below it lives in kernel memory you cannot touch from a normal process. That's why every EDR places its inline hooks at frame 3 and why every direct-syscall technique aims to sidestep frame 3 entirely.

The duplication you might notice kernel32!CreateFileW calling kernelbase!CreateFileW is a real artifact, introduced by Windows 7's MinWin refactor. Most Win32 functionality moved into kernelbase.dll, but kernel32.dll kept thin forwarders for ABI compatibility. Knowing this lets you read MSDN and call stacks correctly: when MS says "this is in kernel32," they almost always mean the implementation lives in kernelbase.

§ 03 What syscall actually does

A single instruction with a lot of CPU machinery behind it.

The syscall instruction is two bytes long: 0F 05. Don't let that fool you it triggers a sequence of CPU operations that's nothing like a normal function call.

syscall (opcode 0F 05)
execute

"Switch to ring 0, jump to whatever the kernel registered in MSR_LSTAR, and let it figure out what you want from EAX."

; What syscall does, microarchitecturally:
;
;  1. RIP    -> RCX                 ; save user-mode return address
;  2. RFLAGS -> R11                 ; save user-mode flags
;  3. RIP    <- IA32_LSTAR (MSR)    ; jump to kernel's syscall handler
;  4. CS, SS <- IA32_STAR (MSR)     ; switch to kernel-mode segments
;  5. mode   <- ring 0              ; privilege change happens here
;
; On Windows, IA32_LSTAR points at nt!KiSystemCall64.
;
; What's NOT pushed:
;   - Stack pointer is not switched by the CPU. Windows has to do that
;     in software via swapgs + GS:[KPCR.UserRsp].
;   - General-purpose registers other than RCX/R11 are NOT saved by
;     the CPU. The kernel saves what it needs.

The two model-specific registers (MSRs) that matter:

IA32_LSTAR (0xC0000082)Holds the address of the kernel's syscall entry point. On Windows, this is nt!KiSystemCall64. Set once at boot by the kernel.
IA32_STAR (0xC0000081)Holds the segment selectors used after the privilege switch (kernel CS/SS, plus user CS/SS to be restored on sysret).

From KiSystemCall64 onwards, the kernel does swapgs to point GS at the per-CPU KPCR, switches to the kernel stack, saves user-mode registers in a trap frame, then indexes the System Service Descriptor Table (SSDT) by EAX to find the actual function pointer for the syscall you requested. That's how the SSN gets translated to "call nt!NtCreateFile".

why the user/kernel split is a security boundary, not a design choice

Without the privilege check the CPU enforces on every instruction, an EDR running in user mode would have no way to defend itself: any process could just overwrite the EDR's code, hooks, or detection logic. The user/kernel split lets the EDR put its core in ring 0 (a kernel driver) and its visibility hooks in ring 3 (DLLs injected into every process). User-mode hooks are convenient and fast, but they live on the attacker's side of the privilege boundary and that asymmetry is the entire reason the syscall arms race exists.

§ 04 System Service Numbers: small integers, big consequences

A volatile lookup index that's been the foundation of every direct-syscall technique since 2019.

The SSN is just an integer. NtAllocateVirtualMemory on a particular Windows 10 build might be 0x18; on a different build, it could be 0x19, 0x36, or something else entirely. Microsoft reorders the SSDT essentially every feature update.

That's a problem for offensive tooling, because you don't know what build your target is running. Three approaches have emerged:

Approach 1 — Hardcode + table lookup

Ship a giant per-build mapping (j00ru's online table is the canonical reference), detect the OS version at runtime, pick the right SSN. Workable but heavy: your binary carries a lot of "extra stuff" that has no other reason to be there, and that extra stuff is itself a detection signature. SysWhispers (the original 2019 release) did this.

Approach 2 — Dynamic resolution from ntdll at runtime

Walk the loaded ntdll.dll, find the Nt* function you want, read the four-byte little-endian integer at offset 4 of its stub. That's the SSN. No hardcoded values, no per-build tables, no version detection. This is what Hell's Gate does.

// Hell's Gate SSN extraction, simplified
PVOID stub = GetProcAddress(ntdll, "NtAllocateVirtualMemory");
// Or, more discreetly: walk PEB.Ldr → ntdll → export table → hash the name

// The unhooked stub looks like this:
//   4C 8B D1            mov   r10, rcx
//   B8 18 00 00 00      mov   eax, 0x18         <-- the SSN
//   F6 04 25 ...        test  byte ptr [..]      ; (introduced in Win 10 2004+)
//   75 03               jne   short
//   0F 05               syscall
//   C3                  ret

DWORD ssn = *(DWORD*)((BYTE*)stub + 4);   // extract the 4-byte SSN

Approach 3 — Approach 2, but resilient to hooks

Approach 2 fails the moment an EDR has overwritten the first few bytes of the stub with a jmp trampoline. Halo's Gate (Reenz0h, 2021) handles this by walking up or down to the next Nt* function in memory, checking its stub, and inferring the hooked function's SSN by adjacent-offset arithmetic. Tartarus' Gate (trickster0, 2021) extends Halo's Gate to handle EDRs whose hook bytes don't match Halo's pattern.

§ 05 The gates: a four-year arms race

Each technique is a counter to the previous detection. Read in order.

timeline of direct-syscall evasion (2019–2024)
SysWhispers · 2019, Jackson_T
First widely-used generator. Hardcoded per-build SSN tables. Solved "EDR hooks ntdll" but introduced a heavy fingerprint the embedded SSN tables themselves became a static-analysis IOC.
Hell's Gate · 2020, am0nsec & RtlMateusz
No hardcoded SSNs. Walk the PEB, find ntdll, parse exports, read the SSN from the live stub at runtime. Drastically smaller binaries. Defeated by: EDRs that hook ntdll the stub Hell's Gate reads is overwritten with a jump.
Halo's Gate · 2021, Reenz0h (Sektor7)
When the target stub is hooked (first byte is 0xE9), walk to the next Nt* function in memory, read its SSN, and add or subtract 1 to compensate (SSNs of adjacent Nt* functions in ntdll are sequential). Defeated by: EDRs that hook with patterns Halo's Gate didn't recognize.
Tartarus' Gate · 2021, trickster0
Extends Halo's pattern recognition to handle additional EDR hook formats (e.g. the 4C 8B D1 E9 ... "preserve-then-jump" form some EDRs use). Same SSN-from-neighbor trick, more permissive byte detection.
Indirect syscalls · 2022, multiple researchers
Direct syscalls have a giveaway: the syscall instruction lives in your .text section, so the kernel's return-address check shows the syscall came from a non-ntdll region. Indirect syscalls fix this: set up arguments yourself, but jmp into ntdll's own syscall;ret gadget. The kernel sees a return address inside ntdll legitimate-looking. Defeated by: EDRs that walk the entire call stack, not just the immediate return address.
Call-stack spoofing · 2022–present, WithSecure et al.
When EDRs inspect the full call stack via ETW-TI or kernel callbacks, even indirect syscalls show an unusual chain. Stack-spoofing techniques (return-address overwrite, frame-pointer manipulation, fake stack frames) construct an entire plausible-looking call chain before performing the syscall. The state of the art in 2024–2026.

Read this list as one continuous conversation between offense and defense. Each row exists because the previous row got detected. If you understand why each step happened, you can read any 2026 EDR-evasion paper and immediately know which detection it's targeting.

§ 06 Direct vs. indirect syscalls: the IOC difference

A few bytes of difference, an entire detection class apart.

Both techniques bypass user-mode hooks on ntdll. The difference is where the syscall instruction lives when it executes.

direct syscall
mov r10, rcx
mov eax, 0x18
syscall            ; in YOUR .text
ret

The syscall opcode is in your shellcode/loader's executable region. The kernel sees a syscall whose return address (in RCX) points into your code, not into ntdll. Easy to detect with a return-address check on entry to KiSystemCall64.

indirect syscall
mov r10, rcx
mov eax, 0x18
jmp [ntdll_syscall_addr]
; ntdll executes:
syscall            ; in ntdll!
ret

You jump to a syscall;ret gadget inside ntdll. The kernel sees a syscall whose return address points into ntdll matches a legitimate API call. Bypasses the simple check, requires full call-stack analysis to detect.

A useful way to remember it: direct = "I'll do the privilege transition myself"; indirect = "I'll set up the parameters but borrow ntdll's legal-looking syscall." The indirect version costs an extra jmp and looks much less like an attack.

§ 07 Detection: what an EDR sees, and where

Five chokepoints, in increasing order of evasion difficulty.

1. User-mode hook on ntdll!Nt* (frontline, easy to bypass)

The first thing every EDR DLL does after injection is overwrite the prologue of selected Nt* functions with a jmp to the EDR's inspection routine. This is the line of defense that every direct-syscall technique exists to bypass. Detection power: high for unsophisticated malware, zero for anyone using even Hell's Gate from 2020.

2. Return-address check inside KiSystemCall64

EDRs with a kernel driver can register an instrumentation callback (NtSetInformationProcess with ProcessInstrumentationCallback) that fires after every syscall. The callback inspects the return address: does it point into a known-good module like ntdll or win32u? If not, you have a direct syscall. Defeated by indirect syscalls.

3. ETW Threat-Intelligence (ETW-TI)

Microsoft-Windows-Threat-Intelligence is a kernel-side ETW provider that fires on a curated list of security-relevant operations NtAllocateVirtualMemory in another process, NtProtectVirtualMemory changing a page to RX, NtMapViewOfSection, NtCreateThreadEx, etc. Crucial property: ETW-TI fires inside the kernel, regardless of whether the caller used direct, indirect, or any other user-mode technique. The events are generated unconditionally; the access control is on who gets to read them.

2026 update the bypass that landed. The "ETW-TI is unreachable without PPL" assumption held for years, and almost every EDR vendor's threat model depends on it. Connor McGarr published in January 2026 a technique that consumes the Threat-Intelligence provider without Antimalware-Light protection and without a kernel driver, by abusing the SecurityTrace flag on AutoLogger ETW sessions. Prelude released a working PoC tested on Windows 11 24H2 and 25H2 (github.com/preludeorg/ThreatIntelligenceConsumer). The implication for purple teamers: an attacker with admin can now read the same kernel telemetry the EDR reads useful for understanding what's actually being detected. The implication for detection engineers: an EDR that assumes its ETW-TI consumer is the only one in the system can be observed, profiled, and outmaneuvered by a sufficiently motivated adversary. The events themselves still fire; the secrecy of who's listening doesn't.
ETW-TI is consumable only by processes running at PsProtectedSignerAntimalware-Light (one specific tier of PPL, not just any PPL), which requires an ELAM-signed driver Microsoft only co-signs for vetted security vendors. That's why Defender, CrowdStrike, SentinelOne, Elastic Defend, and other modern EDRs run their telemetry consumer at this protection level. From an attacker's perspective: if you can't BYOVD or get arbitrary kernel write, you historically couldn't blind ETW-TI from user mode. (See § 07's caveat below that statement got narrower in 2026.) From a defender's perspective: an EDR that runs everything as a regular service, without Antimalware-Light protection, is leaving its eyes closed to the most reliable visibility on Windows.

4. Kernel call-stack capture (the current state of the art)

Combined with ETW-TI, EDRs can capture a full kernel-side call stack at the moment a sensitive syscall fires. Elastic Defend introduced the kernel call-stack capability in 8.8 (May 2023), and expanded it with dedicated direct- and indirect-syscall rules in 8.11 (November 2023). The detection looks for: did the syscall actually come from a frame in ntdll, or did it come from a region with no backing image (private RWX memory)? Even indirect syscalls fail this if the rest of the stack is fake.

5. Hardware breakpoints + VEH

The most aggressive technique defensive code sets DR0–DR3 hardware breakpoints on critical Nt* functions, registers a Vectored Exception Handler that fires on the breakpoint, and inspects RSP to verify the call originated from ntdll. Because there are only 4 hardware breakpoints, only the most security-relevant functions get this treatment (NtProtectVirtualMemory, NtWriteVirtualMemory, NtCreateThreadEx, NtCreateUserProcess are typical). See Best EDR Of The Market V2 by xacone for a working implementation.

Hunt template: with ETW-TI + call stack collection enabled, hunt for NtAllocateVirtualMemory or NtProtectVirtualMemory events whose call stack contains any frame with ImageFileName == NULL meaning the return address points into private memory, not a backed image. False positives: JIT compilers (.NET, V8, Java). Filter on legitimate JIT processes, alert on everything else. Elastic publishes a working rule set; SentinelOne and Defender for Endpoint have equivalents in their managed signatures.

What the four-year story actually means

For a purple-team operator, the practical takeaway is this: direct syscalls alone are detectable in 2026. They still work against legacy AV and against EDRs that don't use ETW-TI which is more environments than you'd expect but in a mature shop running Defender for Endpoint, CrowdStrike Falcon, SentinelOne, or Elastic Defend, you will fire telemetry the moment your shellcode allocates memory. The arms race has moved on to call-stack spoofing, which is itself getting harder as kernel callstack collection becomes standard.

For a detection engineer: build your detections around the kernel-side primitives, not the user-mode ones. Hooks in ntdll are great for casual malware but bypassed by anything sophisticated. ETW-TI, kernel callbacks, and image-load callbacks are the foundation. Everything else is a force multiplier.

§ 08  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all of them without notes, you own the syscall layer.

Why does Windows have an "NT layer" (the Nt* functions in ntdll) at all?
Because the CPU enforces a privilege split between ring 3 (user) and ring 0 (kernel). Code in user mode cannot directly perform privileged operations like memory allocation, process creation, or file I/O. The Nt layer in ntdll provides standardized stubs that issue the syscall instruction, which transitions to ring 0. The kernel uses the SSN in EAX to dispatch to the right service.
What are the four instructions in a typical unhooked Nt* stub on x64, and what does each do?
mov r10, rcx (preserve 1st arg before syscall trashes RCX), mov eax, <SSN> (set the system service number), syscall (privilege transition to ring 0), ret (return after sysret brings us back). On Win 10 2004+ there's an extra test byte ptr [...] + jne that handles WOW64 transitions, but the core 4 are stable.
Why is hardcoding SSN values in your tooling fragile, and what's the standard way to resolve them dynamically?
SSNs are reordered between Windows builds NtAllocateVirtualMemory on Windows 10 1903 is not the same number as on Windows 11 24H2. j00ru maintains the canonical online tracking table. Hell's Gate solves this by walking PEB.Ldr to find ntdll, parsing its export directory to locate the function, and reading the 4-byte SSN directly from the stub at offset +4. No version detection needed, no embedded table.
What problem does Halo's Gate solve that Hell's Gate cannot?
Hell's Gate fails when the target function in ntdll is hooked the EDR has overwritten the first 5 bytes with a jmp trampoline, so reading offset +4 gives garbage. Halo's Gate detects the hook (first byte is 0xE9), walks to the next Nt* function in memory, reads its unhooked SSN, and infers the hooked function's SSN by adjacent-offset arithmetic (since Nt* functions in ntdll have sequential SSNs). Tartarus' Gate extends Halo's pattern detection to additional hook formats.
What is the IOC difference between a direct and an indirect syscall, from the kernel's perspective?
In a direct syscall, the syscall instruction lives in the attacker's .text section, so the saved return address (in RCX, which the kernel reads) points into private memory or non-ntdll code. In an indirect syscall, the attacker jmps to a syscall;ret gadget inside ntdll, so the return address looks legitimate. A simple "is the return address inside ntdll?" check catches direct syscalls and misses indirect ones which is why call-stack spoofing became necessary next.
What is ETW-TI, and why is it the bedrock of modern syscall-based detection?
Microsoft-Windows-Threat-Intelligence, a kernel-side ETW provider that emits events on a curated list of security-sensitive operations (NtAllocateVirtualMemory remote, NtProtectVirtualMemory to RX, NtMapViewOfSection, etc.). Critically, the events themselves fire inside the kernel regardless of how the syscall was invoked direct, indirect, hardware-breakpoint, hooked, unhooked. The access control is on the consumer side: traditionally only PsProtectedSignerAntimalware-Light processes (ELAM-signed by Microsoft) could subscribe. As of January 2026, Connor McGarr's SecurityTrace AutoLogger technique allows admin-level processes to consume the same events without PPL. The events remain unbypassable; the read-side monopoly does not.
An EDR captures a kernel call stack on every NtAllocateVirtualMemory. What does it look for to flag the call as suspicious?
It walks each frame and asks: does the return address point into a backed image (a mapped DLL or EXE on disk), or into private memory? A frame whose return address is in private memory particularly RWX private memory means the call originated from shellcode, not from a legitimate module. Even an indirect syscall that bounces off ntdll will leave higher frames pointing into the attacker's loader. The detection requires call-stack spoofing to bypass, not just indirect-syscall mechanics.
What does the model-specific register IA32_LSTAR hold, and why does that matter for offensive tooling?
It holds the address of the kernel's syscall entry point on Windows, nt!KiSystemCall64. The CPU jumps there on every syscall instruction. From offensive tooling's perspective, this matters because: (a) any kernel-side analysis of where syscalls "come from" is fundamentally a check on the saved RIP / RCX after the jump, not on LSTAR itself; (b) BYOVD attacks that achieve kernel write can in theory point LSTAR elsewhere, but doing so is heavy-handed and immediately visible to integrity checks; (c) understanding LSTAR helps you read kernel-side disassembly when reverse-engineering EDRs.

Five volumes in. Here's the picture so far:

Vol. 03 (PEB walk) + vol. 02 (PE parse) + vol. 05 (SSN extraction). Process hollowing? Vol. 02 (PE) + vol. 03 (image-base patch) + vol. 05 (the syscalls that perform the patch). Manual mapping? All four prior volumes, composed.

The same is true for detection. The frontier in 2026 is no longer "did this process load a suspicious DLL" or "is this binary unsigned" those are baseline. The frontier is did the kernel observe a syscall whose call stack doesn't reconstruct to a legitimate module path. ETW-TI plus call-stack capture, surfaced through Defender for Endpoint or Elastic Defend, is the current state of the art. Bypassing it requires kernel access; defending against the bypass requires the same.
