Windows Data Types,
as they actually behave in memory.

A reading of the Windows type system through the only three verbs that matter when you are attacking or defending a process: read, write, execute.

a primer for operators — not a textbook

Here is the thing that took me too long to admit out loud: nobody is confused by int or char*. What slows people down on Windows is a cosmetic problem pretending to be a technical one. LPCWSTR, PHANDLE, ULONG_PTR — these are not new types. They are thirty years of naming conventions layered on top of ordinary C primitives, and once you see through the names, the whole API surface becomes flat and legible.

I want to give you a mental model I wish someone had given me. We will read every type through three axes. Ignore them and the docs look like a wall of noise. Keep them in the back of your head and CreateFileW's signature reads like an English sentence.

R · Read

Inspection, no side effects

The type represents something you only look at: a size, a flag value, a constant string. The callee will not touch your memory through it.

W · Write

Mutation through a pointer

The type is a pointer the API will write through. You are handing the callee a place to put something. Out-parameters live here.

X · Execute

An identity the kernel will act on

The type refers to a live kernel object or a region of executable memory. Touching it causes the OS to do work — files opened, threads created, code run.

Every Windows type on this page gets tagged with one or more of these. Not because Microsoft tags them that way — they don't — but because this is how your brain needs to encode them if you want to read disassembly without a dictionary.

§ 01 Integers, or: what a DWORD really is

The 32-bit unsigned that got its name from a dead memory model.

Start here because half the API surface is integers with a fancy coat. The name DWORD dates from the 16-bit era, where a word was 16 bits and a double word was 32. On modern Windows, both 32-bit and 64-bit, a DWORD is still exactly what it was in 1985: a 32-bit unsigned integer. Range zero to 0xFFFFFFFF. That's it. That's the type.

DWORD
read

"I am a 32-bit unsigned value. Copy me, compare me, don't dereference me."

DWORD dwFlags = 0x40;   // PAGE_EXECUTE_READWRITE
DWORD pid     = GetCurrentProcessId();
DWORD err     = GetLastError();

You will meet DWORD as: flag arguments, process IDs, error codes, access masks, sizes of small things. The Hungarian prefix dw in dwDesiredAccess tells you at a glance: this is a DWORD, a 32-bit bitmask.

→ operator's view

Access-mask DWORDs are where your tooling lives or dies. PROCESS_ALL_ACCESS is 0x1FFFFF — a neon sign for any half-decent EDR. Request only what you need. PROCESS_QUERY_LIMITED_INFORMATION (0x1000) is almost free, visible in nobody's high-priority rules.

→ defender's view

The DWORD on the wire is the telemetry. Sysmon Event 10 surfaces GrantedAccess as hex. A rule that fires only on GrantedAccess masks containing 0x10 | 0x400 (VM_READ + QUERY_INFORMATION) against lsass.exe catches every textbook credential dumper and almost no legitimate software.

WORD, BYTE
read

Smaller integers. Rare at the public API; everywhere inside PE headers.

WORD  machine    = pNtHeaders->FileHeader.Machine;   // 0x8664 = x64
BYTE  firstByte  = pShellcode[0];                   // e.g. 0xEB (jmp)
SIZE_T
read

"I am an integer large enough to measure any object in this address space."

SIZE_T regionSize = 0x1000;
NtAllocateVirtualMemory(hProc, &base, 0, &regionSize, ...);

Here is what matters: SIZE_T is 32 bits on x86, 64 bits on x64. Mix it up with DWORD when you are building a 64-bit loader and you will silently truncate anything above 4 GiB. The bug will not show up in your test VM. It will show up the day you try to map something real.

ULONG_PTR
readwrite

"I am a pointer dressed up as an integer, so the compiler will let you do arithmetic on me."

PVOID pBase = VirtualAlloc(NULL, 0x1000, MEM_COMMIT, PAGE_READWRITE);

// pBase = pBase + 0x10;                       // won't compile — void* has no size
pBase = (PVOID)((ULONG_PTR)pBase + 0x10);  // cast, add, cast back

This is the canonical move in every manual mapper, every reflective loader, every hooking trampoline: cast a pointer to ULONG_PTR, do math, cast back. If you see this in someone else's shellcode stub, they are walking a structure (usually a PE) by hand.

§ 02 VOID, PVOID, and the P convention

The single most useful naming rule in the entire API.

Here is the one rule that, once internalized, makes every Windows function signature parseable at a glance:

the rule

If a type name starts with P, it is a pointer to the type without the P.

That is the entire rule. PHANDLE is HANDLE*. PDWORD is DWORD*. PSIZE_T is SIZE_T*. An LP prefix means the same thing — "long pointer" — a leftover from 16-bit segmented memory that no longer means anything on x86/x64.

Which means that whenever you see a P-type in a function signature, you are looking at either an input-by-reference or an out-parameter. The function will either read through that pointer or write through it. That is not a stylistic detail; it is the difference between a parameter you initialize and one you just declare.

PVOID / LPVOID
readwrite

"I am void*. I point to bytes; you decide what they mean."

PVOID  pBuffer = VirtualAlloc(NULL, 4096, ..., PAGE_READWRITE);
LPVOID pDest   = (LPVOID)0x1400000000;
WriteProcessMemory(hProc, pDest, pBuffer, 4096, NULL);

PVOID is the universal "pointer to anything" type. It is the return type of VirtualAlloc, the destination in WriteProcessMemory, the shellcode buffer in your manual mapper. On x64 it is 8 bytes. Always.

If you catch yourself writing LPVOID in a greenfield project, you are imitating old code. It is identical to PVOID, which is identical to void*. Pick one and stop caring.

§ 03 HANDLE: the thing that makes Windows Windows

Not a pointer. Not an integer. An index into a per-process table the kernel owns.

If you only remember one concept from this whole notebook, make it this one. A HANDLE is not a raw pointer, and it is not a plain number. It is a token that the kernel hands you in exchange for you asking to touch something it owns: a file, a process, a thread, a registry key, a mutex, a section, an access token, a pipe.

user-mode  →  kernel translation
your code
HANDLE hProc = 0x2AC
→
EPROCESS.ObjectTable
[entry 0x2AC]
→
kernel object
EPROCESS  (lsass.exe)

The handle value you hold in user mode is an index into your process's private handle table. The kernel dereferences it on every call. A HANDLE value of 0x2AC in one process is meaningless — or refers to something completely different — in another.

HANDLE
execute

"I am a receipt. Give me back to the kernel and it will do something on your behalf."

HANDLE hFile = CreateFileW(L"C:\\temp\\notes.txt",
                         GENERIC_READ,       // DWORD access mask
                         0, NULL,
                         OPEN_EXISTING,
                         FILE_ATTRIBUTE_NORMAL,
                         NULL);

if (hFile == INVALID_HANDLE_VALUE) {
    // on x64 this is (HANDLE)-1  ==  0xFFFFFFFFFFFFFFFF
    // NOT the same as NULL, which most other APIs return on failure
    printf("[-] CreateFileW failed: %lu\n", GetLastError());
}

...
CloseHandle(hFile);   // leaked handles are a bug *and* a telemetry tell
the footgun

INVALID_HANDLE_VALUE (from the CreateFile* family) is not NULL. It is -1. Most other Windows APIs return NULL to signal failure. Mix these up and your error-check silently passes on a failed call. I have watched this bug eat an entire afternoon.

HMODULE
readexecute

"I am a HANDLE, but I also happen to be literally the base address of a DLL in your process."

HMODULE hNtdll = GetModuleHandleW(L"ntdll.dll");

// hNtdll *is* the base of ntdll.dll. You can cast it and walk the PE headers.
PIMAGE_DOS_HEADER pDos = (PIMAGE_DOS_HEADER)hNtdll;
PIMAGE_NT_HEADERS pNt = (PIMAGE_NT_HEADERS)((ULONG_PTR)hNtdll + pDos->e_lfanew);

This is the seam that every "resolve APIs by hash" technique exploits. The PEB lets you walk the list of loaded modules without calling any Windows API at all — you get the HMODULE (i.e. base address) of ntdll.dll, then you parse its export table in memory and find NtAllocateVirtualMemory by hashing its name. No GetProcAddress. No import table entry for sensitive functions. That whole family of tradecraft starts with understanding that HMODULE == base address.

→ operator's view

API hashing (ROR13, FNV-1a, DJB2) only works because HMODULE is a real pointer you can parse. The trade-off is that your stub now has to walk PE headers at runtime, which creates its own thumbprint — a loop reading from ntdll's export directory looks different from a normal import resolution.

→ defender's view

Processes that call NtAllocateVirtualMemory but have no IAT entry for it — and no matching LoadLibrary / GetProcAddress in their call history — are the signature. ETW's Microsoft-Windows-Threat-Intelligence provider (kernel-side, Protected-Process-only consumers) surfaces this cleanly.

PHANDLE
write

"I am HANDLE*. I exist so a function can hand you a handle through an out-parameter."

HANDLE    hProc = NULL;
OBJECT_ATTRIBUTES oa  = { ... };
CLIENT_ID        cid = { .UniqueProcess = (HANDLE)1234 };

NTSTATUS st = NtOpenProcess(&hProc,  // <-- PHANDLE: kernel writes here
                            PROCESS_QUERY_INFORMATION,
                            &oa, &cid);

Every time you see PHANDLE, you are passing the address of your local variable so the kernel can deposit a new handle there. Write-axis, no question. Don't initialize the variable — it doesn't matter, the callee overwrites it.

§ 04 Strings: a split down the middle of the API

ANSI and Unicode are not two encodings. They are two parallel universes, and one of them is a lie.

Every string-accepting Windows API has two versions. CreateFileA and CreateFileW. MessageBoxA and MessageBoxW. RegOpenKeyExA and RegOpenKeyExW. The suffix decides which string type you are handing in.

LPCSTR / LPSTR
read

ANSI. 8-bit char. The legacy path. Internally converted to Unicode anyway.

LPCSTR path = "C:\\Users\\Public\\x.txt";  // 22 bytes incl. null
CreateFileA(path, ...);
LPCWSTR / LPWSTR
readexecute

Unicode (UTF-16). 16-bit wchar_t. The native language of the Windows kernel.

LPCWSTR path = L"C:\\Users\\Public\\x.txt"; // 44 bytes incl. null
CreateFileW(path, ...);

Now here is the thing that is genuinely important, and that nobody tells beginners: the "A" functions are a polite fiction. Inside every CreateFileA call, the function converts your ANSI string to UTF-16 using the current code page, then calls CreateFileW. Everything below kernelbase.dll is Unicode. The kernel does not speak ANSI. It never has.

This is why the UNICODE_STRING structure — a length-prefixed UTF-16 buffer — shows up everywhere in the PEB, the TEB, the LDR, and the object manager. When you are writing direct syscalls, there is no "A" option. There is only wchar_t.

Byte counts: the off-by-one that ships in production

char    a[] =  "maldev";  //  7 bytes  (6 chars + 1 null)
wchar_t w[] = L"maldev";  // 14 bytes  (6 × 2 + 2-byte null)

// Always match the unit the API wants:
//    'cb' in a parameter name = count of BYTES
//    'cch' in a parameter name = count of CHARACTERS
→ operator's view

In-memory string scanners look for both encodings. A C2 domain stored as char[] also leaks in UTF-16 as a wchar_t[] if you touch any W API. Encode at rest. Decode late. Zero immediately. Keep the window of plaintext in memory as short as you can make it.

→ defender's view

Command-line telemetry (Sysmon 1, Security 4688 with command-line logging) arrives as UTF-16. File-path ETW (Microsoft-Windows-Kernel-File) arrives as UTF-16. Write your regexes and KQL against UTF-16 semantics, not ASCII. Watch for \??\ and \\?\ prefixes — they bypass MAX_PATH and some user-mode sanitizers.

§ 05 [in] and [out]: annotations that tell you who owns what

Not keywords. Not syntax. Just a very useful convention.

When you read MSDN, parameters come pre-annotated:

HANDLE CreateFileW(
    [in]           LPCWSTR               lpFileName,
    [in]           DWORD                 dwDesiredAccess,
    [in]           DWORD                 dwShareMode,
    [in, optional] LPSECURITY_ATTRIBUTES lpSecurityAttributes,
    [in]           DWORD                 dwCreationDisposition,
    [in]           DWORD                 dwFlagsAndAttributes,
    [in, optional] HANDLE                hTemplateFile
);

The bracketed words are SAL (Source-code Annotation Language). They are purely documentation — stripping them does not change a single byte of compiled code. But they tell you at a glance which arguments are inputs, which are outputs, and which you may pass NULL. That is a lot of information for seven characters.

Read the signature above and you now know: CreateFileW takes seven inputs and writes back nothing through its parameters. The only way it communicates success is through its HANDLE return value. If you wanted data back via a pointer, you would see [out] somewhere, which you do not. That is an entire paragraph of understanding from ten seconds of reading.

§ 06 Two error models, and why mixing them will ruin your day

The Win32 layer and the native layer speak different languages about failure.

Win32: sentinel + GetLastError()

Most user-mode Windows APIs do this dance:

HANDLE h = CreateFileW(...);
if (h == INVALID_HANDLE_VALUE) {
    DWORD err = GetLastError();   // <-- MUST be called immediately
    switch (err) {
        case 2:  /* ERROR_FILE_NOT_FOUND  */ break;
        case 5:  /* ERROR_ACCESS_DENIED   */ break;
        case 87: /* ERROR_INVALID_PARAMETER */ break;
    }
}

The function returns a sentinel (NULL, FALSE, (DWORD)-1, or INVALID_HANDLE_VALUE) and stashes the real error code in thread-local storage. You retrieve it with GetLastError, and the rule is: call it immediately. Any intervening API call can clobber it.

Native: NTSTATUS returned directly

Functions exported from ntdll.dll with Nt or Zw prefix don't play the GetLastError game. They return a 32-bit NTSTATUS directly, and zero means success.

#define STATUS_SUCCESS ((NTSTATUS)0x00000000L)

NTSTATUS st = NtOpenProcess(&hProc, PROCESS_QUERY_INFORMATION, &oa, &cid);
if (st != STATUS_SUCCESS) {
    printf("[!] NtOpenProcess: 0x%08X\n", st);
}

// Or better, use the macro that also treats "informational" statuses as OK:
#define NT_SUCCESS(Status) (((NTSTATUS)(Status)) >= 0)

if (!NT_SUCCESS(st)) { ... }
why this matters for direct syscalls

When you bypass kernel32 / kernelbase and invoke Nt* through your own syscall stubs — Hell's Gate, Halo's Gate, Tartarus Gate, SysWhispers, Freshy-Calls, whatever your flavor — GetLastError is meaningless. The Win32-to-NTSTATUS translation only happens inside the wrapper functions in kernelbase.dll. At the syscall boundary, all you have is whatever is sitting in RAX when syscall returns. Read the NTSTATUS. It is all you get.

§ 07 Putting it together

Read this table the way you would read chord shapes — memorize once, then you know the whole song.

TypeWhat it really isAxis
DWORDunsigned 32-bit int. Flags, PIDs, error codes, access masks.read
WORD / BYTE16-bit and 8-bit unsigned. PE headers, shellcode bytes.read
SIZE_Tpointer-sized unsigned int. Sizes of memory objects.read
ULONG_PTRpointer-sized unsigned int, for doing arithmetic on pointers.read write
VOID / PVOIDvoid and void*. Generic memory pointers.read write
HANDLEopaque index into your process's handle table. Kernel object reference.execute
HMODULEa HANDLE that is also literally the base address of a loaded module.read execute
PHANDLE, PDWORD, ...pointer to the corresponding type. Usually an out-parameter.write
LPCSTR / LPSTRconst char* / char*. ANSI. The "A" APIs.read
LPCWSTR / LPWSTRconst wchar_t* / wchar_t*. UTF-16. The "W" APIs and the kernel itself.read execute
NTSTATUS32-bit status code returned by every Nt* / Zw* function.read
three rules

1. P means "pointer to". No exceptions worth caring about.

2. W means Unicode, which is what the kernel actually speaks. Prefer it for anything you want to survive contact with direct syscalls.

3. Two error models. GetLastError for Win32, NTSTATUS for native. Never mix them up; the conversion only exists inside the wrappers.

None of this is hard. The obstacle is that it was all named thirty years ago, by different people, for a computer that no longer exists. Strip away the prefixes and you are looking at a handful of integers, a handful of pointers, and one genuinely interesting abstraction (the handle). That is the whole kit. From here, reading MSDN becomes reading English, disassembly becomes reading pseudocode, and EDR telemetry becomes an annotated audit trail of everything the machine did on your behalf.
