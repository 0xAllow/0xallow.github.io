Process Injection,
or: composing six volumes' worth of primitives into seven techniques.

A walk through the seven canonical injection methods that shaped 2017–2024 offensive Windows tradecraft — classic remote thread, hollowing, APC, doppelgänging, herpaderping, ghosting, early bird — with the kernel telemetry each one fires.

blacksunCUBE field notebook · vol. 07 (part 1 of 2) · builds on vol. 02 (PE), 04 (loader), 05 (syscalls), 06 (hooking)

Six volumes prepared the ground. Now we compose. Every "process injection technique" you've ever heard of decomposes into a small set of primitive operations: open a remote process, allocate memory there, write data, change protections, redirect a thread. The differences between techniques are which subset of primitives they use, in what order, and which detection chokepoint they target.

This volume covers seven techniques. Part 2 (vol. 08) will cover the rest — thread hijacking variants, AtomBombing, transacted hollowing, callback abuse, and the 2024+ generation: Early Cascade Injection, EDR-Preloading, and module stomping. The split is by complexity, not importance: the seven here are the foundational techniques, the ones every other technique builds on.

For each, we cover four things:

  1. The mechanism — what API/syscall sequence makes it work.
  2. The structures touched — PEB fields written, image sections unmapped, transactions opened.
  3. The kernel telemetry — what fires in ETW-TI / kernel callbacks regardless of user-mode evasion.
  4. The detection chokepoint — the single signal that catches this in 2026, even if every other layer is bypassed.

Same axes you know:

R · Read

Reconnaissance

Opening the target, querying its PEB, reading its image base, locating threads. The "looking around" phase before any modification.

W · Write

Payload placement

Allocating memory in the remote process, writing the payload, changing protections, modifying thread context. Where most telemetry fires.

X · Execute

The trigger

Resuming the suspended thread, queuing an APC, hijacking a thread. The single instant when foreign code starts running in the host.

§ 01 The shared primitive set

Eight syscalls, composed differently. That's the whole taxonomy.

Before we walk the seven techniques, here are the primitives every one of them composes from. This list is exhaustive for the techniques in this volume:

PrimitiveSyscall (or pair)What it does
Open target processNtOpenProcess / NtCreateUserProcessGet a handle with rights to read/write/inject. OpenProcess for existing targets, CreateProcess with CREATE_SUSPENDED for new ones.
Allocate remote memoryNtAllocateVirtualMemoryReserve a region in the target's address space. Almost always with PAGE_EXECUTE_READWRITE initially.
Write payloadNtWriteVirtualMemoryCopy shellcode or a PE image into that region.
Change protectionNtProtectVirtualMemoryOptionally flip the region from RWX to RX after the write, to look less suspicious.
Unmap imageNtUnmapViewOfSectionRemove the original PE image from a suspended process. The signature primitive of process hollowing.
Read contextNtGetContextThreadRead a thread's registers (RIP, RSP, etc.). Pre-step for context-modification injection.
Redirect threadNtSetContextThreadOverwrite RIP to make the thread start executing your payload on resume.
Queue APCNtQueueApcThread(Ex)Add an asynchronous procedure call to a thread's APC queue. Fires when the thread enters an alertable state (or when NtTestAlert empties the queue).
Resume / start threadNtResumeThread / NtCreateThreadExStart execution. The trigger.

Almost every detection rule for process injection in production EDR products keys on combinations of two or three of these calls happening cross-process within a short time window. The hard part for offensive code is not finding a "novel primitive" — it's composing the existing primitives in a way that doesn't pattern-match a known sequence.

§ 02 Classic Remote Thread

The textbook injection. Old, loud, still works against unsupervised endpoints.

Classic Remote Thread Injection
MITRE T1055.002 / T1055.003
~2002

Open a target, allocate memory, write payload, create a thread that starts at the payload. Five API calls, no tricks.

canonical flow · classic remote thread injection
1
OpenProcess(PROCESS_ALL_ACCESS, FALSE, pid)
obtain handle to target
2
VirtualAllocEx(hProc, NULL, size, MEM_COMMIT, PAGE_EXECUTE_READWRITE)
allocate RWX in target
3
WriteProcessMemory(hProc, addr, payload, size, NULL)
copy shellcode in
4
CreateRemoteThread(hProc, NULL, 0, addr, NULL, 0, NULL)
spawn thread at payload
5
CloseHandle(hProc)
cleanup
What fires: step 1 produces a process-handle event with GrantedAccess mask 0x1FFFFF — full access to a process the caller has no business touching. Sysmon Event 10 logs this; Defender for Endpoint surfaces it. Step 2 fires ETW-TI ALLOCVM_REMOTE. Step 3 fires WRITEVM_REMOTE. Step 4 fires the PsSetCreateThreadNotifyRoutine kernel callback with the source PID, target PID, and start address — if the start address points to private RWX memory in the target, that's a near-certain IOC. Detection in 2026: trivial across all major EDRs.

This technique is no longer viable against any modern EDR — it's the canonical baseline that every detection rule starts from. We document it here because all subsequent techniques reduce to "do the same thing, but try to evade one of these five telemetry events."

§ 03 Process Hollowing (RunPE)

Create a legitimate-looking process, scoop out its insides, fill with malware. The 2010s flagship.

Process Hollowing / RunPE
MITRE T1055.012
~2011

Spawn a benign-looking process (svchost.exe, explorer.exe, notepad.exe) suspended, unmap its image, copy in a malicious PE, fix up its entry point, resume.

canonical flow · process hollowing
1
CreateProcess("svchost.exe", CREATE_SUSPENDED, ...)
spawn suspended target
2
NtQueryInformationProcess(ProcessBasicInformation) → PEB
find PEB.ImageBaseAddress
3
NtUnmapViewOfSection(hProc, ImageBase)
scoop out the original PE
4
VirtualAllocEx(hProc, ImageBase, SizeOfImage, MEM_COMMIT, PAGE_EXECUTE_READWRITE)
re-allocate at same base
5
WriteProcessMemory: PE headers + each section
copy malicious PE in
6
SetThreadContext: ctx.Rip = malicious_entry_point
redirect main thread to payload
7
ResumeThread(hThread)
execution begins inside fake svchost

The defining primitive is step 3: NtUnmapViewOfSection on a remote process's primary image section. Microsoft's own MITRE technique entry calls this out as the most distinctive indicator: "The call to ZwUnmapViewOfSection or NtUnmapViewOfSection on a remote process is the most distinctive indicator of process hollowing. This API call is rarely used in legitimate software."

Some implementations skip the unmap entirely and just overwrite the original image bytes (vol. 02's PE sections happen to align such that you can write a smaller PE into the same address space). This is what VBA-RunPE does, which is exactly why Cortex XDR added explicit signatures for it — the unmap was the simple detection signal, and dropping it forces defenders to look at the in-memory image vs. on-disk image hash mismatch instead.
Detection in 2026: two signatures in parallel. (1) Cross-process NtUnmapViewOfSection on the primary image section — high-fidelity, low false-positive. Sysmon's Event ID 25 (introduced in Sysmon 13.00, January 2021) fires explicitly when "the mapped image of a process doesn't match the on-disk image file" (Microsoft's wording). (2) PEB.ImageBaseAddress mapped region whose hash doesn't match the executable on disk. Volatility's hollowfind plugin and malfind have caught hollowing for over a decade.

§ 04 APC Injection & Early Bird

Use the asynchronous procedure call mechanism to make a thread execute your code "incidentally."

APC Injection (and Early Bird variant)
MITRE T1055.004
2018 (Early Bird), ~2005 (basic APC)

Queue an APC to a remote thread; when the thread enters an alertable state (or when NtTestAlert empties the queue at thread start), the APC fires and runs your code.

The mechanic is structurally different from the previous techniques: you don't create a new thread. You hijack an existing one's APC queue. Every Windows thread has a dispatcher mechanism that drains pending APCs at well-defined points — on entry, on exit from blocking syscalls, on WaitForSingleObjectEx with bAlertable=TRUE, or via NtTestAlert.

Cyberbit's 2018 disclosure of Early Bird made this technique famous: the canonical chain at thread startup is LdrInitializeThunk → LdrpInitialize → _LdrpInitialize → NtTestAlert → KiUserApcDispatcher, and the APC fires before the application's main — before, in many cases, the EDR's hooks have been installed.

canonical flow · Early Bird APC injection
1
CreateProcess("svchost.exe", CREATE_SUSPENDED, ...)
spawn suspended target
2
VirtualAllocEx(hProc, NULL, size, ..., PAGE_EXECUTE_READWRITE)
RWX in target
3
WriteProcessMemory(hProc, addr, shellcode, size, NULL)
copy shellcode
4
QueueUserAPC((PAPCFUNC)addr, hThread, 0)
queue APC pointing at shellcode
5
ResumeThread(hThread)
NtTestAlert fires APC before main()
Outflank's October 2024 research re-evaluated Early Bird against modern EDRs: the technique still bypasses some EDRs whose hook installation happens after NtTestAlert. But the cross-process APC queueing is itself a high-fidelity IOC (NtQueueApcThread from one process to a freshly-spawned suspended child is unusual). Outflank concluded Early Bird "remains a valuable injection technique, even though it is less effective as an evasion method against modern EDRs." Pair this with what we covered in vol. 04 (loader) and vol. 06 (DLL injection): the same race window that makes Early Bird possible is being closed by EDRs that install their hooks from kernel mode rather than via user-mode APC.
Detection in 2026: NtQueueApcThread with source PID different from target PID, where the target is in SUSPENDED state and was created by the source within seconds. Pair with the APC routine pointing into RWX private memory. Defender for Endpoint, Elastic Defend, and CrowdStrike Falcon all have rules for this pattern. Special User APCs (NtQueueApcThreadEx with the special bit, RS5+) bypass the alertable-state requirement but produce the same telemetry.

§ 05 Process Doppelgänging

Abuse NTFS transactions to execute a malicious image whose disk file never visibly contains malware.

Process Doppelgänging
MITRE T1055.013
2017 (Liberman & Kogan, BlackHat EU)

Open an NTFS transaction, write malicious bytes to a legitimate file inside the transaction, map the transacted file as an image section, roll back the transaction, then create a process from the section. The disk never contains the malware.

This is the first technique in the volume that exploits an undocumented OS feature for evasion rather than just composing primitives differently. Transactional NTFS (TxF) was introduced in Vista to allow atomic file operations — a sequence of writes that either fully commit or fully roll back. Microsoft has since deprecated TxF for new applications, but the kernel still supports it. The catch: image sections created from transacted files retain their backing data after rollback.

canonical flow · process doppelgänging
1
NtCreateTransaction → hTransaction
open NTFS transaction
2
CreateFileTransacted("legit.exe", hTransaction)
open file inside the transaction
3
WriteFile(hFile, malicious_pe_bytes, ...)
overwrite with malware (in-transaction only)
4
NtCreateSection(SEC_IMAGE, hFile)
map malicious bytes as an image section
5
NtRollbackTransaction(hTransaction)
file on disk reverts to original benign content
6
NtCreateProcessEx(hSection, ...) + NtCreateThreadEx
create process from the malicious section
why this defeated AVs in 2017

Antivirus engines scanned files at write time and at process creation. Doppelgänging ensured (a) the file write happens inside a transaction the AV doesn't inspect, (b) the file content visible at process creation is the rolled-back benign version, and (c) the kernel happily creates a process from an image section whose backing file no longer matches what's mapped. From the security product's perspective, the process appears to be running legit.exe; the in-memory image is something else entirely.

The 2017 disclosure used NtCreateProcessEx — an undocumented older syscall that creates a process from a section without firing the standard process creation callbacks at the same point as NtCreateUserProcess. This is the seam that Microsoft eventually closed.

Detection in 2026: Microsoft Defender for Endpoint added explicit detection for the doppelgänging class in 2022, focusing on the FILE_OBJECT abnormalities visible at process creation time. The MpCreateProcessNotifyRoutineEx callback (in WdFilter.sys) inspects the FileObject's transaction state. Doppelgänging is reliably caught by Defender, Elastic Defend, and CrowdStrike in 2026. Cyberwarfare Labs verified that Windows Defender returns 0xC0000022 (ACCESS_DENIED) when the doppelganging chain attempts thread creation. The technique is a known dead-end against mainstream EDRs.

§ 06 Process Herpaderping

Same problem as doppelgänging, solved without transactions: modify the file on disk after the image section is mapped.

Process Herpaderping
MITRE T1055 variant
2020 (Johnny Shaw, jxy-s)

Write the payload PE to disk, map it as a section, then overwrite the file on disk with a decoy before the kernel's process-creation callback fires. The callback inspects the disk file (now decoy), not the mapped section (still malicious).

Herpaderping exploits a specific timing fact established in Elastic's research and Microsoft's own blog: PsSetCreateProcessNotifyRoutineEx callbacks are not invoked at process creation — they're invoked when the first thread is inserted. That gap, between section mapping and first-thread-insertion, is the herpaderping window.

canonical flow · process herpaderping
1
CreateFile("target.exe", FILE_SHARE_READ | WRITE) + WriteFile(payload)
payload written to disk
2
NtCreateSection(SEC_IMAGE, hFile)
map payload as image section
3
NtCreateProcessEx(hSection, ...)
process object created (no callback yet!)
4
SetFilePointer(hFile, 0) + WriteFile(decoy_pe_bytes)
overwrite disk with benign decoy
5
NtCreateThreadEx(hProc, ...)
first thread inserted → callback fires → AV scans disk → sees decoy

The structural difference from doppelgänging: no NTFS transactions. Just an open file handle, a section, and a write. The defender's process-creation scan sees the decoy, not the payload. The mapped section in memory still contains the payload — and that's what the new process executes.

Detection in 2026: mixed. Microsoft Defender added detection in their 2022 blog post specifically for this class; Sysmon's Event ID 25 (image tampering) catches the on-disk vs. in-memory mismatch. However, jxy-s's own GitHub README (which remains the authoritative source) noted that "Defender doesn't catch Herpaderping" as of disclosure — that statement is now outdated for Defender for Endpoint, but some commercial AVs still miss this technique because they trust the disk image at thread-insert time. Elastic Defend's process.Ext.defense_evasions category is the clean reference detection.

§ 07 Process Ghosting

Take herpaderping further: delete the file before the callback fires. Section in memory is now backed by a non-existent file.

Process Ghosting
MITRE T1055 variant
2021 (Gabriel Landau, Elastic)

Mark a file delete-pending, write payload, map as section, close the handle (file is now deleted), create a process from the section. The process backing-file no longer exists when the callback runs.

The crucial OS quirk: Windows prevents modification or deletion of mapped executable files only after the binary is mapped into an image section. So the trick is to set up deletion before mapping, do the mapping, and let the deletion complete after.

canonical flow · process ghosting
1
NtCreateFile(GENERIC_WRITE | DELETE)
file with delete-on-close intent
2
NtSetInformationFile(FileDispositionInformation, DeleteFile=TRUE)
mark for delete-pending
3
WriteFile(hFile, payload, ...)
write malicious PE
4
NtCreateSection(SEC_IMAGE, hFile)
map as image section
5
CloseHandle(hFile)
file deletion completes — now fileless
6
NtCreateProcessEx(hSection) + NtCreateThreadEx
process from a section whose backing file no longer exists

Landau's original PoC demonstrated Windows Defender (at the time) attempting to scan the file fail twice: once because the file was delete-pending, and again because the file no longer existed. The process executed unimpeded.

Microsoft's response when Landau disclosed this in May 2021 was: "does not meet our bar for servicing" — the same response they gave to herpaderping's disclosure a year earlier. This is unusual for Microsoft, and it tells you something about the threat model: Microsoft considers process tampering attacks to be the security product's problem to detect, not the kernel's problem to prevent. The kernel happily continues to permit these techniques in 2026; only the in-product detection layer catches them.
Detection in 2026: Elastic Security shipped explicit ghosting detection in 2021. Defender for Endpoint added detection in their 2022 update (referenced in Microsoft's "Using process creation properties to catch evasion techniques" blog). The detection inspects the FILE_OBJECT's state at process-creation callback time and flags abnormalities — specifically, files that are delete-pending or have already been deleted. Ghosting reliably catches detection in mature EDRs in 2026, but legacy AVs and lighter EDR products still miss it.

§ 08 Detection — the cross-cutting view

Every technique above defeats at least one detection layer. Here's the full lattice.

The seven techniques target different chokepoints. Walking through them in order shows the historical arms race: each new technique was designed to defeat one specific detection that caught the previous generation.

The five detection chokepoints in 2026

1 · ETW-TI cross-process events

ALLOCVM_REMOTE, WRITEVM_REMOTE, PROTECTVM_REMOTE. Fire on every cross-process memory operation regardless of how invoked. Catches classic, hollowing, APC patterns.

2 · Sysmon Event ID 25 (image tampering)

Fires when "the mapped image of a process doesn't match the on-disk image file, or the image file is locked for exclusive access." Sysmon 13.00+. Catches hollowing, herpaderping, ghosting.

3 · FileObject abnormality at PsCreateProcessNotify

Defender for Endpoint & Elastic Defend (2021+) inspect the FileObject when the create-process callback fires. Detect transacted files, delete-pending files, and files with active write handles. Catches doppelgänging, herpaderping, ghosting.

4 · Cross-process APC queueing

NtQueueApcThread(Ex) with source PID ≠ target PID. High-fidelity by itself; near-perfect when paired with target-was-suspended-by-source. Catches APC / Early Bird.

5 · In-memory PE hash diff vs. on-disk

Volatility's hollowfind, malfind, impscan; Defender's behavioral analysis. Catches anything that swaps PE content after process creation. Particularly powerful for forensics; less commonly real-time.

§ 09  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all without notes, you have the foundational injection taxonomy memorized.

Name the canonical 5-call sequence for classic remote thread injection.
OpenProcess(PROCESS_ALL_ACCESS) → VirtualAllocEx(RWX) → WriteProcessMemory → CreateRemoteThread → CloseHandle. Each step fires a distinct telemetry event: handle-open with 0x1FFFFF mask, ALLOCVM_REMOTE, WRITEVM_REMOTE, kernel thread-create callback. This is the baseline against which every other technique is measured for "what did it evade?"
What is the most distinctive single API call that indicates process hollowing, and why?
NtUnmapViewOfSection (or ZwUnmapViewOfSection) called cross-process, on a remote process's primary image section. MITRE explicitly notes this is "rarely used in legitimate software." The unmap is what turns a legitimate svchost.exe process into a hollow shell ready to receive malware. Hollowing variants that skip the unmap (like VBA-RunPE) defeat this specific detection but still get caught by image-hash-vs-disk mismatches.
Why does Early Bird APC injection trigger before main, and what's the call chain?
When a thread starts, LdrInitializeThunk → LdrpInitialize → _LdrpInitialize → NtTestAlert runs as part of thread setup. NtTestAlert empties the APC queue, calling KiUserApcDispatcher for each pending APC. This happens before the application's main entry point. If you've queued an APC pointing to your shellcode and the thread was suspended, resuming it makes NtTestAlert fire your APC immediately, before any application or EDR-injected hook code runs.
Process Doppelgänging uses Transactional NTFS. What's the four-word summary of the core trick?
"Transact, write, map, rollback." The malicious PE bytes are written to a legitimate file inside an NTFS transaction. An image section is created from the transacted file (containing malicious bytes). The transaction is rolled back — the file on disk reverts to benign. But the kernel-cached image section still contains the malicious bytes, and a process can be created from it. AV scanning the file sees benign content; the running process executes malicious code.
Why does Process Herpaderping work without using NTFS transactions?
Because PsSetCreateProcessNotifyRoutineEx callbacks fire on first-thread-insertion, not on process object creation. The window between NtCreateProcessEx (which creates the process object from an image section) and NtCreateThreadEx (which inserts the first thread) is large enough to overwrite the on-disk file with a benign decoy. When the callback fires and the AV scans the disk, it sees the decoy. The mapped section in memory still has the original payload. Same effect as Doppelgänging without TxF.
What's the OS-level fact that Process Ghosting exploits?
Windows prevents modification or deletion of executable files only after they're mapped into an image section. So the technique: open a file with DELETE access, mark it delete-pending via NtSetInformationFile(FileDispositionInformation, DeleteFile=TRUE), write the payload, map it as a section, close the handle (deletion completes — the file is gone), then create a process from the section. The process now has a backing file that doesn't exist, so AV scans of the file path return "file not found."
Sysmon Event ID 25 fires for which class of techniques?
Image tampering. Specifically, when "the mapped image of a process doesn't match the on-disk image file, or the image file is locked for exclusive access." Introduced in Sysmon 13.00 (January 2021) as a direct response to herpaderping's public disclosure. Catches: process hollowing (in-memory PE differs from disk), herpaderping (decoy on disk, payload in memory), ghosting (backing file deleted), and most other PE tampering. Does not catch: APC injection (no PE tampering involved), classic remote thread (writes to private memory, not the image).
Why does Microsoft consider process tampering "not meeting their bar for servicing", and what does this mean for defenders?
Microsoft's stated position (from herpaderping and ghosting disclosures, 2020 and 2021) is that the kernel's job is to faithfully implement transactions, file deletion, and process creation as designed; preventing abuse of those primitives is the security product's responsibility. This is a deliberate boundary. For defenders it means: the kernel will not be patched to refuse these operations; detection must come from the EDR layer (FileObject inspection, ETW-TI, behavioral analysis). Sysmon Event 25 was Microsoft's compromise — surface the signal, let products decide what to do.

§ 10 Closing thought

Seven volumes in. Here's the picture so far:

  • Vol. 01 — the alphabet (data types).
  • Vol. 02 — the static skeleton (PE format).
  • Vol. 03 — the living state (PEB / TEB / LDR).
  • Vol. 04 — the machinery that builds that state (the loader).
  • Vol. 05 — the border crossing (syscalls).
  • Vol. 06 — how the EDR rewrites bytes near that border.
  • Vol. 07 — how attackers compose all of the above into seven foundational injection techniques.

The dominant pattern across all seven techniques: each one targets one specific detection chokepoint, and detection layers built on top of newer techniques rapidly close the gap. Hollowing was world-class in 2014, baseline in 2026. Doppelgänging was state-of-the-art in 2017, reliably caught in 2026. Herpaderping and ghosting got ~3 years before mainstream EDRs caught up. The half-life of an injection technique against mature commercial EDRs is roughly the time between its public disclosure and the next major Sysmon / Defender / Elastic Defend release.

That's both the bad news for offense (everything in this volume is now telemetry-rich) and the good news for defense (modern stacks reliably catch the foundational techniques). The arms race has moved to part 2: thread hijacking variants, callback abuse, and the 2024 generation that targets the process-creation race window itself.
