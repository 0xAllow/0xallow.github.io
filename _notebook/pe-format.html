The PE Format,
or: how an .exe becomes a process.

A reading of the Portable Executable format for people who need to write loaders, unhook DLLs, and build detections ; not for people who need to pass a compiler exam.

blacksunCUBE field notebook · vol. 02 · builds on vol. 01 (data types)

Every PE explainer on the internet starts the same way: a giant diagram of the file layout, then a breathless walk through every field of every header. You read it once, retain maybe 20%, and come back to it every three months when something breaks.

I want to do something different here. We are going to read the PE format the way the loader reads it ; in the order the OS actually needs each piece ; because that is the order in which each piece starts to mean something. A field you cannot explain in terms of "what does the loader do with this" is a field you have not really learned.

Same color axes as vol. 01 ; and this time, the colors are not a metaphor. PE sections carry literal R/W/X flags in their characteristics, and almost every interesting thing in offensive tooling revolves around those flags being something they should not be.

R · Read

Bytes you consult

Headers, metadata, constants, string tables. Loader looks at these to understand what to map.

W · Write

Bytes that mutate

Globals, the IAT after binding, relocations being applied. Things the process will modify at runtime.

X · Execute

Bytes the CPU will fetch

Code pages. .text. The only memory that should ever get the execute bit ; and when something else does, you are looking at an attack.

§ 01 The single most important thing: disk ≠ memory

A PE file has two lives. You cannot understand either without understanding the transition.

On disk, a PE file is a compact archive: headers packed tight, sections stored with their raw sizes, alignment at 512 bytes (historically one disk sector). In memory, the same file is a sprawling region: headers mapped to a page boundary, each section expanded to a larger virtual size, gaps zero-filled, alignment at 4096 bytes (the CPU page size).

The loader's job ; the entire job, stripped of ceremony ; is to read the on-disk layout, allocate a region of virtual memory, copy each section to its new home, fix up the pointers, and jump to the entry point. Everything else in this notebook is detail about how that map gets read.

Two vocabularies to internalize now, because you will trip over them in every tool:

File offset (raw)Byte position inside the .exe as it sits on disk. Aligned to FileAlignment, usually 0x200.
RVA ; relative virtual addressByte position inside the mapped image, measured from ImageBase. Aligned to SectionAlignment, usually 0x1000.
VA ; virtual addressImageBase + RVA. The actual address a pointer in the running process would contain.

A PE parser is, in large part, a machine for translating between these three. Get the translation wrong and nothing else you do will work; get it right and half the subsequent complexity evaporates.

If you have ever wondered why VirtualSize and SizeOfRawData on a section header are different numbers ; this is the reason. SizeOfRawData is how many bytes the section takes on disk (multiple of 0x200). VirtualSize is how many bytes it wants in memory (often larger, because .bss-style zero-filled regions are not stored on disk at all ; the loader just allocates zeros).

§ 02 The DOS header: a 40-year-old relic you still need

Almost entirely vestigial. One field matters. It unlocks everything else.

IMAGE_DOS_HEADER
read

"I am a fossil layer. Ignore me, except for one offset at the end that tells you where the real header starts."

typedef struct _IMAGE_DOS_HEADER {
    WORD  e_magic;       // 'MZ'  (0x5A4D)  ; the two bytes every PE begins with
    WORD  e_cblp;
    WORD  e_cp;
    // ... 25 fields of DOS-era junk we will never touch ...
    LONG  e_lfanew;      // offset to IMAGE_NT_HEADERS  <-- the only field that matters
} IMAGE_DOS_HEADER, *PIMAGE_DOS_HEADER;

The PE format starts with a complete MS-DOS executable header so that if someone accidentally ran a Windows binary on DOS, they would get the friendly "This program cannot be run in DOS mode" message instead of a crash. That scenario stopped mattering around 1997. The header stayed.

For you, the only field in this entire structure that matters is e_lfanew ; a 4-byte offset pointing to the real header. Every PE parser does the same two-step:

PIMAGE_DOS_HEADER  pDos = (PIMAGE_DOS_HEADER)pImageBase;
PIMAGE_NT_HEADERS  pNt  = (PIMAGE_NT_HEADERS)((ULONG_PTR)pImageBase + pDos->e_lfanew);

That cast-through-ULONG_PTR should look familiar from vol. 01. This line, or a small variation of it, opens every manual mapper, reflective loader, and API-hashing stub ever written.

Signal for defenders: a valid PE on disk or in memory always starts with MZ at offset 0 and has a valid e_lfanew pointing somewhere reasonable (not into the header itself, not past the file). YARA rules and memory scanners use these invariants as their coarsest first filter. Shellcode that maps a PE manually but forgets to copy the DOS header or zeros it out is still a PE semantically ; but memory-scanning an address space for regions that start with MZ is the first thing any memory scanner does. That is why stealthier loaders sometimes erase the MZ/PE signatures after loading.

§ 03 IMAGE_NT_HEADERS: where the PE actually begins

Three structures, glued together. Confusingly named "Optional" when it is not.

typedef struct _IMAGE_NT_HEADERS64 {
    DWORD                 Signature;       // 'PE\0\0'  (0x00004550)
    IMAGE_FILE_HEADER     FileHeader;      // machine, section count, flags
    IMAGE_OPTIONAL_HEADER OptionalHeader;  // everything the loader actually needs
} IMAGE_NT_HEADERS64, *PIMAGE_NT_HEADERS64;

Three things nested together. The Signature is the four bytes "PE\0\0" ; the second magic number (after MZ) that confirms this is really a PE file. Then two sub-structures.

The FileHeader ; what this binary is

IMAGE_FILE_HEADER
read
typedef struct _IMAGE_FILE_HEADER {
    WORD   Machine;              // 0x8664 = x64, 0x14C = x86, 0xAA64 = ARM64
    WORD   NumberOfSections;     // how many IMAGE_SECTION_HEADERs follow
    DWORD  TimeDateStamp;        // compiler timestamp, often forged
    DWORD  PointerToSymbolTable; // dead field in practice
    DWORD  NumberOfSymbols;      // also dead
    WORD   SizeOfOptionalHeader; // size of the thing below
    WORD   Characteristics;      // flags: EXE vs DLL, executable, etc.
} IMAGE_FILE_HEADER, *PIMAGE_FILE_HEADER;

Three fields here carry their weight. Machine tells you the architecture ; absolutely critical if you are writing cross-architecture tooling, and a common defender signal when a process unexpectedly loads an ARM64 module on an x64 box, or vice versa. NumberOfSections bounds the section header array you are about to walk. Characteristics is a bitmask with two flags you care about:

IMAGE_FILE_DLL             // 0x2000   ; this is a DLL, not an EXE
IMAGE_FILE_EXECUTABLE_IMAGE // 0x0002   ; bytes are a valid image (not an object file)
TimeDateStamp is supposedly the Unix time at which the linker built the binary. In practice: compilers like MSVC with /Brepro deterministic builds replace it with a hash, malware families often zero it or set it to an obviously-wrong value, and state-tier actors forge it to match benign binaries. Never trust it as forensic evidence on its own ; but an unexpectedly consistent timestamp across a campaign's samples is its own signal.

The OptionalHeader ; everything the loader actually needs

IMAGE_OPTIONAL_HEADER64
readexecute

"Optional" is a 30-year-old misnomer. Without me, the loader has no idea where to put your code, where to jump first, or how much memory to reserve.

typedef struct _IMAGE_OPTIONAL_HEADER64 {
    WORD        Magic;                // 0x20B = PE32+ (64-bit), 0x10B = PE32 (32-bit)
    BYTE        MajorLinkerVersion;
    BYTE        MinorLinkerVersion;
    DWORD       SizeOfCode;
    DWORD       SizeOfInitializedData;
    DWORD       SizeOfUninitializedData;

    DWORD       AddressOfEntryPoint;  // RVA of first instruction to execute
    DWORD       BaseOfCode;

    ULONGLONG   ImageBase;            // preferred VA to load at (e.g. 0x140000000 for EXEs)
    DWORD       SectionAlignment;     // in-memory alignment, usually 0x1000
    DWORD       FileAlignment;        // on-disk alignment, usually 0x200

    // ... OS version fields ...

    DWORD       SizeOfImage;          // total bytes to reserve when mapping
    DWORD       SizeOfHeaders;
    DWORD       CheckSum;
    WORD        Subsystem;              // 2 = GUI, 3 = console, ...
    WORD        DllCharacteristics;   // ASLR, DEP, CFG flags  ; matters for defense

    // ... stack/heap reserve/commit sizes ...

    DWORD               NumberOfRvaAndSizes;  // always 16 in practice
    IMAGE_DATA_DIRECTORY DataDirectory[16];  // RVA+size pairs, indexed by purpose
} IMAGE_OPTIONAL_HEADER64, *PIMAGE_OPTIONAL_HEADER64;

Four fields here are the ones you will write code against over and over:

ImageBaseThe VA where the module wants to be loaded. If the OS can honor this, no relocations are needed. If it can't (ASLR, conflicts), the relocation section is walked to fix up every absolute pointer.
AddressOfEntryPointRVA of the first instruction. For an EXE, this is where the CRT stub starts. For a DLL, this is DllMain (called with DLL_PROCESS_ATTACH).
SizeOfImageTotal bytes to reserve. Your manual mapper passes this to VirtualAlloc. Get it wrong and your sections overflow into neighboring allocations.
DataDirectory[16]The table of contents. Each entry is a { RVA, Size } pair pointing to a major PE structure ; imports, exports, relocations, TLS, exception data.
Signal for defenders: DllCharacteristics tells you whether the image opts into modern mitigations ; IMAGE_DLLCHARACTERISTICS_DYNAMIC_BASE (ASLR), NX_COMPAT (DEP), GUARD_CF (Control Flow Guard), HIGH_ENTROPY_VA (64-bit ASLR). A legitimate modern binary has all of these set; binaries that opt out are either old, niche, or suspicious. ETW's Microsoft-Windows-Kernel-Process and EDR image-load telemetry commonly surface this as part of module load events ; hunt for unsigned modules loaded into signed processes with weak DllCharacteristics.

§ 04 Section headers: the literal R/W/X we have been talking about

An array, right after the NT headers, one entry per section. This is where the color axes become law.

IMAGE_SECTION_HEADER
readwriteexecute
typedef struct _IMAGE_SECTION_HEADER {
    BYTE   Name[8];                   // ".text\0\0\0", ".rdata\0\0", ...
    union { DWORD PhysicalAddress;
            DWORD VirtualSize; }    Misc;  // bytes the section wants in memory
    DWORD  VirtualAddress;             // RVA where the section will be mapped
    DWORD  SizeOfRawData;              // bytes stored on disk (FileAlignment)
    DWORD  PointerToRawData;           // file offset where bytes begin
    DWORD  PointerToRelocations;        // obj files only
    DWORD  PointerToLinenumbers;        // dead
    WORD   NumberOfRelocations;
    WORD   NumberOfLinenumbers;
    DWORD  Characteristics;            // the R/W/X bits + section attributes
} IMAGE_SECTION_HEADER;

The Characteristics field is a bitmask, and three bits of it are the whole game:

IMAGE_SCN_MEM_READ      0x40000000   // the page will be readable   (R)
IMAGE_SCN_MEM_WRITE     0x80000000   // the page will be writable   (W)
IMAGE_SCN_MEM_EXECUTE   0x20000000   // the page will be executable (X)

When the loader maps your binary, each section gets its memory protection derived from these three bits. A typical layout:

.textCode. R-X. You read instructions from it; the CPU executes from it. You never write to it ; at least not legitimately.
.rdataRead-only data: import/export tables, string literals, const globals. R--.
.dataInitialized writable globals. RW-. Never X.
.bssZero-initialized writable globals. Often merged into .data. RW-.
.relocBase relocation table. R--. The loader reads this, then discards it.
.rsrcResources: icons, manifests, version info. R--.
.pdatax64 exception handling unwind info. R--.
the single most important rule in PE security

A section that is both writable AND executable (RWX, 0xE0000000) is almost never legitimate in a modern compiled binary. Legitimate code is R-X; legitimate data is RW-. Self-modifying code sections ; .text marked RWX ; are a screaming red flag. Packed malware that decrypts its payload in place is the textbook source of RWX sections on disk. Memory pages that become RWX at runtime (via VirtualProtect) are the textbook source at runtime.

→ operator's view

Because RWX is a known tell, modern loaders play the W⊕X game: allocate RW, write the payload, flip to R-X with VirtualProtect, then execute. The page is never simultaneously W and X. This is cheaper than it sounds and defeats lazy RWX scanners ; but leaves the VirtualProtect call itself as the signal.

→ defender's view

Hunt for private commit regions transitioning to RX after having been RW. Sysmon doesn't cover this cleanly; ETW's Microsoft-Windows-Kernel-Memory and EDR-specific kernel callbacks do. A single VirtualProtect to RX on a private page inside a non-JIT process is nearly always interesting.

§ 05 Data directories: the table of contents

Sixteen slots. Each one points to a structure that does real work at load time or runtime.

The DataDirectory array at the end of the OptionalHeader is how the PE tells the loader "go find this sub-structure over here." Each entry is just:

typedef struct _IMAGE_DATA_DIRECTORY {
    DWORD VirtualAddress;   // RVA of the structure
    DWORD Size;             // size in bytes
} IMAGE_DATA_DIRECTORY;

The sixteen slots are indexed by well-known constants. You do not memorize all sixteen. You memorize the six that do real work:

[0] EXPORTFunctions this module exposes to others. DLLs.
[1] IMPORTFunctions this module needs from other DLLs. Everyone.
[2] RESOURCEIcons, version info, manifests, embedded data.
[3] EXCEPTIONx64 unwind info (a table of RUNTIME_FUNCTION entries). Every x64 function must be here or exceptions break.
[5] BASERELOCHow to patch absolute addresses if loaded off-base. Essential when ASLR moves the module.
[9] TLSThread Local Storage ; and the infamous TLS callbacks that run before the entry point.
[12] IATThe Import Address Table. After binding, the actual function pointers to imported functions live here.

§ 06 Imports and exports: how DLLs find each other

If you understand nothing else at runtime, understand these two tables. They are 80% of the attack surface.

The export directory ; what a DLL offers

IMAGE_EXPORT_DIRECTORY
read
struct _IMAGE_EXPORT_DIRECTORY {
    DWORD Characteristics;
    DWORD TimeDateStamp;
    WORD  MajorVersion;
    WORD  MinorVersion;
    DWORD Name;                 // RVA to the DLL's own name string
    DWORD Base;                 // ordinal base (usually 1)
    DWORD NumberOfFunctions;    // entries in the address array
    DWORD NumberOfNames;        // entries in the name arrays (often fewer!)
    DWORD AddressOfFunctions;   // RVA to array of function RVAs
    DWORD AddressOfNames;       // RVA to array of name-string RVAs
    DWORD AddressOfNameOrdinals;// RVA to array of WORD ordinals
};

Resolving "where is NtAllocateVirtualMemory inside ntdll.dll" reduces to: find the export directory, walk AddressOfNames until you hit the string, take the index, look it up in AddressOfNameOrdinals to get an ordinal, look that up in AddressOfFunctions to get the RVA. Add the RVA to the DLL's base address. Done. That is GetProcAddress, stripped of ceremony.

NumberOfFunctions and NumberOfNames differ because some exports are ordinal-only (no name). ws2_32.dll is the famous example ; many winsock functions export by ordinal only. If you write a name-based resolver and forget this, your code silently fails to find those functions. Ask me how I know.

The import directory ; what a DLL demands

IMAGE_IMPORT_DESCRIPTOR (one per imported DLL)
readwrite
struct _IMAGE_IMPORT_DESCRIPTOR {
    DWORD OriginalFirstThunk;   // RVA to INT  (Import Name Table)   ; immutable
    DWORD TimeDateStamp;
    DWORD ForwarderChain;
    DWORD Name;                 // RVA to name of DLL being imported
    DWORD FirstThunk;           // RVA to IAT  (Import Address Table)  ; LOADER WRITES HERE
};

Two parallel arrays of pointers, one "before" and one "after":

// Before binding (as stored on disk):
INT[]  =  { &"NtAllocateVirtualMemory", &"NtProtectVirtualMemory", ... }
IAT[]  =  { &"NtAllocateVirtualMemory", &"NtProtectVirtualMemory", ... }

// After the loader binds (at runtime, in memory):
INT[]  =  { &"NtAllocateVirtualMemory", &"NtProtectVirtualMemory", ... }  // unchanged
IAT[]  =  { 0x7FF8B4A21C30,            0x7FF8B4A22140,             ... }  // real addrs

The INT (Import Name Table) stays as a list of name pointers ; this is what a disassembler shows you as "this imports function X." The IAT, starting the same way, gets overwritten by the loader with the actual resolved addresses. At runtime, when the .text section calls an imported function, it does so through an indirect jump via the IAT:

  call qword ptr [rip + __imp_NtAllocateVirtualMemory]   ; indirect through IAT entry
→ operator's view · IAT hooking

Because the IAT is a table of function pointers in writable memory, hooking an import is as simple as finding the IAT entry and overwriting it with your trampoline. This is one of the oldest hooking techniques; it is also why many EDRs enumerate every module's IAT at load time and alert on entries that don't point inside the expected target DLL.

→ defender's view · absence is a signal

A process whose IAT does not import NtAllocateVirtualMemory, NtProtectVirtualMemory, or CreateRemoteThread ; but which nevertheless executes those calls ; is doing manual resolution (API hashing, direct syscalls, or both). Tools like pe-sieve and hollows_hunter look for exactly this mismatch between declared imports and actual behavior.

§ 07 Relocations and TLS callbacks: the runtime fix-ups

Two small structures that do large things.

Base relocations ; patching pointers when the base moves

A PE is compiled assuming it will load at ImageBase. If that address is already taken ; and with ASLR, it virtually always is ; the loader maps the image somewhere else and then has to go through the image and patch every absolute pointer. The .reloc section tells it where those pointers are.

struct _IMAGE_BASE_RELOCATION {
    DWORD VirtualAddress;   // RVA of a 4KB page to fix up
    DWORD SizeOfBlock;      // size of this block including the header
    // followed by an array of 16-bit entries:
    //    upper 4 bits = type (IMAGE_REL_BASED_DIR64 on x64)
    //    lower 12 bits = offset within the page
};

The loader computes a delta (ActualBase - PreferredImageBase), then walks the relocation blocks, and at each offset inside each page, adds the delta to the 8-byte pointer stored there. A modern x64 binary might have tens of thousands of these. It's bulk work. It's also why DLLs built without /DYNAMICBASE (no .reloc) cannot load if their preferred base is taken ; the loader has no map to fix them up.

TLS callbacks ; code that runs before main

IMAGE_TLS_DIRECTORY
execute
struct _IMAGE_TLS_DIRECTORY64 {
    ULONGLONG StartAddressOfRawData;
    ULONGLONG EndAddressOfRawData;
    ULONGLONG AddressOfIndex;
    ULONGLONG AddressOfCallBacks;  // VA to a null-terminated array of function pointers
    DWORD     SizeOfZeroFill;
    DWORD     Characteristics;
};

Here is what matters: TLS callbacks execute before the entry point. Before main. Before DllMain in many cases. The loader calls every function in AddressOfCallBacks on process attach, thread attach, thread detach, and process detach.

why TLS callbacks are infamous

Malware has used TLS callbacks since at least the mid-2000s as an anti-debug and anti-analysis trick. An analyst who sets a breakpoint on the entry point misses any code that ran before it. Modern debuggers handle this correctly, but the technique still appears in unpackers and commodity loaders. From a detection standpoint: the presence of TLS callbacks is itself benign (the MSVC CRT uses them), but an unsigned binary with TLS callbacks pointing into code-like sections, especially in combination with high-entropy .text, is a strong prior for a packer.

Signal for defenders: YARA has a pe.number_of_tls_callbacks field. Combine with pe.characteristics checks, section entropy, and signing status for a surprisingly effective packer heuristic. In memory-scanning mode (pe-sieve, hollows_hunter), a mismatch between the on-disk TLS callback list and the in-memory one is a strong indicator of runtime patching.
§ 08  ·  the 30-second recall

Flashcards for durable memory

If you can answer these without looking up, you own the material. Click to reveal.

What is the one field in IMAGE_DOS_HEADER that actually matters, and why?
e_lfanew. It's the file offset (and, once the image is mapped, the RVA) to IMAGE_NT_HEADERS. Every PE parser reads this to skip the DOS relic and find the real headers.
A PE section has Characteristics = 0xE0000000. What's in it, and why do you care?
It's RWX ; readable, writable, and executable. Modern legitimate binaries should never ship with this. It is the classic fingerprint of packed or self-modifying code. Either the author is naive, or the section contains a decryption stub that writes the real payload into itself before jumping there.
Why are SizeOfRawData and VirtualSize often different on the same section?
SizeOfRawData is the count of bytes stored on disk, aligned to FileAlignment (0x200). VirtualSize is the count of bytes the section wants in memory. The difference is the zero-filled region (.bss-style data) that doesn't need to be stored on disk ; the loader just allocates zeros. You map VirtualSize bytes; you copy SizeOfRawData bytes into it.
What is the difference between the INT and the IAT, before and after the loader runs?
Before binding: both arrays contain RVAs to the imported function names. After binding: the INT (Import Name Table, OriginalFirstThunk) still contains name RVAs ; it is the immutable record of what was imported. The IAT (Import Address Table, FirstThunk) has been overwritten by the loader with the resolved absolute addresses. At runtime, calls go indirectly through the IAT.
A binary is loaded at a different base than ImageBase. What happens next?
The loader computes a delta = (ActualBase − ImageBase). It then walks the .reloc directory. For each relocation entry, it adds the delta to the absolute pointer at that location. This patches globals, function pointers, and all other base-relative values. If the binary has no .reloc (built without /DYNAMICBASE) and the preferred base is taken, loading fails.
You're asked to call NtAllocateVirtualMemory without importing it. Walk the steps at 10,000 feet.
Walk the PEB (gs:[0x60] on x64) to get Ldr, walk the InLoadOrderModuleList to find ntdll.dll and get its base (this is the HMODULE). Cast the base to IMAGE_DOS_HEADER, follow e_lfanew to the NT headers, read DataDirectory[0] (export dir), walk AddressOfNames (or hash them) to find your function, map through AddressOfNameOrdinals, then index AddressOfFunctions for the final RVA. Add to the ntdll base = function VA. No imports, no GetProcAddress.
Why do TLS callbacks matter for both offense and defense?
They run before the entry point. From an offensive angle, they let code execute before most analysts' first breakpoint, making them an anti-analysis trick of choice for packers. From a defensive angle, the combination of "unsigned binary + TLS callbacks pointing into high-entropy sections" is a robust packer heuristic. YARA exposes pe.number_of_tls_callbacks for exactly this reason.
What does DllCharacteristics tell you, and what's the red flag?
It's a bitmask describing which process-level mitigations the binary opts into ; DYNAMIC_BASE (ASLR), NX_COMPAT (DEP), GUARD_CF (CFG), HIGH_ENTROPY_VA (64-bit ASLR). Any modern compiled binary should have all of these set. A binary that opts out ; particularly one missing DYNAMIC_BASE ; is either ancient, deliberately weakened, or suspicious.

§ 09 Closing thought

The PE format is baroque, but it is not deep. Once you internalize the disk/memory split, the role of e_lfanew, and the fact that sections carry literal R/W/X flags, the rest is table-lookup. Every exotic technique you will read about ; reflective loading, process hollowing, manual mapping, IAT hooking, module stomping, API hashing, DLL unhooking ; is a recombination of operations on the structures we just covered. You are not learning eight separate techniques. You are learning one structure well enough that eight techniques become obvious.
