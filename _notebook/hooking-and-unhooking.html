Hooking & Unhooking,
or: how the EDR moves into your house, and how an attacker tries to evict it.

A walk through the bytes that get rewritten when a security product loads, the ways those bytes get put there, and the five families of techniques attackers use to clean them up all with the detection signature each one leaves behind.

blacksunCUBE field notebook · vol. 06 · builds on vol. 04 (loader) & vol. 05 (syscalls)

Vol. 05 ended with this claim: by the end of vol. 06, you should be able to look at any process and tell, by reading ntdll's first 16 bytes, whether an EDR has visited. That's the goal. By the time you finish this volume, you should also know which EDR pattern was used, what the EDR's DLL is doing on each call, and what would happen if you tried to unhook including the detection signal that would fire as you did.

Three concrete things matter here:

  1. The byte patterns themselves. A hook is a small, recognizable change to function prologue bytes. There are three or four canonical shapes; you'll memorize them in this volume.
  2. The injection mechanism. EDR DLLs don't appear by magic. There's a specific kernel-callback-driven sequence that brings them into your process before main runs. Understanding it tells you why "starting suspended" used to work and why modern EDRs have closed that gap.
  3. The taxonomy of unhook approaches. Five distinct technique families, each with its own detection signature. Knowing them lets you read any "EDR bypass loader" on GitHub and immediately classify the approach and predict what would catch it.

Same axes you know:

R · Read

Recognize the patch

The byte patterns at function prologues that mark a hooked binary. Five-byte E9 jumps, fourteen-byte absolute jumps, hardware breakpoints with no patch at all.

W · Write

Apply / remove

Changing memory protection on the .text section, writing the trampoline, restoring originals from a clean source. Every step leaves telemetry.

X · Execute

The redirect

What runs when control transfers to the EDR's analysis routine, and what happens (or doesn't) when it returns.

§ 01 What a hook actually is

A small, deliberate corruption of function prologue bytes nothing more, nothing less.

Strip away every blog post and conference talk; a user-mode API hook is one thing: the first few bytes of a function in a loaded DLL have been overwritten with a jump to somewhere else. Everything in the next 60 pages of this notebook is a variation on that idea.

The function being hooked is almost always in ntdll.dll, because that's where the syscall stubs live (vol. 05) and that's the last user-mode chokepoint before kernel transition. The somewhere else the hook jumps to is an analysis routine inside the EDR's own DLL, also loaded into the process. After the analysis routine decides "this call is safe", it jumps to a trampoline a small chunk of memory the EDR allocated, containing the original bytes it overwrote, followed by a jump back into the original function past the patched bytes.

That's the whole mechanism. Patch + trampoline. Everything else is engineering.

ntdll!NtAllocateVirtualMemory · same function, before & after EDR hook
Before EDR loads
7ffa`c476f0f0 4c 8b d1 ; mov r10, rcx 7ffa`c476f0f3 b8 18 00 00 00 ; mov eax, 18h (SSN) 7ffa`c476f0f8 f6 04 25 ... ; test byte ptr [..] 7ffa`c476f100 75 03 ; jne short 7ffa`c476f102 0f 05 ; syscall 7ffa`c476f104 c3 ; ret
After EDR DLL injection
7ffa`c476f0f0 e9 a3 12 02 00 ; jmp EDR.dll!Inspect 7ffa`c476f0f5 cc cc cc ; int3 padding (or original bytes) 7ffa`c476f0f8 f6 04 25 ... ; test byte ptr [..] 7ffa`c476f100 75 03 ; jne short 7ffa`c476f102 0f 05 ; syscall 7ffa`c476f104 c3 ; ret

Notice what didn't change: the syscall instruction itself, and the bytes after the patch. The first 5 bytes of the stub are gone replaced with a single JMP rel32 (opcode E9, plus 4-byte signed offset to the EDR's analysis routine). The original 5 bytes (4c 8b d1 b8 18) are now in the EDR's trampoline elsewhere in memory.

When code calls NtAllocateVirtualMemory:

  1. CPU lands on byte 0 of the hooked stub, executes E9 a3 12 02 00, jumps to the EDR's analysis routine.
  2. EDR routine inspects arguments (RCX, RDX, R8, R9, stack), call stack, and decides: allow, block, or alert.
  3. If allowed, EDR jumps to its trampoline.
  4. Trampoline executes the original 5 bytes (mov r10, rcx + mov eax, 18h) and then jmps back to NtAllocateVirtualMemory + 5, where the unpatched test byte ptr [..] instruction continues normally.
  5. The original syscall fires, kernel does its work, returns to caller. The caller never knows the EDR was there.
The 4c 8b d1 b8 18 00 00 00 sequence at the top of every Nt* function is not a coincidence it's the same 8-byte pattern (mov r10, rcx + mov eax, ssn) that vol. 05 introduced. That 8-byte regularity is precisely what lets Hell's Gate find SSNs without symbols, and precisely what lets a hook detector find the patched functions.

§ 02 Three flavors of inline hook

Same idea, three byte-level shapes learn them and you can recognize any EDR's patch on sight.

Flavor 1 Five-byte relative jump (the classic)

JMP rel32 opcode E9
execute

5 bytes. Reaches anywhere within ±2 GiB of the patch site. The default for AVs and lightweight EDRs.

; Patch bytes:
;    E9 <rel32>
;
; Where <rel32> is computed as:
;    target_address - (patch_address + 5)
;
; Limitation: the target must live within ±2 GiB of the patched function.
; Almost always true in practice — both ntdll and the EDR's DLL are mapped
; in the same process address space, well within 2 GiB of each other.

This is what Microsoft Detours uses, what most commercial AVs use, and what Hell's Gate / Halo's Gate detect by checking for byte 0xE9 at offset 0 of an Nt* stub. If you see E9 at the prologue of any Nt* function, the function is hooked.

Flavor 2 Fourteen-byte absolute jump (used when ±2 GiB isn't enough)

FF 25 + qword absolute indirect jump
execute

14 bytes. Reaches anywhere in 64-bit space. More complete coverage of Nt* stubs that have multiple complete instructions in the first 14 bytes.

; Patch bytes (14 total):
;    FF 25 00 00 00 00         ; jmp [rip+0]
;    <qword target>             ; absolute 8-byte address

; Or alternatively, MOV-IMM + JMP register (12 bytes, slightly less common):
;    48 B8 <qword target>       ; mov rax, target
;    FF E0                       ; jmp rax

Some EDRs use this when they want to be sure they overwrite enough bytes to capture every possible first instruction (the prologue of NtAllocateVirtualMemory includes a test with an immediate operand depending on version, that might be 9 or 10 bytes by itself). Recognizable by FF 25 or 48 B8 at offset 0.

Flavor 3 Single-instruction replacement ("atomic" hook)

Rarer, but worth knowing: some EDRs replace exactly one middle instruction with the jump. Specifically, the mov eax, <SSN> instruction at offset 3 (5 bytes itself) is replaced with the E9 jump. The advantage is atomicity: a single 5-byte instruction is rewritten as a single 5-byte instruction, no torn-write window. The first 3 bytes (4c 8b d1) survive.

; Atomic mid-function hook:
;
;    4C 8B D1                    ; mov r10, rcx        (untouched)
;    E9 <rel32>                  ; jmp EDR             (was: mov eax, SSN)
;    <original padding bytes>
;
; This is exactly the pattern that Tartarus' Gate (vol. 05) recognizes
; via its 4C 8B D1 E9 ... heuristic.
Detection counterpart: a hook scanner walks every Nt* export and reads bytes at offsets 0 and 3. The unhooked stub starts with 4c 8b d1 b8; if those 4 bytes don't match, something has been rewritten. False-positive rate is essentially zero on production Windows because Microsoft's syscall stubs follow this exact pattern. This is also exactly what Halo's / Tartarus' Gate use to recognize hooks meaning offensive and defensive parties have the same detection logic running in the same process at the same time.

§ 03 IAT & EAT hooks: the alternative no-one really uses anymore

An older approach worth knowing because it occasionally still appears in the wild.

Vol. 02 introduced the Import Address Table (IAT): an array of function pointers the loader fills in with the resolved addresses of every imported function, so when your code calls VirtualAlloc, the actual jump target is read from IAT[VirtualAlloc]. An IAT hook replaces that pointer with one to your inspection routine. No function bytes are modified anywhere the original VirtualAlloc is untouched. You just intercept the dispatch layer above it.

The Export Address Table (EAT) is the same idea on the producer side: each exported function in a DLL has its address listed in the EAT. Patching the EAT means anyone who later calls GetProcAddress gets your address, not the real one.

ApproachWhat gets modifiedWhat it catchesWhy it's mostly obsolete
IAT hookFunction pointer in your module's import tableCalls that go through that specific importOnly catches that one importing module. Direct calls via GetProcAddress or any code that bypasses the IAT sail through. EDRs need universal coverage; inline hooks at the source DLL provide it.
EAT hookFunction pointer in source DLL's export tableFuture GetProcAddress lookupsCatches nothing already-resolved. Static IAT entries (resolved at load time) sail through. Only useful for late-bound calls.

You'll still see IAT hooks in some EDR-internal contexts (e.g., for non-syscall callbacks), and certain DLP/insider-threat tools use them. For the syscall coverage problem "see every Nt* call in this process" inline hooks at ntdll dominate because they catch all callers in one shot.

§ 04 Hardware-breakpoint hooks (no patch at all)

An EDR-side and offense-side technique. The bytes never change the CPU's debug machinery does the redirect.

The CPU has 4 hardware breakpoint registers: DR0 through DR3, with control via DR6/DR7. Each can hold a linear address; when execution (or read/write, depending on configuration) reaches that address, the CPU raises a #DB debug exception before the instruction at that address executes. If a Vectored Exception Handler is registered, it catches the exception, inspects the CPU context (which the OS hands to the handler, including the volatile registers), modifies what it wants, and returns control resumes wherever RIP now points.

This gives you a hook that doesn't touch the target's bytes at all. The function looks pristine to a memory-integrity scanner. The only artifact is in the thread's debug registers and in the registered exception handler.

VEH + DR0 hook (typical pattern)
readwrite
1. AddVectoredExceptionHandler(FIRST, MyHandler);
2. GetThreadContext(thread, &ctx);
3. ctx.Dr0 = (DWORD64) target_function;          // where to break
4. ctx.Dr7 |= 0x1;                                // L0 = 1 — enable DR0 break
5. SetThreadContext(thread, &ctx);

// When target_function executes, CPU raises #DB.
// MyHandler fires. ctx->Rip points to target_function.
// Inspect args (Rcx/Rdx/R8/R9), modify if you want, then either
// step over (set TF=1, return EXCEPTION_CONTINUE_EXECUTION)
// or redirect (overwrite ctx->Rip with a different address and return).

The 2024 evolution: VEH²

The standard pattern above leaks a major IOC: SetThreadContext writing debug registers transitions through NtSetContextThread in the kernel, which in turn fires events on the Microsoft-Windows-Kernel-Audit-API-Calls ETW provider. Defenders watch for that. CrowdStrike researchers (Onofri & Sonawane, BlackHat MEA 2023) demonstrated VEH²: register the VEH, trigger your own exception, and modify the debug registers from inside the handler, where the OS has already given you a writeable CONTEXT struct so no SetThreadContext call is needed.

That single change makes the technique substantially harder to spot. The CONTEXT struct passed to the VEH has volatile Dr0–Dr7 fields; modifying them inside the handler is invisible to the kernel-audit provider that watches NtSetContextThread. fluxsec and others have published Rust ports of the same pattern (2024).

Hardware-breakpoint hooks are a hard capability for both sides. Offensively, they let you intercept individual function calls without modifying any DLL bytes. Defensively, EDRs use the same primitive in reverse setting hardware breakpoints on syscall instructions inside ntdll to catch indirect-syscall callers (EvilBytecode/Detecting-Indirect-Syscalls, 2025). Same instruction, different team. As always with these chapters: the same primitive serves whoever installs it first.

§ 05 How EDR DLLs actually arrive in your process

A kernel callback fires before your main runs, queues an APC, and the rest is inevitable.

The EDR's hooks live in a DLL. That DLL has to be in your process before any code you care about runs otherwise the unhooked window between main and DLL injection is exactly where malware would do all its work. Modern EDRs solve this through the PsSetCreateProcessNotifyRoutine(Ex) + PsSetLoadImageNotifyRoutine kernel callbacks (vol. 04), combined with a kernel-to-user APC.

Key insight: between step 5 and step 8, there's a real window of milliseconds during which ntdll is loaded but not yet hooked. This is the window Perun's Fart, EDR-Preloading, and Early Cascade Injection all target. If you can get your code to execute in this window via APC injection earlier than the EDR's APC, or via a TLS callback in the EXE you can capture a clean copy of ntdll, or stop the EDR DLL from loading at all.

The arms race in process startup

This is why the offensive research community spent 2023–2025 attacking the startup race. Outflank's Early Cascade Injection (October 2024) exploited LdrLoadDll internals during process creation to land before the EDR's APC fired. MalwareTech's EDR-Preloading (February 2024) hooked KiUserApcDispatcher in a parent-spawned suspended child to neutralize the EDR's incoming APC. Each technique targets one of the steps in the table above.

Modern EDRs respond by registering their callback as early as possible (kernel autoload at boot, hooks installed in the very first APC slot), and by having their kernel driver inject hooks directly from kernel mode if the user-mode APC racing fails. fluxsec's "Sanctum" research-EDR (2024) demonstrates the kernel-to-user APC path with shellcode that calls LdrLoadDll the same pattern used by 0xPrimo's KMDllInjector.

§ 06 Five families of unhook

Each one trades stealth for reliability differently. Know them, predict their detection signatures.

unhook taxonomy · offensive techniques, ordered by year of mainstream adoption
1. Refresh from disk — ired.team / classic, 2019+
Open C:\Windows\System32\ntdll.dll, map it as a section, copy the clean .text bytes over the hooked ones in your loaded ntdll. Original technique. Conceptually simple, widely copy-pasted.
Trade-off: opens a file handle to ntdll.dll from a non-system process high-fidelity IOC. NtCreateSection with SEC_IMAGE on ntdll is logged via ETW. EDRs with file-access correlation flag this immediately. Also leaves two copies of ntdll in memory if not properly unmapped. Detection: very easy.
2. Refresh from KnownDLLs — variant, 2020+
Instead of opening ntdll.dll from disk, open the kernel-maintained \KnownDlls\ntdll.dll section directly via NtOpenSection. The KnownDLLs section was created at boot from a clean ntdll.dll, before any EDR could hook it. Same overwrite procedure as #1, but no disk I/O.
Trade-off: avoids the disk read, but NtOpenSection(\KnownDlls\ntdll.dll) is itself an unusual call from non-loader code. EDRs with object-callback monitoring flag access to KnownDLLs entries. Detection: medium.
3. Perun's Fart (suspended child) — Reenz0h / Sektor7, 2022
Spawn a process in CREATE_SUSPENDED state, read its ntdll from memory before the EDR's APC has fired in that child, copy clean syscall stubs back to your own process. No disk read. Exploits the millisecond window between steps 5 and 8 in the injection sequence above.
Trade-off: creating a suspended child process is itself a moderate IOC (process injection precursor). Modern EDRs that register their kernel callback early enough can hook the child before the parent reads its memory. Detection: medium-high; works less reliably against mature EDRs in 2026.
4. Process forking / cloning — reprgm Attack of the Clones, 2023
Use NtCreateProcessEx to clone the current process. The clone has no threads (so process/thread kernel callbacks don't fire), is invisible in Task Manager, and inherits the parent's memory map including any clean ntdll. Copy .text from the clone, terminate it, profit.
Trade-off: still requires writing to your own ntdll!.text, which means VirtualProtect on a non-private region. ETW-TI's PROTECTVM_LOCAL event fires. Detection: harder than #1–3 at the kernel-callback layer, but the protect-and-write pattern remains the chokepoint.
5. Selective in-memory unhook — S12 Dark Development, 2025
Combines #3 / #4 with surgical replacement: instead of overwriting the entire .text section, identify only the hooked stubs (offset-0 byte 0xE9 or 0xFF) and copy only those individual 23-byte syscall prologues from the clean copy. Less memory traffic, smaller telemetry footprint, optionally restoring hooks before exit.
Trade-off: still requires NtProtectVirtualMemory on the hooked region. Reduces but doesn't eliminate the patching IOC. State of the art for 2025/2026 detection-aware loaders. Detection: hardest of the five the protection change is smaller and the write is brief, but ETW-TI still observes it.

Across all five techniques, the unavoidable chokepoint is the same: writing to the .text section of ntdll requires changing its memory protection, which fires telemetry. Direct-syscall unhookers exist (calling NtProtectVirtualMemory via SSN to avoid the user-mode hook on VirtualProtect), but the kernel-side ETW-TI PROTECTVM_LOCAL event fires regardless of how the syscall was reached. Vol. 05's lesson applies: the events fire in the kernel, regardless of invocation method.

honest assessment for 2026

Against legacy AV and EDRs that rely primarily on user-mode hooks, all five unhook families work. Against Defender for Endpoint, CrowdStrike Falcon, SentinelOne, Elastic Defend products that deemphasize user-mode hooks in favor of kernel callbacks, ETW-TI, and call-stack analysis unhooking does almost nothing on its own. The hwbp/NTDLL-Unhook README summarizes: "Advanced EDRs like Microsoft Defender for Endpoint and Elastic don't even use usermode hooks anymore since they rely on kernel callbacks and ETW telemetry, so unhooking does literally nothing against them." This is true. Modern offensive operations against mature EDRs combine unhooking with call-stack spoofing (vol. 05) and careful telemetry awareness.

§ 07 Detection: how a defender catches all of this

Five chokepoints, each with a clear telemetry signature.

1. Hook-integrity hashing

The simplest defensive primitive: at DLL-load time, after the EDR has installed its own hooks, take a SHA-256 of the .text section of ntdll. Periodically rehash and compare. Any change including unhooking back to the original bytes produces a different hash. fluxsec's research-EDR (2024) implements exactly this with a Rust thread that hashes every few seconds. Detection cost: trivial. False-positive rate: zero, because nothing legitimate modifies ntdll!.text after process startup.

2. ETW-TI PROTECTVM_LOCAL (event ID 7)

Every NtProtectVirtualMemory call against a region inside ntdll with target protection PAGE_EXECUTE_READWRITE (or PAGE_EXECUTE_WRITECOPY) fires a PROTECTVM_LOCAL event on the Threat-Intelligence ETW provider. Jonathan Johnson published the canonical analysis (June 2024). The event includes: process ID, target base address, region size, old protection, new protection. This is the chokepoint that makes all five unhook families detectable.

3. SetThreadContext on debug registers

For VEH-based hardware-breakpoint hooks (§ 04), the conventional pattern uses NtSetContextThread to write Dr0–Dr7. This fires events on Microsoft-Windows-Kernel-Audit-API-Calls. CrowdStrike's VEH² research bypasses this specific telemetry, but the bypass itself produces other artifacts namely, an exception being raised intentionally inside the process, which is itself trackable via the same audit provider's exception events.

4. Call-stack analysis on NtProtectVirtualMemory

Even if the protect call is made via a direct syscall to evade user-mode hooks, kernel-side ETW callstack capture (Elastic 8.8+) records the stack. A NtProtectVirtualMemory targeting ntdll's .text section, called from a stack frame that doesn't end in ntdll or kernelbase, is high-confidence malicious.

5. The NtCreateSection(SEC_IMAGE, ntdll.dll) signature

Specific to refresh-from-disk and refresh-from-KnownDLLs unhooking. Re-mapping ntdll.dll as a section image from a non-loader thread is unusual enough that several EDRs alert on it directly. Combined with subsequent NtProtectVirtualMemory on the original ntdll, it's a near-certain unhook attempt.

Hunt query template (Defender for Endpoint / KQL-flavored): for every VirtualProtect event where the target memory region overlaps a known DLL's .text section AND the requested protection includes write permission AND the calling process is not a debugger or installer, alert. False-positive rate is very low in mature environments. Pair with the parent process and module hash for triage.
§ 08  ·  the 30-second recall

Flashcards

Eight questions. If you can answer all without notes, you can read any 2026 EDR-bypass loader and predict its detection signature.

In bytes, what does a typical 5-byte inline hook on NtAllocateVirtualMemory look like, and where do the original bytes go?
The first 5 bytes of the function (4c 8b d1 b8 18: mov r10, rcx + start of mov eax, 18h) are overwritten with E9 <rel32> a 5-byte relative jump to the EDR's analysis routine. The original 5 bytes are preserved in a separate memory region called the trampoline, allocated by the EDR. After the analysis routine inspects the call, it jumps to the trampoline, which executes the original bytes and then jumps back to NtAllocateVirtualMemory + 5 to continue normal execution.
When would an EDR use a 14-byte absolute jump (FF 25) instead of a 5-byte relative jump?
When 5 bytes isn't enough to capture full instructions (the prologue might begin with multi-byte instructions whose boundaries don't align cleanly at 5 bytes), or when the analysis routine lives more than ±2 GiB from the patch site. The 14-byte form (FF 25 00 00 00 00 <qword>) is an absolute indirect jump that reaches anywhere in 64-bit space. The 12-byte alternative (48 B8 <qword> FF E0: mov rax, target; jmp rax) is also seen.
Why are IAT hooks not used by EDRs for syscall coverage?
IAT hooks only catch calls that go through that specific module's import table. Code that resolves VirtualAlloc via GetProcAddress at runtime, or that calls NtAllocateVirtualMemory directly via syscall, sails through the IAT entirely. EDRs need to catch every caller in the process; placing inline hooks at the source DLL (ntdll) achieves that with one patch site per syscall.
Hardware-breakpoint hooks via VEH don't modify any function bytes. What's the IOC instead?
Two artifacts: (1) the thread's debug registers (Dr0–Dr7) contain non-zero breakpoint addresses, and (2) the conventional setup uses NtSetContextThread to write those registers, which fires Microsoft-Windows-Kernel-Audit-API-Calls ETW events. CrowdStrike's VEH² (2023) bypasses #2 by modifying the volatile CONTEXT from inside an exception handler, but the registered exception handler itself is also an artifact.
Walk through how an EDR DLL gets into a brand-new process what fires when?
(1) Parent calls CreateProcess. (2) Kernel builds the new EPROCESS. (3) PsSetCreateProcessNotifyRoutineEx callback fires the EDR driver receives notification. (4) EDR driver queues a user-mode APC with shellcode that calls LdrLoadDll on the EDR DLL. (5) New process begins at ntdll!LdrInitializeThunk. (6) NtTestAlert drains the APC queue, EDR APC fires. (7) Shellcode calls LdrLoadDll(EDR.dll). (8) EDR's DllMain calls VirtualProtect + writes hooks. (9) Loader continues, eventually calls main with hooks already in place.
Perun's Fart spawns a suspended process to harvest a clean ntdll. Why does this work, and when does it fail?
It works because ntdll is loaded immediately when a process is created (it's the very first DLL), and there's a measurable window between when ntdll is mapped and when the EDR's user-mode APC fires to install hooks. A suspended child has ntdll in memory but no APCs have run yet. Reading the child's ntdll!.text at this moment yields clean syscall stubs. It fails when the EDR is registered as a kernel-driven hook injector that operates from kernel mode (no user-mode APC race window), or when the EDR's callback runs early enough to hook the child before the parent reads it.
An attacker uses a direct syscall to call NtProtectVirtualMemory(ntdll!.text, RWX), bypassing the user-mode hook. Does this evade detection?
No. The user-mode hook is bypassed, but the kernel-side ETW-TI PROTECTVM_LOCAL event fires inside NtProtectVirtualMemory's kernel implementation, regardless of how the syscall was invoked. The event includes the target base address, requested protection, and source process. An EDR consuming ETW-TI sees the protection change to RWX on ntdll's .text section an extremely high-fidelity IOC. Attempts to bypass this require kernel-level write access (BYOVD, etc.), not user-mode tricks.
Why does the README of hwbp/NTDLL-Unhook say "unhooking does literally nothing" against modern EDRs?
Because Defender for Endpoint, Elastic Defend, and similar mature EDRs deemphasize user-mode hooks in favor of kernel callbacks (PsSetLoadImageNotifyRoutine, etc.) and ETW-TI. Unhooking removes the user-mode hook bytes but those weren't the EDR's primary detection mechanism in the first place. The kernel-side telemetry continues firing identically before and after unhooking. Removing user-mode hooks does help against legacy AV and lighter EDRs that rely heavily on inline hooking, but against the modern tier, it's at best a partial measure that needs to be paired with call-stack spoofing and other techniques.

§ 09 Closing thought

Six volumes in. Here's the picture so far:

  • Vol. 01 the alphabet (data types).
  • Vol. 02 the static skeleton (PE format).
  • Vol. 03 the living state (PEB / TEB / LDR).
  • Vol. 04 the machinery that builds that state (the loader).
  • Vol. 05 the border crossing (syscalls).
  • Vol. 06 how the EDR rewrites bytes near that border, and how attackers undo it.

A defining theme of the last two volumes is this: user-mode hooks were the entire EDR detection model in 2018. By 2026, they are a residual layer that catches commodity malware while real telemetry happens in the kernel. Hell's Gate, Halo's Gate, Tartarus' Gate, Perun's Fart, EDR-Preloading, Early Cascade Injection brilliant techniques, all of them, and all primarily targeting a defense-in-depth layer that mature EDRs no longer rely on as their core. The honest answer for purple team operators in 2026 is: unhooking gets you past the doormat; the burglar alarm is in the walls.

For detection engineers, the inverse: don't build your strategy around hook integrity alone. ETW-TI, kernel callbacks, and call-stack capture are your bedrock. Hook integrity is a confirmation signal, not a primary one.
