DirtyFrag How the Page-Cache
Write Primitive Works

Two independent kernel subsystems IPsec xfrm and the RxRPC transport contain logic errors that allow an unprivileged process to write arbitrary data into the kernel page cache of files it does not own. Chained together, they produce a deterministic, universal local privilege escalation on every major Linux distribution. This post breaks down the root cause, the exploitation logic, and what defenders should know.

blacksunCUBE · research ref · original discovery: Hyunwoo Kim (@v4bel) · V4bel/dirtyfrag

1. Background the page-cache write bug class

The Linux kernel keeps file data in an in-memory structure called the page cache. When a process reads a file, the kernel maps its pages into the page cache and returns data from there. Writing back to disk goes through the same structure. Normally, writing to a page-cache page requires appropriate permissions on the underlying file a process cannot modify a file it has no write access to.

The Dirty Pipe vulnerability (CVE-2022-0847, discovered 2022) proved that this assumption can break: certain kernel code paths that splice network buffers into file-backed pages do not always check the write-protect flag on those pages. The result is an unprivileged process injecting arbitrary bytes into the page cache of any readable file including /etc/passwd, SUID binaries, and the vDSO.

DirtyFrag extends this same class. The name reflects the mechanism: the vulnerability corrupts the frag member of struct sk_buff the kernel's socket buffer structure. A fragment inside a socket buffer is made to point at a page-cache page that belongs to a file the attacker cannot write, and then a kernel operation writes into that fragment, landing the data in the wrong page.

Key distinction from Dirty Pipe
Dirty Pipe required a race condition a timing window between two kernel operations. DirtyFrag is a deterministic logic bug: no race, no timing dependency, no kernel panic on failure. The success rate is effectively 100%. This makes it significantly more reliable in practice.

2. The primitive what the attacker gains

Both CVE-2026-43284 and CVE-2026-43500 produce the same primitive: the ability to write controlled bytes into the page cache of an arbitrary readable file. The write bypasses file system permissions entirely because it happens at the level of kernel memory, not through normal file I/O.

The classic exploitation target is /etc/passwd: the attacker overwrites the root user's password hash field with the hash of a known password, then runs su root to obtain a root shell. Alternatively, a SUID binary or the vDSO page can be targeted for code injection.

C · simplified attacker-visible effect
// What the attacker controls after triggering the primitive:
// a 4-byte (xfrm) or larger write into the page cache of any readable file.

// Target: overwrite root's password hash in /etc/passwd
// File is world-readable (mode 0644) but root-owned  write should be denied.
// After triggering the primitive, the page cache contains attacker data.

int fd = open("/etc/passwd", O_RDONLY);   // succeeds  file is readable
// ... trigger xfrm or rxrpc write path ...
// page cache now contains modified password hash
// next read() of /etc/passwd returns attacker-controlled data
system("su root");  // uses known hash → root shell

The write is ephemeral in one sense: it modifies the page cache, not the on-disk file. However, on most configurations the page will eventually be written back to disk, and the modification persists until the system is rebooted or the page is evicted. For the purpose of obtaining a root shell, the in-memory modification is sufficient.

3. CVE-2026-43284 xfrm-ESP path

What is xfrm?

xfrm is the Linux kernel's IPsec transformation framework. It handles encryption and decryption of network packets for VPN protocols. The ESP (Encapsulating Security Payload) transform is responsible for encrypting and authenticating packet payloads.

To use xfrm, a process sets up a Security Association (SA) and a Security Policy via an AF_KEY socket. On most distributions, an unprivileged process can do this inside a user namespace created with unshare(CLONE_NEWUSER).

The root cause

When the xfrm ESP transform decrypts an incoming packet, it builds a struct sk_buff (socket buffer) to hold the decrypted data. The socket buffer uses a scatter-gather array of fragments (skb_frag_t). Each fragment is a reference to a memory page plus an offset and a length.

The logic error occurs in the transition between the encrypted input buffer and the decrypted output buffer. The kernel reuses pages from the input buffer in the output fragment list without checking whether those pages belong to the page cache of a file and without verifying that the calling process has write permission on that file.

An attacker can craft the situation so that the output fragment points at a page-cache page of a target file. The decryption output which the attacker controls by choosing what to encrypt is then written into that page.

Introduced / patched
The vulnerable code path was introduced in commit cac2661c53f3 (2017-01-17) and has been present in all kernels since. The mainline patch is f4c50a4034e6, merged into upstream. As of 2026-05-10, no distribution has backported this patch. The vulnerability has been in production kernels for approximately 9 years.

Constraint: namespace creation privilege

Triggering the xfrm path requires CLONE_NEWUSER the ability to create a user namespace. This is enabled by default on most distributions. Ubuntu with AppArmor can restrict this via apparmor_restrict_unprivileged_userns=1, which blocks the xfrm path on hardened Ubuntu systems. This is the reason the second CVE was needed.

4. CVE-2026-43500 RxRPC path

What is RxRPC?

RxRPC is a reliable transport protocol used by the AFS (Andrew File System) distributed filesystem. It is implemented as an AF_RXRPC socket family in the Linux kernel. The module is rxrpc.ko.

On most distributions, rxrpc.ko is not installed or not autoloaded. On Ubuntu, the module is included in the default kernel package and loaded automatically, making it available to any local process.

The root cause

A similar fragment-reference logic error exists in the RxRPC receive path. When RxRPC reassembles incoming data packets into socket buffers for delivery to userspace via rxrpc_recvmsg(), it manages fragment lists in struct sk_buff in a way that can cause page-cache pages of unrelated files to appear as fragment destinations.

Crucially, triggering this path requires no special privileges: any local user can open an AF_RXRPC socket and interact with it. No namespace creation is involved.

Patch status critical
CVE-2026-43500 has been reserved for tracking but no patch exists in any kernel tree as of 2026-05-10. Every Ubuntu system with rxrpc.ko loaded which is the default configuration is fully vulnerable to this path regardless of any other mitigations applied, including the Copy Fail algif_aead blacklist and the xfrm patch.

5. Why chain two vulnerabilities?

Neither variant alone covers all targets:

6. Relationship to Copy Fail and Dirty Pipe

Dirty Pipe (CVE-2022-0847) was the first public demonstration of a page-cache write primitive in the Linux kernel. It used the pipe + splice path. The fix involved correctly resetting the PIPE_BUF_FLAG_CAN_MERGE flag.

Copy Fail found the same sink the same page-cache write location via the algif_aead (AEAD crypto interface) source. The recommended mitigation was to blacklist algif_aead.

CVE-2026-43284 (xfrm) shares the same sink as Copy Fail but uses the xfrm ESP transform as the source. The algif_aead blacklist does not block it: different source, same destination. Any system that applied the Copy Fail mitigation and considered itself protected remains fully vulnerable to CVE-2026-43284.

DirtyFrag is therefore not just a new vulnerability it is the demonstration that the underlying sink was never properly fixed, and that new sources reaching it continue to be discoverable.

7. Mitigation

The only currently available mitigation is to prevent the vulnerable modules from loading. The official command (from the researcher and linux-distros):

SHELL · official mitigation blacklist all three modules immediate + persistent 
sh -c "printf 'install esp4 /bin/false\ninstall esp6 /bin/false\ninstall rxrpc /bin/false\n' \
  > /etc/modprobe.d/dirtyfrag.conf; rmmod esp4 esp6 rxrpc 2>/dev/null; true"

Side effects: this disables IPsec ESP transport (esp4/esp6) and the AFS/RxRPC transport (rxrpc). Evaluate operational impact before applying in environments that use VPN or AFS. If IPsec is required, prioritise applying the distro backport of f4c50a4034e6 when available.

Note on existing mitigations
Systems that applied the Copy Fail algif_aead blacklist are still vulnerable to both CVEs in DirtyFrag. Systems that applied the xfrm patch (f4c50a4034e6) are still vulnerable to CVE-2026-43500 (rxrpc, no patch exists). The module blacklist is the only complete mitigation available today.
